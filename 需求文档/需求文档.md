# AI 赋能全流程测试平台需求文档

## 1. 文档信息

| 项目 | 内容 |
|---|---|
| 产品名称 | SmartHub AI 测试平台 |
| 文档类型 | 产品需求文档（PRD） |
| 文档版本 | V1.0 |
| 产品定位 | AI 协助、人工审核的全生命周期测试平台 |
| 目标用户 | 测试、产品、研发、架构、项目管理及质量管理人员 |

## 2. 项目背景

传统测试工作在需求分析、影响评估、用例设计、自动化执行和结果分析之间存在明显断点：

- 需求依赖人工阅读，边界条件、异常场景和隐含约束容易遗漏；
- 技术方案、代码变更与测试范围缺少自动关联；
- 测试用例和自动化脚本编写成本高，复用率与覆盖率难以持续提升；
- UI 自动化脚本容易因页面元素变化失效，维护成本高；
- 执行日志、截图、调用链和错误堆栈分散，定位根因耗时；
- 测试资产未形成持续更新、可检索和可追溯的知识体系。

本项目拟建设一个贯穿测试生命周期的 AI 赋能平台，将 AI Agent 嵌入需求分析、技术方案分析、测试设计、用例执行和测试报告环节，实现从“人工逐步操作”转向“AI 生成与分析、规则自动校验、人工审核决策”。

## 3. 产品目标

### 3.1 总体目标

构建 Agentic Testing Loop，使需求、技术方案、代码、测试用例、执行结果、缺陷和报告形成闭环，并持续沉淀为项目知识。

```text
需求文档
   ↓
AI 需求分析 ──> 人工确认
   ↓
AI 技术影响分析 ──> 人工确认测试范围
   ↓
AI 生成测试用例与脚本 ──> 规则校验 + 人工审核
   ↓
测试编排执行 ──> 受控自愈
   ↓
日志聚类与根因分析
   ↓
测试报告、缺陷与知识沉淀
   └──────────────────────> 反馈到下一轮分析
```

### 3.2 业务目标

- 提升需求测试点、边界条件和异常场景的覆盖率；
- 缩短从需求提交到形成可执行测试方案的时间；
- 基于代码影响范围推荐最小回归测试集；
- 降低测试用例和自动化脚本的编写及维护成本；
- 将测试执行结果转化为可读、可追溯的风险和根因结论；
- 建立跨需求、代码、用例、执行、缺陷和报告的追踪关系；
- 对 AI 的输入、依据、输出、审核和使用结果进行完整审计。

### 3.3 产品原则

- AI 辅助而非无约束替代：关键测试资产由人工审核后生效；
- 证据优先：AI 结论必须标明需求、代码、知识库或运行数据来源；
- 确定性优先：可用规则、静态分析和测试工具完成的工作不交由大模型猜测；
- 安全优先：代码、需求、测试数据和日志按企业权限边界使用；
- 可降级：AI 不可用时，基础测试管理和执行能力仍可使用；
- 可度量：AI 生成内容的采纳率、准确率、覆盖率和节省时间均可统计。

## 4. 项目范围

### 4.1 本期范围

- Markdown 需求和技术方案上传、解析、版本管理；
- 需求文档附带 PNG/JPEG/WebP 页面原型图并进行多模态解析；
- 上传 HTML 交互原型包或登记受控原型地址，解析页面结构与交互流程；
- 项目知识库构建、增量同步和混合检索；
- 需求结构化分析、完整性分析、新旧版本差异分析；
- Git 代码仓库、提交、分支和 Pull/Merge Request 变更分析；
- 需求、技术模块、风险与测试点关联；
- 功能、接口和 UI 测试用例辅助生成；
- Playwright 自动化脚本生成、校验和受控执行；
- 测试计划、测试集、环境和执行任务管理；
- UI 定位器候选自愈与人工确认；
- 日志聚类、失败归因、缺陷建议和测试报告生成；
- 全链路证据引用、人工审核、版本管理和审计。

### 4.2 暂不纳入首期

- 完全无人值守地修改生产代码；
- 未经审核自动提交或合并 AI 生成的测试脚本；
- 未经确认永久写回自愈后的定位器；
- 取代专业性能测试、安全扫描和代码扫描工具；
- 仅凭大模型结论自动关闭缺陷或发布生产版本；
- 首期同时支持所有测试框架和所有文档格式。

## 5. 用户角色与权限

| 角色 | 主要权限 |
|---|---|
| 平台管理员 | 租户、模型、知识库、集成、权限和审计配置 |
| 项目管理员 | 项目成员、仓库、环境、流水线和质量门禁配置 |
| 产品/需求人员 | 上传需求、查看需求分析、确认歧义和验收标准 |
| 测试负责人 | 审核测试策略、分配任务、批准测试计划和报告 |
| 测试工程师 | 编辑用例、审核 AI 内容、执行测试、确认缺陷 |
| 开发/架构人员 | 查看影响分析、代码关联、失败证据和根因建议 |
| 只读/审计人员 | 查看授权项目数据、报告、证据和审计记录 |
| Agent 服务账号 | 在最小授权范围内读取资料和执行指定工具 |

平台采用租户、项目、知识库和环境四级权限隔离。生产环境执行、敏感数据访问和资产发布必须具备独立权限。

## 6. 核心业务对象

平台至少管理以下对象：

- 项目与版本；
- 需求文档、附件、原型图、需求版本与结构化需求项；
- 技术方案与架构资料；
- 代码仓库、分支、提交和变更集；
- 知识库文档、Chunk、向量和证据快照；
- 测试策略、测试点、测试用例、数据集和自动化脚本；
- 测试计划、测试套件、执行任务和运行环境；
- 执行步骤、日志、截图、视频、Trace 和覆盖率；
- 失败簇、根因分析、缺陷与处理状态；
- 测试报告、质量门禁和发布结论；
- AI 任务、Prompt/模型版本、引用、审核和反馈记录。

## 7. 全流程业务需求

### 7.1 需求接入与管理

#### 7.1.1 文档接入

平台应支持：

- 上传 Markdown 需求文档；
- 上传需求时可附带一张或多张页面原型图；
- 从配置的本地目录自动监听 Markdown 新增、修改、删除；
- 记录主文档与附件的来源、内容 Hash、版本、上传人和更新时间；
- 同一路径覆盖、新建版本和拒绝覆盖三种冲突策略；
- 上传完成后异步解析并展示处理状态；
- 用户明确发布后才将需求或分析方案同步至正式知识库。

#### 7.1.2 结构化需求解析

系统应从 PRD 中识别并结构化：

- 业务背景、目标和范围；
- 角色、权限和前置条件；
- 功能需求和业务规则；
- 数据字段、状态和生命周期；
- 接口、事件和外部依赖；
- 性能、安全、兼容性和可用性要求；
- 验收标准、限制条件、歧义和待确认问题。

每个需求项需保留原文位置、文档版本和引用信息，并允许人工编辑、拆分、合并和确认。

#### 7.1.3 页面原型图解析

上传需求文档时，用户应能够附带 PNG、JPEG 或 WebP 页面原型图，并可在 Markdown 中使用相对路径引用图片，例如 `![登录页](assets/login.png)`。系统应支持：

- 将主文档与附件作为同一个“需求包”保存和版本化；
- 按 Markdown 图片引用、文件名或人工绑定关系关联原型图与需求章节；
- 生成图片缩略图并保留原图，记录尺寸、Hash、来源和解析状态；
- 使用 OCR 与多模态模型识别页面名称、区域、控件、文本、表格和交互提示；
- 识别输入框、按钮、选择器、弹窗、列表、分页、空状态、错误状态和权限差异；
- 提取必填、长度、格式、默认值、禁用条件和可见性等候选规则；
- 将图片解析结果与文字需求合并，发现图文不一致和缺失说明；
- 根据页面控件和状态生成 UI 测试点，但明确标记图片推断内容；
- 用户可以在可视化页面中框选、纠正或确认识别结果。

原型图只能说明页面可见信息，不能单独证明后端业务规则。AI 从图片推断出的交互、校验和状态必须标记置信度；文字需求与图片冲突时不得静默选择其一，应生成待确认问题。

#### 7.1.4 HTML 交互原型解析

平台应将 HTML 交互原型作为可执行原型处理，而不是当作静态图片。首期支持上传包含 `index.html` 及本地资源的 ZIP 原型包，后续可支持登记经过授权的 HTTP(S) 原型地址。系统应支持：

- 安全解压原型包，拒绝路径穿越、符号链接、超量文件和压缩炸弹；
- 在一次性隔离浏览器中加载原型，默认禁止访问公网、内网和宿主文件系统；
- 采集 DOM、可访问性树、可见文本、控件属性、页面截图和控制台错误；
- 识别页面、路由、表单、弹窗、菜单、分页、状态切换和候选交互路径；
- 在明确的步数、页面数和时间预算内遍历安全交互；
- 对导航、输入、选择、展开等低风险动作生成交互轨迹；
- 对删除、提交、付款、授权等语义上的高风险动作只记录候选，不自动触发；
- 将交互轨迹转换为页面状态图和 UI 测试点；
- 将文字需求、静态原型图和 HTML 行为进行三方一致性检查；
- 允许用户录制或指定关键流程，以弥补自动遍历无法覆盖的复杂交互。

每个分析结论应引用原型版本、入口页面、DOM/可访问性定位信息、交互轨迹和对应截图。HTML 中的脚本与文本均是不可信输入，不得调用 Agent 工具、读取平台凭据或突破网络隔离。

#### 7.1.5 Markdown 在线查看与编辑

用户应能在平台内查看和编辑当前需求包及有权限访问的相关 Markdown 文档。系统应支持：

- Markdown 源码、渲染预览和分屏编辑三种模式；
- 标题目录、全文搜索、行号、语法高亮和图片相对路径预览；
- 编辑需求主文档、设计方案和用户有写权限的知识库文档；
- 保存前校验 UTF-8、文件大小、Markdown 引用和附件完整性；
- 显示未保存状态，支持手动保存、自动草稿和离开页面提醒；
- 每次正式保存生成不可变文档版本，记录作者、时间、修改说明和内容 Hash；
- 支持历史版本查看、Diff、恢复为新版本，不直接覆盖历史记录；
- 使用版本号或 ETag 进行乐观锁控制，发生并发编辑时提示冲突合并；
- 根据权限区分只读、可编辑、可发布和可删除；
- 编辑需求文档后将已有分析标记为可能过期，由用户选择重新分析；
- 编辑正式知识库文档后进入既有增量向量同步任务，不在保存请求内等待 Embedding；
- AI 提出的修改必须先以 Diff 展示，经用户确认后才写入新版本。

保存需求分析工作区文档不等于发布到正式知识库。发布仍需独立操作和权限审核。

### 7.2 AI 需求分析

AI 在该阶段扮演“查漏补缺者”和“规则校验者”。系统应支持：

- 根据历史需求、业务规范和验收标准检索相关资料；
- 联合分析文字需求与页面原型图，识别页面状态、交互路径和图文冲突；
- 解析 HTML 原型的真实页面结构和受控交互轨迹，补充状态及流程测试点；
- 检测需求中的缺失字段、歧义、矛盾和不可测试描述；
- 识别正常流程、异常流程、边界值、状态转换和权限差异；
- 将需求转换为功能树、思维导图或状态机模型；
- 比较需求新旧版本，输出新增、修改、删除和影响项；
- 生成需求覆盖清单和待确认问题；
- 对每条分析结论提供来源、置信度和是否需要人工确认的标记。

需求分析结果未经审核不得作为正式测试基线。

#### 7.2.1 一键需求 AI 分析

需求详情页应提供“一键 AI 分析”操作。触发前系统固定当前需求文档、附件、原型和目标知识库版本快照，随后异步执行：

```text
文档与附件完整性检查
  → 需求结构化提取
  → 图片/HTML 原型融合
  → 知识库检索
  → 相关需求、冲突、缺口和影响分析
  → 测试点与验收建议
  → 生成分析报告或设计方案草稿
```

用户可选择分析模板、目标知识库和分析范围。平台应展示当前阶段、百分比、耗时、取消和失败重试；同一文档版本与相同配置的重复点击应复用已有任务或要求确认，不得无提示重复计费。文档在任务期间发生修改时，本次结果仍绑定触发时版本，并提示新版本需要重新分析。

#### 7.2.2 上下文 AI 对话

需求详情页应提供与当前需求关联的 AI 对话能力。对话可在用户授权范围内引用：

- 当前需求文档及选中的历史版本；
- 页面原型图、HTML 原型状态和交互轨迹；
- 结构化需求项、分析结论和生成方案；
- 用户选择的项目知识库、代码变更、用例和测试结果。

AI 回答中的项目事实必须提供可点击引用，定位到文档版本、行号/Chunk、图片区域、HTML 状态或代码位置。检索不到依据时应明确说明。用户可以选择一段 Markdown 或某个需求项作为局部上下文，并要求 AI 解释、查漏、生成验收标准、提出测试点或给出修改建议。

对话中的修改建议默认只生成 Patch/Diff，不直接改写文档；用户确认后才创建新版本。会话应支持新建、重命名、历史查看、继续对话、删除和导出，并记录模型、引用快照及工具调用。切换项目或权限被撤销后，历史消息不得继续泄露已无权访问的原文。

### 7.3 AI 技术方案与代码影响分析

#### 7.3.1 技术方案分析

平台应结合需求与项目知识库分析：

- 关联已有架构、模块、接口、数据表和业务规则；
- 识别与现有方案的复用点、冲突点和缺口；
- 分析兼容性、数据迁移、权限、安全和运维风险；
- 输出受影响组件、上下游依赖和建议测试类型；
- 生成带证据引用的测试策略草案。

#### 7.3.2 代码变更分析

平台应接入 Git 仓库并支持：

- 选择分支、Commit、Pull Request 或 Merge Request 作为分析范围；
- 读取代码 Diff、文件变化、函数调用关系和依赖图；
- 结合静态分析、代码覆盖率和历史缺陷识别高风险变更；
- 将变更文件和函数关联到需求、模块及已有测试用例；
- 推荐新增测试、需要更新的测试以及可执行的最小回归集；
- 展示推荐依据，不允许只返回无法解释的风险分数。

静态分析结果是确定性证据，LLM 负责解释和补充，不应替代编译器、类型检查器或专业扫描工具。

### 7.4 AI 测试策略与用例生成

#### 7.4.1 测试点生成

系统应按需求项生成测试点，至少覆盖：

- 主流程和备选流程；
- 正向、反向和异常场景；
- 等价类、边界值和组合条件；
- 状态迁移和幂等性；
- 角色、权限和数据隔离；
- 并发、超时、重试和故障恢复；
- 性能、安全、兼容性和可观测性；
- 受代码变更影响的回归场景。

#### 7.4.2 测试用例生成

每条用例至少包含：

- 用例编号、名称、优先级和测试类型；
- 关联需求、技术模块、风险和来源引用；
- 前置条件、环境和测试数据；
- 操作步骤与逐步预期结果；
- 清理步骤、自动化适用性和标签；
- AI 置信度、生成版本和审核状态。

系统应提供草稿、待审核、已批准、已拒绝和已废弃状态。AI 新生成或修改的用例必须以 Diff 形式供测试人员审核。

#### 7.4.3 接口测试用例

平台应读取 OpenAPI/Swagger 定义并辅助生成：

- 正常请求和典型业务流程；
- 必填缺失、非法类型、长度和边界值；
- 鉴权、越权和租户隔离；
- 状态码、响应 Schema 和业务错误码校验；
- 幂等、重复请求、并发、超时和限流；
- 上下游故障和重试场景；
- 可配置的测试数据和环境变量。

#### 7.4.4 UI 测试用例与脚本

平台应支持基于需求、页面 DOM、截图及设计稿生成 UI 测试建议，并优先输出 Playwright 脚本。生成脚本必须：

- 优先使用 `data-testid`、ARIA role、label 等稳定定位方式；
- 避免未经说明的固定等待时间；
- 包含明确断言、测试数据和清理逻辑；
- 通过格式化、语法检查、静态检查和沙箱试运行；
- 展示生成依据、代码 Diff 和潜在风险；
- 经人工批准后才能进入正式测试资产库。

### 7.5 测试计划与执行

#### 7.5.1 测试计划

系统应支持：

- 按版本、需求、变更集或发布任务创建测试计划；
- 由 AI 推荐测试范围、用例优先级和最小回归集；
- 人工调整后锁定计划版本；
- 配置环境、浏览器、并发度、重试策略和时间窗口；
- 对接 Jenkins、GitLab CI 或其他 CI/CD 流水线；
- 支持手工、接口、UI 和外部测试工具结果汇总。

#### 7.5.2 执行编排

平台应支持：

- 立即执行、定时执行和流水线触发；
- 并行、串行、依赖和失败终止策略；
- 实时显示任务、套件、用例和步骤状态；
- 采集日志、截图、视频、网络请求、Trace 和资源指标；
- 超时、重试、取消和中断恢复；
- 隔离执行环境及敏感测试数据；
- AI 服务不可用时继续执行已批准的确定性脚本。

#### 7.5.3 风险驱动执行

系统可结合以下信息推荐执行优先级：

- 最近代码变更及调用影响；
- 需求重要度和业务风险；
- 历史失败率和缺陷密度；
- 用例最近执行时间；
- 代码覆盖率和模块依赖；
- 环境和数据可用性。

推荐结果须可解释，测试负责人可人工覆盖。

### 7.6 UI 自动化自愈

当 UI 定位器失效时，Self-Healing Agent 可根据以下信息生成候选元素：

- 当前 DOM 与历史 DOM 快照；
- role、label、文本、属性和相邻元素；
- 页面截图和视觉位置；
- 用户操作上下文和目标断言。

自愈流程：

```text
定位失败
  → 收集 DOM/截图/上下文
  → 生成候选元素及相似度
  → 在当前运行中尝试候选
  → 标记本次结果为“自愈执行”
  → 生成定位器修改建议
  → 人工审核后写回正式脚本
```

控制要求：

- 低置信度时不得自动继续；
- 涉及删除、付款、授权等高风险操作时禁止自动自愈；
- 自愈成功不等于测试正常，报告必须单独展示；
- 未经审核不得永久修改测试脚本或基线；
- 保存旧定位器、新定位器、证据、置信度和审核记录；
- 自愈候选仍需通过后续业务断言，不能仅以“点击成功”作为通过依据。

### 7.7 AI 测试结果与根因分析

#### 7.7.1 失败聚类

平台应对大量失败结果进行归一化和聚类：

- 去除时间戳、随机 ID 等噪声；
- 综合错误类型、堆栈、日志模板、失败步骤和环境；
- 将相似失败聚合为失败簇；
- 展示影响的用例、环境、版本和首次出现时间；
- 允许人工合并、拆分和标记聚类结果。

#### 7.7.2 根因分析

AI 应在授权范围内关联：

- 测试步骤、断言和失败截图；
- 应用日志、错误堆栈和 Trace；
- API 请求响应和数据库诊断信息；
- 最近代码提交和变更模块；
- 历史相似缺陷和解决方案；
- 环境故障、测试数据和第三方依赖状态。

根因输出必须包含推测结论、证据、置信度、其他可能原因和建议验证动作。“疑似由某次提交导致”不得直接表述为已确认事实。

#### 7.7.3 缺陷辅助

系统应支持从失败簇生成缺陷草稿，包括：

- 标题、影响范围、严重程度建议；
- 环境、版本、复现步骤和实际/预期结果；
- 日志、截图、Trace 和关联用例；
- 疑似根因和相关 Commit；
- 去重建议和相似历史缺陷。

缺陷由人工确认后再提交至缺陷管理系统。

### 7.8 AI 测试报告

系统应生成面向不同角色的报告：

- 管理摘要：质量结论、发布建议和主要风险；
- 测试范围：需求、变更、测试类型和环境；
- 执行数据：计划数、执行数、通过率、失败率和阻塞率；
- 需求覆盖：需求到测试点、用例和结果的追踪矩阵；
- 缺陷分析：新增、遗留、严重程度、失败簇和趋势；
- 根因摘要：已确认原因与 AI 推测原因分开展示；
- 自愈情况：触发次数、成功率、待审核修改；
- 风险与建议：未覆盖项、低置信度结论和发布阻断项；
- 证据附件：日志、截图、引用和分析版本。

报告支持在线查看、Markdown/PDF 导出、版本对比和审核签署。发布结论由有权限的人员确认，AI 只能提供建议。

### 7.9 知识库与 RAG

平台知识库应包括：

- 历史需求、验收标准和业务规则；
- 技术方案、架构文档、API 定义和数据字典；
- 测试规范、测试用例和自动化脚本；
- 历史执行结果、缺陷、根因和解决方案；
- 项目术语、组件关系和权限规范。

知识库应支持 Markdown 上传与本地目录监听两种入口，并采用结构化 Chunk、Hash 增量 Embedding、版本原子切换和混合检索。AI 输出引用必须固定到具体文档版本和 Chunk。

未经审核的 AI 产物、临时需求和原始敏感日志默认不进入正式知识库。发布时需经过权限检查、脱敏和人工确认。

## 8. AI Agent 能力要求

### 8.1 Agent 类型

| Agent | 主要职责 |
|---|---|
| 需求分析 Agent | 需求提取、差异、歧义、边界和状态分析 |
| 技术分析 Agent | 架构关联、代码影响和风险分析 |
| 测试设计 Agent | 测试策略、测试点和用例生成 |
| 脚本生成 Agent | 接口/UI 自动化脚本生成与修复建议 |
| 执行编排 Agent | 测试集推荐、优先级和执行状态协调 |
| 自愈 Agent | 定位器候选识别和受控修复建议 |
| 诊断 Agent | 失败聚类、根因推测和缺陷草稿 |
| 报告 Agent | 数据汇总、风险摘要和报告生成 |

### 8.2 Agent 编排规则

- Agent 只能调用明确授权的工具；
- 每次任务记录输入、模型、Prompt 版本、工具调用、输出和审核结果；
- Agent 输出必须符合预定义 Schema；
- 高风险动作必须设置人工审批节点；
- 失败任务支持重试、取消、超时和断点恢复；
- Agent 不得把文档中的提示注入内容当作系统指令；
- 模型输出与工具执行结果必须分开保存；
- 关键判断支持规则校验或第二模型复核，但最终责任仍由人工承担。

## 9. 追溯与审核需求

平台应建立以下追踪链：

```text
需求项
  ↔ 技术模块/代码变更
  ↔ 风险与测试点
  ↔ 测试用例/自动化脚本
  ↔ 测试计划/执行结果
  ↔ 失败簇/缺陷
  ↔ 测试报告/发布结论
```

任一 AI 产物应能查看：

- 生成时间、生成者、模型和 Prompt 版本；
- 使用的知识库、代码和运行证据；
- AI 原始建议与人工修改 Diff；
- 审核人、审核时间和审核意见；
- 后续执行结果及用户反馈。

## 10. 外部系统集成

平台需预留或支持：

- GitHub、GitLab、Gitea 等 Git 服务；
- Jenkins、GitLab CI 等 CI/CD；
- Jira、禅道或其他需求与缺陷系统；
- OpenAPI/Swagger；
- Playwright，后续可扩展 Cypress；
- 日志、APM、Trace 和监控平台；
- SSO、LDAP/OIDC 和企业权限系统；
- 云端或私有化部署的 LLM、Embedding 和 Reranker；
- PostgreSQL/pgvector，规模扩大后可评估专用向量数据库。

集成凭据必须加密保存并按项目授权，禁止出现在日志和模型上下文中。

## 11. 非功能需求

### 11.1 安全与隐私

- 支持私有化部署和本地模型；
- 项目代码、需求和测试数据严格隔离；
- 对模型请求进行字段级脱敏和内容审计；
- 生产数据库默认只读且不向 Agent 直接暴露通用查询权限；
- 自动化执行使用沙箱、最小权限账号和网络白名单；
- 敏感操作具备审批、二次确认和完整审计；
- 数据按租户策略支持保留期限、归档和删除；
- 明确禁止将企业数据用于公共模型训练，除非获得授权。

### 11.2 可靠性

- AI 模型不可用时，测试资产管理和已批准脚本执行不受影响；
- 分析与执行任务均采用持久化队列；
- 长任务支持状态查询、重试、取消和断点恢复；
- 生成失败不得覆盖最后一个已审核版本；
- 外部系统故障不应导致核心任务状态丢失；
- 重复回调和重复任务必须幂等处理。

### 11.3 性能与容量

第一阶段建议目标：

- 普通 Markdown 上传接口在 2 秒内返回异步任务 ID；
- 95% 的普通需求文档在 5 分钟内生成首版分析；
- 10 万知识库 Chunk 下混合检索 P95 小于 500 ms；
- 支持至少 50 个并发测试执行任务，具体按执行资源池配置；
- 测试执行实时状态延迟不超过 3 秒；
- 大日志和视频采用对象存储，数据库保存索引及元数据。

最终容量和时延指标须根据部署规模及模型性能校准。

### 11.4 可用性

- 各 AI 结论以“事实、推测、建议、待确认”分类展示；
- 用户可查看引用原文、代码位置和运行证据；
- 支持在 AI 草稿和人工确认版本之间对比；
- 高风险操作提供明确提示，不能静默执行；
- 报告和用例支持搜索、过滤、批量操作和导出。

### 11.5 可观测性

平台应监控：

- Agent 任务成功率、耗时、重试和失败阶段；
- Token、Embedding、Reranker 使用量与费用；
- 检索召回数、引用有效率和无证据结论数；
- AI 用例采纳率、人工修改率和执行通过率；
- 自愈触发率、候选准确率和人工批准率；
- 测试执行队列、运行时间、失败率和环境稳定性；
- 报告生成耗时与用户反馈。

## 12. AI 风险控制

### 12.1 幻觉控制

- 项目事实必须有引用；
- 无证据时明确标记未知；
- 输出需通过结构 Schema 和引用存在性校验；
- 用规则、静态分析和实际执行验证 AI 生成代码；
- 高置信度不是事实确认，关键结论仍需人工审核。

### 12.2 生成代码的确定性控制

AI 生成脚本后必须依次经过：

```text
格式化
  → 语法/类型检查
  → 静态安全检查
  → 测试清单规则校验
  → 隔离环境试运行
  → 展示 Diff
  → 人工审核
  → 发布为正式脚本
```

### 12.3 数据安全控制

- 支持按任务选择本地或云端模型；
- 敏感项目可强制使用私有模型；
- 发送模型前过滤密钥、个人信息和生产数据；
- 对知识库检索结果执行权限过滤后才能进入模型上下文；
- 保留模型数据去向、使用策略和审计记录。

## 13. 核心页面需求

### 13.1 工作台

- 待审核 AI 任务；
- 当前测试计划和执行状态；
- 质量风险、阻断问题和趋势；
- 最近失败簇和自愈建议；
- 模型费用和任务健康状态。

### 13.2 需求分析页

- Markdown 源码/预览/分屏编辑、目录、保存状态和版本 Diff；
- 原文与结构化需求对照；
- 新旧版本差异；
- 功能树、状态机、歧义和缺口；
- 相关知识、影响模块和证据；
- 人工确认、修改和评论。
- 固定展示“一键 AI 分析”入口及任务进度；
- 提供可收起的上下文 AI 对话面板、会话列表和引用查看；
- AI 文档修改建议以 Diff 呈现，用户确认后保存新版本。

### 13.3 Markdown 文档中心

- 按项目、知识库、来源和文档状态浏览 Markdown；
- 支持只读查看、在线编辑、搜索、预览和版本历史；
- 显示同步状态、引用关系、关联需求和最后分析时间；
- 支持上传附件、修复失效引用和查看图片/HTML 原型；
- 按权限执行保存、发布、恢复和删除；
- 支持选择文档发起 AI 对话或需求分析。

### 13.4 测试设计页

- 需求、风险、测试点和用例四级关联视图；
- AI 批量生成与去重建议；
- 用例 Diff、覆盖矩阵和审核操作；
- 自动化脚本编辑、校验和试运行。

### 13.5 测试执行页

- 计划、套件、用例和步骤实时状态；
- 日志、截图、视频、网络和 Trace；
- 失败原因、自愈过程和重试控制；
- 并行度、环境和执行资源状态。

### 13.6 报告与诊断页

- 质量摘要和发布建议；
- 需求覆盖与执行趋势；
- 失败聚类和根因证据图；
- 缺陷草稿、自愈统计和风险列表；
- 报告审核、导出和版本对比。

### 13.7 系统管理页

- 模型、Prompt 模板和 Agent 工具配置；
- 知识库、仓库、CI/CD、缺陷系统和环境配置；
- 用户、角色、项目权限和审批流；
- 审计日志、费用额度和数据保留策略。

## 14. 关键状态模型

### 14.1 AI 产物状态

```text
draft → validating → pending_review → approved → published
   └──────────────> failed
pending_review → rejected → draft
published → deprecated
```

### 14.2 测试执行状态

```text
queued → preparing → running → passed/failed/blocked/cancelled
                         └────> retrying → running
```

### 14.3 根因结论状态

```text
suggested → investigating → confirmed/rejected
```

报告中必须明确区分 suggested 与 confirmed。

## 15. 核心指标

### 15.1 效率指标

- 需求到测试方案的平均时间；
- 单条用例平均编写和审核时间；
- 自动化脚本平均维护时间；
- 失败到初步根因建议的平均时间；
- 最小回归集相对全量用例的减少比例。

### 15.2 质量指标

- 需求项提取覆盖率；
- 需求到用例追踪覆盖率；
- AI 用例有效采纳率；
- 引用有效率和事实引用支持率；
- 回归缺陷漏测率；
- 根因 Top-3 建议命中率；
- 自愈候选人工确认准确率。

### 15.3 治理指标

- AI 产物人工审核覆盖率；
- 未经授权的数据访问次数；
- 无引用项目事实数量；
- AI 生成代码校验通过率；
- 模型费用、单任务成本和缓存复用率。

“脚本编写时间缩短 70%”和“维护成本接近于零”应作为业务假设，通过试点基线验证后再设为正式 KPI，不在需求阶段直接承诺。

## 16. 验收标准

### 16.1 全流程验收场景

给定一份新需求及对应代码变更，平台应能够：

1. 上传、解析并形成结构化需求项；
2. 在线查看和编辑 Markdown，保存后生成可追溯的新版本；
3. 点击一次操作发起完整 AI 需求分析并查看阶段进度；
4. 围绕当前需求和授权知识库进行带引用的 AI 对话；
5. 检索历史需求、规范和技术文档；
6. 输出需求差异、歧义、边界及待确认项；
7. 关联代码变更并给出可解释的影响模块和风险；
8. 生成测试点、功能用例、接口用例和 UI 脚本草稿；
9. 对生成脚本完成规则校验和沙箱试运行；
10. 经人工批准后形成测试计划并执行；
11. 采集日志、截图和运行证据；
12. 对失败进行聚类并生成带置信度的根因建议；
13. 生成需求覆盖、执行结果、缺陷和风险报告；
14. 全流程可追溯到需求、代码、用例、结果和知识库证据；
15. AI 不可用时仍可查看资产并执行已批准脚本。

### 16.2 安全验收

- 不同项目用户无法跨权限检索文档、代码和日志；
- 密钥和敏感数据不会出现在模型请求及普通日志中；
- 未审批的 AI 脚本不能进入正式执行计划；
- 自愈结果未经审核不能写回正式脚本；
- 所有高风险工具调用均可通过审计记录还原。

### 16.3 质量验收

- 每个 AI 生成测试点可追溯到需求或技术证据；
- 原型图生成的测试点可追溯到具体图片版本及标注区域；
- HTML 原型生成的测试点可追溯到原型版本、页面、元素和交互轨迹；
- 每个项目事实引用均能打开对应版本原文；
- 测试用例通过必填字段和规则校验；
- AI 生成脚本通过语法和静态检查后才能试运行；
- 根因报告区分事实、推测和待验证项；
- 通过固定评测集持续衡量需求覆盖、用例质量和根因命中率。

## 17. 分阶段建设计划

### Phase 1：AI 测试设计闭环（MVP）

- 项目、用户、角色和基础权限；
- Markdown 需求上传及知识库增量同步；
- 需求包及页面原型图上传、OCR/多模态解析和人工校正；
- HTML 原型 ZIP 上传、隔离渲染、DOM/可访问性解析和受控流程采集；
- Markdown 文档中心、在线编辑、预览、版本 Diff 和并发冲突控制；
- 一键需求 AI 分析、任务进度、取消与重试；
- 当前需求上下文 AI 对话、知识库引用和建议 Diff 确认；
- 结构化需求分析、知识检索和引用；
- 需求差异、边界、缺口和测试点生成；
- 功能与接口测试用例生成及人工审核；
- 测试资产、计划、手工结果和基础报告；
- AI 任务、版本、审计和反馈记录。

### Phase 2：代码与自动化执行

- Git Diff、静态分析和影响范围；
- 最小回归测试集推荐；
- Playwright 脚本生成、校验和沙箱试运行；
- 测试环境、执行器、CI/CD 和实时结果；
- 日志、截图、视频和 Trace 采集。

### Phase 3：智能诊断与自愈

- UI 定位器候选自愈及审核写回；
- 失败日志聚类和相似缺陷检索；
- 多源证据根因分析；
- 缺陷草稿和测试报告智能生成；
- 风险驱动执行和质量门禁建议。

### Phase 4：平台化与持续优化

- 多租户、资源池和成本治理；
- 多模型路由、私有模型和领域评测；
- 多测试框架、更多文档格式和项目管理集成；
- 基于人工反馈持续优化检索、模板和模型；
- 组织级质量指标和跨项目知识复用。

## 18. 依赖与前置条件

- 项目需求、技术文档和历史测试资产可合法接入；
- Git、CI/CD、缺陷系统具备可用 API 和服务账号；
- 测试环境可以被隔离执行器访问；
- 项目可提供稳定的 OpenAPI、测试账号和必要的 `data-testid`；
- 企业明确模型使用、数据分类、脱敏和保留策略；
- 建立由产品、测试、研发、安全共同参与的 AI 产物审核机制；
- 准备一套真实需求、用例、缺陷和根因组成的离线评测集。

## 20. 总结

本平台的核心不是用大模型替代测试人员，而是把 AI 作为贯穿全流程的分析和生产力层：以需求和代码为输入，以知识库和确定性工具为依据，以测试用例和执行器为行动载体，以日志、缺陷和报告形成反馈，并由人工在关键节点完成审核和决策。

产品建设应优先完成可追溯的测试设计闭环，再逐步扩展代码影响分析、自动化执行、自愈和根因诊断。任何 AI 能力都必须同时具备证据、校验、权限、审核、降级和度量机制，才能成为可在企业环境长期运行的测试平台。
