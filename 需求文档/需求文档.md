## 2. 项目背景

项目资料通常分散在需求文档、技术方案、接口说明、测试用例、测试报告和缺陷复盘中，缺少统一的版本、来源、同步状态和检索入口。后续 AI 需求分析与技术方案分析若没有稳定、可追溯的资料基础，就无法可靠引用项目事实。

SmartHub 采用“先知识库基础、后智能分析”的建设顺序：先将独立维护的项目资料沉淀为可版本化、可检索的知识资产，再以固定资产版本和证据定位作为后续 AI 能力的输入。

## 3. 产品目标

### 3.1 总体目标

建设一个以项目知识资产为基础的分析平台，使后续能力能够在明确资料范围内输出带来源引用的结论。

```text
项目资料
  ↓
项目知识库：接入、版本、同步、检索、定位
  ↓
Phase 2：AI 需求分析与人工确认
  ↓
Phase 3：AI 技术方案分析与人工评审
  ↓
后续：Git/代码、测试设计与执行、诊断、自愈、治理
```

### 3.2 分阶段目标

- **Phase 1**：建立可靠、可恢复、可追溯的项目知识库和知识库配置能力。
- **Phase 2**：基于固定的知识资产版本和检索证据，完成 AI 需求分析。
- **Phase 3**：基于已确认需求和项目知识库，完成 AI 技术方案分析。
- **后续阶段**：逐步接入 Git/代码影响、测试设计与执行、诊断、自愈、权限治理和企业集成。

### 3.3 产品原则

- **资料优先**：先保证资料来源、版本和检索定位正确，再建设 AI 分析能力。
- **版本优先**：检索、分析和引用必须指向确定的资产版本，不能以“当前最新内容”代替历史事实。
- **独立维护优先**：项目内独立维护、有来源和版本的资料是知识库资产；临时中间结果不是自动入库资料。
- **增量与可恢复**：资料更新只处理变化内容；失败不能破坏上一个可检索版本。
- **逐步演进**：第一期不以 Git、权限、测试执行或复杂多格式解析作为交付依赖。

## 4. 统一术语与资料边界

### 4.1 核心对象

```text
项目
  └─ 项目知识库
       └─ 知识资产
            └─ 资产版本
                 └─ Chunk / 索引版本
```

| 对象 | 定义 |
|---|---|
| 项目 | 资料、配置和后续智能分析的归属边界。 |
| 项目知识库 | 项目下统一管理、同步和检索资料的集合。第一期每个项目创建一个默认知识库。 |
| 知识资产 | 一份独立维护的项目资料逻辑实体，例如“支付模块重构需求”或“订单 API 规范”。 |
| 资产版本 | 知识资产某次不可变内容快照，记录来源、内容 Hash、时间和同步状态。 |
| Chunk | 资产版本经解析后用于检索与定位的结构化片段。 |
| 索引版本 | 同一知识库在固定解析、切分和 Embedding 配置下的可检索集合版本。 |
| AI 派生产物 | 后续 AI 产生的分析 findings、对话消息、方案草稿和任务中间结果。它们关联输入资产版本，但不会自动成为知识资产。 |

### 4.2 知识资产范围

以下独立维护的项目资料均属于项目知识库内容，并按资料类型、来源、版本和状态管理：

- 各版本需求文档、验收标准和业务规则；
- 技术方案、架构资料、接口文本/规范和数据字典；
- 测试规范、测试用例和测试报告；
- 缺陷复盘、问题解决方案和项目术语说明；
- 其他具有明确来源、维护责任和版本记录的项目说明资料。

第一期完整支持 UTF-8 Markdown 与纯文本接入。接口定义、测试用例和报告可先以 Markdown/纯文本形式维护；OpenAPI、Excel、PDF、Word 等专用格式解析器属于后续扩展，不影响统一资产模型。

### 4.3 统一边界

本产品不再以“分析工作区”和“正式知识库”区分两套资料库。资料是否进入知识库，取决于它是否已经被保存为**独立维护的项目资料版本**：

- 人工创建、导入或确认保存的需求、技术方案、API 说明、测试用例和报告，均为知识资产，可按通用同步流程进入检索。
- AI 分析结论、会话消息、临时草稿和任务中间结果不因产生而自动成为知识资产。
- 后续如将 AI 草稿人工保存为独立技术方案或报告版本，则该版本作为新的知识资产进入同步流程。

## 5. 产品阶段与范围

### 5.1 Phase 1：项目知识库构建与配置

**目标**：建立项目资料的接入、版本、增量同步、基础检索、配置和状态可观测能力，为后续分析提供可靠输入。

**本期范围**：

- 项目和默认项目知识库创建、查看与配置；
- Markdown/纯文本资料上传与受控本地目录监听；
- 统一知识资产类型、来源、逻辑路径、版本和状态管理；
- 内容校验、冲突策略、不可变资产版本和活动版本管理；
- 结构化切分、内容 Hash、增量 Embedding、同步任务、失败重试和来源对账；
- 知识库资料浏览、按类型/来源/状态过滤、基础关键词/向量检索和原文定位；
- Chunk、Embedding、检索参数和索引重建配置；
- 同步任务、错误、重试、索引版本和配置快照的可观测展示。

**本期明确不做**：

- AI 需求分析、AI 对话、原型图/HTML 原型解析；
- AI 技术方案生成或技术影响分析；
- Git、代码仓库、分支、Commit、Pull Request、静态分析和代码影响；
- 测试点/用例/脚本生成、Playwright、测试执行、CI/CD、日志诊断和自愈；
- 用户角色、细粒度权限、ACL、审批流、SSO 和审计治理；
- OpenAPI、Excel、PDF、Word 等专用格式解析器。

### 5.2 Phase 2：AI 需求分析

在 Phase 1 知识库稳定后，基于已入库的需求资产版本与选定检索范围，完成结构化需求提取、资料关联、缺口/歧义/冲突/边界分析、证据引用和人工确认。

Phase 2 不包含技术方案生成、Git/代码分析、测试脚本生成、测试执行和权限治理。

### 5.3 Phase 3：AI 技术方案分析

基于已确认的需求分析结果和项目知识资产，生成可评审的技术方案草稿。方案需区分项目资料事实、技术建议、无证据假设和待确认风险，并保存固定的输入与证据快照。

Phase 3 不接入 Git、代码 Diff、函数调用图、静态分析、CI/CD 或测试执行器。

### 5.4 Phase 4+：后续能力

以下能力在前三期完成后单独规划，不构成当前阶段依赖：

- Git/代码影响分析、分支与变更集关联；
- 测试设计、测试用例/脚本生成、Playwright 与其他框架；
- 测试计划、执行器、CI/CD、日志/截图/Trace 采集；
- 失败聚类、根因分析、质量报告和自愈；
- 多知识库、用户/角色/ACL、审批、审计、SSO、组织级治理；
- OpenAPI、Excel、PDF、Word、外部系统和更多资料来源适配。

## 6. 跨阶段数据与证据约束

后续阶段必须复用 Phase 1 的资产模型，不得另建无法关联版本的资料容器。

- 检索结果至少返回 `projectId`、`knowledgeBaseId`、`assetId`、`assetVersionId`、资料类型、标题路径、`chunkId` 和原文定位信息；
- AI 分析任务必须保存输入资产版本、检索范围、配置版本和证据快照；
- 资产版本更新后不覆盖历史引用；旧结论仍能按原快照查看，并可被标记为可能过期；
- 后续权限设计不得改变资产、版本、Chunk 和证据引用的稳定标识；
- Git 与外部系统资料未来作为新的来源类型接入，不得破坏 Phase 1 的上传/目录监听同步语义。

## 7. 第一阶段核心页面

### 7.1 项目知识库页

- 查看项目默认知识库、资料数量、资料类型分布和索引状态；
- 按资料类型、来源、同步状态和逻辑路径浏览知识资产；
- 查看资产版本、当前活动版本、最近成功同步时间和最近错误；
- 上传 Markdown/纯文本资料，或查看受控目录资料的同步结果；
- 执行基础检索并定位至固定资产版本、标题路径和原文位置；
- 查看或重试失败同步任务。

### 7.2 知识库配置页

- 配置默认资料类型、受控目录、允许的接入来源和冲突策略；
- 配置 Markdown/纯文本解析、Chunk 大小、重叠和预处理版本；
- 配置 Embedding Provider、模型、维度和基础检索参数；
- 显示草稿/已保存状态及配置变更影响；
- 对影响索引兼容性的配置发起受控重建，并展示旧索引持续可用、进度、失败或取消状态。

## 8. 第一阶段验收总览

第一阶段详细验收场景见 [第一期-项目知识库构建与配置需求文档.md](第一期-项目知识库构建与配置需求文档.md)。总体验收以以下事实为准：

1. 项目资料可以按统一知识资产模型接入、分类、版本化和同步；
2. 未变化内容不会重复产生活动 Chunk 或重复 Embedding；
3. 同步失败、取消或重建失败时，上一个活动索引仍可检索；
4. 检索结果可定位至确定资产版本与原文位置；
5. 配置变更可追踪，影响索引的变更通过异步重建生效；
6. 验收不依赖 AI、Git、测试执行、权限或外部系统能力。

## 9. 依赖与前置条件

第一阶段仅需要：

- 可用于试点的 Markdown/纯文本项目资料；
- 已确定的资料类型分类和逻辑路径约定；
- 可访问的 Embedding 服务或本地 Embedding 运行环境；
- 受控上传存储或受控本地资料目录；
- 用于验证检索和同步行为的样例资料集。

Git API、CI/CD、测试环境、测试账号、`data-testid`、角色权限策略和外部缺陷系统均不是第一阶段前置条件。

## 11. 待确认事项

1. 每个项目首期允许接入的 Markdown/纯文本资料规模和保留周期；
2. 资料类型、逻辑路径和维护责任的项目约定；
3. 首期选用的 Embedding 服务、模型、维度和模型数据存放边界；
4. 受控目录监听与网页上传是否同时启用，以及两者覆盖同一逻辑路径时的默认冲突策略；
5. 第一阶段检索质量评测的样例问题、期望来源和验收阈值；
6. OpenAPI、Excel、PDF、Word 等专用格式解析器的后续接入优先级。
