## 2. 项目背景

项目资料通常分散在需求文档、技术方案、接口说明、测试用例、测试报告和缺陷复盘中，缺少统一的版本、来源、同步状态和检索入口。后续 AI 需求分析与技术方案分析若没有稳定、可追溯的资料基础，就无法可靠引用项目事实。

SmartHub 采用“先知识库基础、后智能分析”的建设顺序：先将独立维护的项目资料沉淀为可版本化、可检索的知识资产，再以固定资产版本和证据定位作为后续 AI 能力的输入。

## 3. 产品目标

### 3.1 总体目标

建设一个以项目知识资产为基础的分析平台，使后续能力能够在明确资料范围内输出带来源引用的结论。

```text
项目资料
  ↓
项目知识库：接入、版本、同步、检索、定位
  ↓
Phase 2：启用 RequirementAnalysisAgent 进行需求分析与评审
  ↓
Phase 3：启用 TechnicalSolutionAnalysisAgent 进行技术方案分析
  ↓
Phase 4：多个测试 Agent 协作完成用例设计、编写与执行
  ↓
后续：Git/代码、诊断、自愈、治理与企业集成
```

### 3.2 分阶段目标

- **Phase 1**：建立可靠、可恢复、可追溯的项目知识库和知识库配置能力。
- **Phase 2**：在 Pi Agent Framework 上创建并启用一个 `RequirementAnalysisAgent`，基于固定知识资产和证据完成需求分析、结构化 Finding 与人工处置闭环。
- **Phase 3**：在同一框架上创建并启用一个独立的 `TechnicalSolutionAnalysisAgent`，基于已确认需求和项目知识库完成技术方案分析与评审。
- **Phase 4**：创建测试分析、测试用例设计、测试脚本编写和测试执行等多个 Agent，通过显式多 Agent 工作流完成测试设计、编写和执行。
- **后续阶段**：逐步接入 Git/代码影响、失败诊断、自愈、权限治理和企业集成。

### 3.3 产品原则

- **资料优先**：先保证资料来源、版本和检索定位正确，再建设 AI 分析能力。
- **版本优先**：检索、分析和引用必须指向确定的资产版本，不能以“当前最新内容”代替历史事实。
- **项目版本隔离**：项目知识库与系统配置是项目级共享资源；需求评审、技术方案、测试执行和报告等项目空间数据必须归属明确的项目版本。未创建或未选择项目版本时，不得进入需求分析工作台或启动分析。
- **单平台单项目**：当前产品形态固定为一个 SmartHub 平台实例服务一个项目，不提供项目创建、项目选择和跨项目切换。平台初始化时自动建立当前项目及其默认知识库，用户只管理该项目下的项目版本。
- **独立维护优先**：项目内独立维护、有来源和版本的资料是知识库资产；临时中间结果不是自动入库资料。
- **增量与可恢复**：资料更新只处理变化内容；失败不能破坏上一个可检索版本。
- **逐步演进**：第一期不以 Git、权限、测试执行或复杂多格式解析作为交付依赖。
- **确定性外层、Agent 内层**：业务状态、快照、取消、路由、校验、正式落库和高风险审批由 SmartHub 控制；Agent 只在单次运行内调用已授权工具。
- **最小权限与证据优先**：Prompt、模型、工具、MCP、联网和外部操作必须版本化并按风险授权；外部内容不能绕过固定证据与人工确认规则。
- **统一 Agent 框架、独立业务 Agent**：Phase 2～Phase 4 统一使用 Pi Agent Framework；各阶段创建职责单一、独立版本化的业务 Agent，不以一个万能 Agent 覆盖全部流程。
- **单 Agent 先行、多 Agent 演进**：Phase 2 和 Phase 3 分别由一个业务 Agent 完成核心分析；Phase 4 再引入明确角色、任务依赖和产物交接的多 Agent 协作。

## 4. 统一术语与资料边界

### 4.1 核心对象

```text
平台实例（单项目）
  └─ 当前项目（系统初始化，不由用户切换）
       ├─ 项目知识库（项目级共享）
       │    └─ 知识资产
       │         └─ 资产版本
       │              └─ Chunk / 索引版本
       └─ 项目版本（项目空间隔离边界）
            └─ 项目版本需求绑定
                 └─ 引用固定需求资产版本
```

| 对象 | 定义 |
|---|---|
| 项目 | 平台实例内唯一的资料、配置和智能分析归属边界，由系统初始化；当前版本不提供项目管理和切换能力。 |
| 项目版本 | 项目空间内需求评审及后续设计、执行、报告数据的强制隔离边界，状态为 `open`、`locked` 或 `archived`。只有 `open` 版本允许新增或替换需求绑定并发起新分析。 |
| 项目知识库 | 当前唯一项目下统一管理、同步和检索资料的集合。平台初始化时创建一个默认知识库。 |
| 知识资产 | 一份独立维护的项目资料逻辑实体，例如“支付模块重构需求”或“订单 API 规范”。 |
| 资产版本 | 知识资产某次不可变内容快照，记录来源、内容 Hash、时间和同步状态。 |
| Chunk | 资产版本经解析后用于检索与定位的结构化片段。 |
| 索引版本 | 同一知识库在固定解析、切分和 Embedding 配置下的可检索集合版本。 |
| 项目版本需求绑定 | 将项目版本与一个确定的需求 `assetVersionId` 关联的引用关系。它不复制知识资产内容，用于决定该项目版本可分析哪些需求版本。 |
| 需求评审运行 | 对固定需求资产、索引、检索、模型、Agent、Prompt、工具与权限快照执行的一次可取消、可追溯运行。 |
| Pi Agent Framework | SmartHub Phase 2～Phase 4 共用的 Agent 框架，提供模型交互、消息状态、工具循环、事件和取消传播等基础能力。 |
| 业务 Agent | 基于 Pi Agent Framework 创建的版本化业务执行单元，具有稳定 ID、类型、Prompt、模型配置、工具、权限和执行限制；不同阶段使用独立业务 Agent。 |
| `RequirementAnalysisAgent` | Phase 2 唯一业务 Agent，负责需求分析、证据检索、Finding 生成和评审问答。 |
| `TechnicalSolutionAnalysisAgent` | Phase 3 独立业务 Agent，负责基于已确认需求和项目知识生成、校验并提交技术方案分析结果。 |
| 多 Agent 工作流 | Phase 4 用于协调多个测试 Agent 的显式任务图，固定参与 Agent、输入、依赖、产物、审批和汇总规则，不依赖自由对话决定流程。 |
| 受控工具 / MCP 适配器 | 具有输入输出 Schema、权限、风险、超时、幂等和数据策略的确定性能力；MCP Tool 必须转换为统一工具描述后才能使用。 |
| AI 派生产物 | 后续 AI 产生的分析 findings、对话消息、方案草稿和任务中间结果。它们关联输入资产版本，但不会自动成为知识资产。 |

### 4.2 知识资产范围

以下独立维护的项目资料均属于项目知识库内容，并按资料类型、来源、版本和状态管理：

- 各版本需求文档、验收标准和业务规则；
- 技术方案、架构资料、接口文本/规范和数据字典；
- 测试规范、测试用例和测试报告；
- 缺陷复盘、问题解决方案和项目术语说明；
- 其他具有明确来源、维护责任和版本记录的项目说明资料。

第一期完整支持 UTF-8 Markdown 与纯文本接入。接口定义、测试用例和报告可先以 Markdown/纯文本形式维护；OpenAPI、Excel、PDF、Word 等专用格式解析器属于后续扩展，不影响统一资产模型。

### 4.3 统一边界

本产品不再以“分析工作区”和“正式知识库”区分两套资料库。资料是否进入知识库，取决于它是否已经被保存为**独立维护的项目资料版本**：

- 人工创建、导入或确认保存的需求、技术方案、API 说明、测试用例和报告，均为知识资产，可按通用同步流程进入检索。
- AI 分析结论、会话消息、临时草稿和任务中间结果不因产生而自动成为知识资产。
- 后续如将 AI 草稿人工保存为独立技术方案或报告版本，则该版本作为新的知识资产进入同步流程。
- 项目知识库、知识资产、模型来源和系统管理配置为项目级或系统级共享资源；需求评审及后续设计、执行、报告数据必须按 `projectVersionId` 隔离。
- 项目版本只通过绑定引用固定资产版本，不复制知识资产。知识资产产生新版本时，不得自动改变既有项目版本的绑定。
- 未创建或未选择项目版本时，需求分析入口只展示“新建项目版本”引导，不展示其他版本的需求、评审、运行、Finding 或问答数据。
- `projectId` 可作为服务端内部归属键保留，但由部署配置或系统初始化确定，不出现在普通用户的项目选择、创建流程中，也不由前端任意提交。

## 5. 产品阶段与范围

### 5.1 Phase 1：项目知识库构建与配置

**目标**：建立项目资料的接入、版本、增量同步、基础检索、配置和状态可观测能力，为后续分析提供可靠输入。

**本期范围**：

- 平台初始化唯一项目及其默认知识库，用户查看与配置当前项目知识库；本期不提供项目创建和项目切换；
- Markdown/纯文本资料通过网页/API 受控上传；未来 Wiki 等来源通过标准连接器接入；
- 统一知识资产类型、来源、逻辑路径、版本和状态管理；
- 内容校验、冲突策略、不可变资产版本和活动版本管理；
- 结构化切分、内容 Hash、增量 Embedding、持久化异步任务、失败重试和中断恢复；
- 知识库资料浏览、按状态/逻辑路径过滤、基础关键词/向量检索和固定版本原文定位；资料类型仅作为资产和检索证据元数据展示；
- 检索范围仅支持逻辑路径筛选；来源保留为资产和证据元数据，不作为筛选条件；
- Chunk、Embedding、检索参数和索引重建配置；
- 同步任务、错误、重试、索引版本和配置快照的可观测展示。

**本期明确不做**：

- AI 需求分析、AI 对话、原型图/HTML 原型解析；
- AI 技术方案生成或技术影响分析；
- Git、代码仓库、分支、Commit、Pull Request、静态分析和代码影响；
- 测试点/用例/脚本生成、Playwright、测试执行、CI/CD、日志诊断和自愈；
- 用户角色、细粒度权限、ACL、审批流、SSO 和审计治理；
- OpenAPI、Excel、PDF、Word 等专用格式解析器。

### 5.2 Phase 2：需求评审与大模型配置

**目标**：在 Phase 1 知识库稳定后，以 Pi Agent Framework 为统一框架，创建并启用首个业务 Agent——`RequirementAnalysisAgent`。该 Agent 基于固定需求资产版本和证据范围完成可执行、可复核、可人工处置的需求分析与评审，并使用统一的生成式模型配置、路由、工具和运行观测能力。

**本期范围**：

- 创建、选择并切换项目版本；只有 `open` 项目版本允许绑定需求和发起新评审，`locked`、`archived` 版本只读；
- 将项目知识库中 `ready` 的固定需求资产版本绑定到当前项目版本；新项目版本可选择继承上一版本的有效需求绑定，但不继承评审、运行、Finding 和问答结果；
- 需求评审页支持上传 Markdown、纯文本和 ZIP 压缩包，复用 Phase 1 的校验、解析、索引流程，并在资产版本达到 `ready` 后自动绑定到当前项目版本；ZIP 中图片只作为附件保存，不参与本期正文解析；
- 从当前项目版本已绑定的固定需求资产版本创建评审，固定 `projectVersionId`、绑定关系、输入 Hash、活动索引、检索范围和检索结果快照；
- 每个需求评审运行只启动一个 `RequirementAnalysisAgent` 执行实例，由它完成结构化需求摘要、功能树以及缺口、歧义、冲突、边界、验收风险和测试风险识别；
- Finding 关联固定资产版本、Chunk、标题路径和原文位置；无充分证据的内容标记为待确认、模型推测或外部参考；
- Finding 人工确认、驳回、解决、待跟进和补充评论，保留模型初始输出、历史运行和处置记录；
- 基于成功评审运行开展有固定上下文和证据引用的评审问答；
- 受控注册生成式模型来源、模型能力、模型配置版本、智能路由、重试、降级、健康检测和调用观测；
- Pi Agent Framework 负责 `RequirementAnalysisAgent` 的模型交互、消息状态、工具循环和取消传播；PostgreSQL Job/Worker 负责排队、租约、恢复、取消和最终业务状态；
- 固定 Pi Agent Framework/Runtime、`agentId`、Agent 类型与定义、Prompt、工具集、MCP、网络策略、权限和执行限制版本，并记录模型与工具调用；
- 通过统一 Tool Runtime 受控使用内部检索、联网与 MCP 能力；工具按 `read`、`write-reversible`、`write-high-risk` 分级，写操作遵循幂等、补偿和人工确认策略；
- 对提示词注入、SSRF、越权工具调用、凭据泄露、迟到结果和无限 Agent 循环实施明确阻断与观测。

**本期产品非目标**：

- 技术方案生成、Git/代码分析、测试脚本生成、测试执行和完整组织权限治理；
- 面向普通用户的通用可视化 Prompt/Agent 编排、任意 Agent 发布、任意工具/MCP/联网端点注册；
- 多 Agent 协作和 Agent 间任务委派；第二期只启用一个 `RequirementAnalysisAgent`；
- 浏览器录入任意外部密钥、Shell/代码执行环境，以及不受控外部写操作；
- AI 结果、问答或外部操作自动改写需求原文、正式知识资产、Finding 人工处置或活动索引。

上述产品非目标不影响第二期将 Pi Agent Framework 作为正式业务框架并启用 `RequirementAnalysisAgent`，也不排除该 Agent 依赖受控 Prompt、Tool Runtime、联网、MCP 和外部操作执行器；所有依赖必须预注册、版本化、最小授权、可取消、可审计，并且不能绕过 SmartHub 业务状态和结果校验。

### 5.3 Phase 3：技术方案分析 Agent

**目标**：在 Pi Agent Framework 上创建并启用一个独立的 `TechnicalSolutionAnalysisAgent`，基于 Phase 2 已确认的需求 Finding、人工处置、固定证据和项目知识资产生成可评审的技术方案分析结果。

- `TechnicalSolutionAnalysisAgent` 与 `RequirementAnalysisAgent` 使用不同的稳定 ID、类型、Agent 定义、Prompt、工具集、权限和执行限制，不通过修改第二期 Agent 的 Prompt 来兼任；
- 每次方案分析只启动一个技术方案分析 Agent，固定需求评审运行、知识资产、模型、Agent、Prompt、工具和证据快照；
- 输出区分项目资料事实、技术建议、无证据假设和待确认风险，并通过受控结果提交和人工评审形成方案版本；
- 复用 Pi Agent Framework、模型路由、Tool Runtime、MCP/联网治理、调用观测和人工确认基础能力；
- Phase 3 不引入多 Agent 协作，也不以 Git、代码 Diff、函数调用图、静态分析、CI/CD 或测试执行器作为必需能力。

### 5.4 Phase 4：测试设计、编写与执行多 Agent

**目标**：在 Pi Agent Framework 上创建多个职责独立的测试业务 Agent，通过显式多 Agent 工作流完成需求到测试结果的设计、编写、执行和汇总闭环。

首期角色至少包括：

- `TestAnalysisAgent`：读取已确认需求和技术方案，拆分测试范围、风险、测试点与覆盖目标；
- `TestCaseDesignAgent`：根据测试点设计结构化测试用例、前置条件、步骤、期望结果和优先级；
- `TestScriptAuthoringAgent`：将已确认用例编写为受控测试脚本或自动化资产，并提交 Patch/Diff 供审核；
- `TestExecutionAgent`：在隔离测试环境中执行已批准脚本，采集状态、日志、截图、Trace 和结果证据；
- `TestResultAnalysisAgent`：汇总执行结果、失败分类、覆盖缺口和待确认问题，不自动篡改原用例或脚本。

多 Agent 工作流必须显式定义任务拓扑、参与 Agent 版本、输入输出 Schema、依赖、并发、超时、重试、审批、共享产物和汇总规则。Agent 间通过版本化产物和结构化交接记录协作，不允许依靠自由对话决定发布、执行和高风险操作。

### 5.5 Phase 5+：后续能力

以下通用产品能力后续单独规划。底层 Agent、Tool、联网或 MCP 适配器可以被前述阶段受控复用，但不代表对应通用产品能力已经交付：

- Git/代码影响分析、分支与变更集关联；
- 深度失败聚类、根因分析、质量报告、缺陷联动和自愈；
- 多知识库、用户/角色/ACL、组织级审批、审计、SSO 和企业治理；
- OpenAPI、Excel、PDF、Word、外部系统和更多资料来源适配；
- 通用 Prompt/Agent 可视化编排、Skill/Tool/MCP 市场、任意端点管理、多项目管理和跨项目自动化。

## 6. 跨阶段数据与证据约束

后续阶段必须复用 Phase 1 的资产模型，不得另建无法关联版本的资料容器；同时必须以项目版本作为项目空间业务数据的隔离边界。

- 检索结果至少返回 `projectId`、`knowledgeBaseId`、`indexVersionId`、`assetId`、`assetVersionId`、资料类型、来源、逻辑路径、标题路径、`chunkId` 和原文定位信息；页面必须按返回的 `assetVersionId` 打开证据原文；
- 需求评审、运行、Finding、问答及后续阶段产物必须保存 `projectVersionId`；需求评审还必须保存 `requirementBindingId`，服务端不得从“当前版本”或“最新版本”隐式推断作用域；
- 切换项目版本后只能读取该版本的绑定、评审、运行、Finding 和问答；跨版本比较必须由用户显式选择双方版本，不得在默认列表、统计或上下文中混合数据；
- AI 分析任务必须保存输入资产版本、检索范围、证据快照，以及 Pi Agent Framework/Runtime、`agentId`、Agent 类型与定义、模型、Prompt、工具集、MCP、网络、权限和执行限制版本；
- 多 Agent 工作流还必须保存 `workflowRunId`、任务拓扑、参与 Agent 与执行 ID、父子关系、依赖、交接产物版本、审批、汇总结果和终止原因；
- Agent 与工具事件是过程记录，不直接代表业务运行成功；只有服务端完成结构化 Schema、证据和一致性校验并原子保存结果后，运行才能成功；
- 联网、MCP 和外部内容均视为不可信数据，不能修改系统 Prompt、权限、路由、审批和终止策略；不能映射为固定项目证据时，只能作为外部参考或待确认信息；
- 模型调用、工具调用和外部操作确认必须关联运行并脱敏记录；取消、失租约或达到执行限制后的迟到结果不得进入正式结论；
- 资产版本更新后不覆盖历史引用；旧结论仍能按原快照查看，并可被标记为可能过期；
- 后续权限设计不得改变资产、版本、Chunk 和证据引用的稳定标识；
- Wiki、Git 与外部文档系统未来通过 Source Connector 接入，不得破坏 Phase 1 的上传、候选索引和固定版本证据语义。

## 7. 阶段核心页面

### 7.1 项目知识库页

- 查看项目默认知识库、资料数量、资料类型分布和索引状态；
- 按同步状态和逻辑路径浏览知识资产；资料类型和来源仅作为资产元数据展示；
- 查看资产版本、当前活动版本、最近成功同步时间和最近错误；
- 上传 Markdown/纯文本资料，并查看异步同步结果；
- 执行基础检索并定位至固定资产版本、标题路径和原文位置；
- 查看或重试失败同步任务。

### 7.2 知识库配置页

- 配置默认资料类型、逻辑路径规则和上传冲突策略；
- 配置 Markdown/纯文本解析、Chunk 大小、重叠和预处理版本；
- 选择部署端已注册的 Embedding 来源与模型，配置维度和基础检索参数；远程端点和密钥不进入页面或知识库配置；
- 显示草稿/已保存状态及配置变更影响；
- 对影响索引兼容性的配置发起受控重建，并展示旧索引持续可用、进度、失败或取消状态。

### 7.3 需求评审页

- 未创建或未选择项目版本时展示空状态和“新建项目版本”主操作，禁止加载需求分析工作台；
- 只展示当前项目版本绑定的需求，明确区分项目发布版本（如 `V3.6`）与需求资产版本（如 `V2.3`）；
- 支持从知识库加入既有 `ready` 需求版本，或上传 Markdown、纯文本、ZIP 并在解析完成后加入当前 `open` 项目版本；
- 从当前项目版本绑定的固定需求资产版本创建评审，展示项目版本、绑定、输入、索引、检索、模型、`RequirementAnalysisAgent`、Prompt、工具与权限快照；
- 展示真实排队、执行、工具调用、结果提交、服务端校验、成功、失败和取消状态；
- 查看结构化摘要、功能树、Finding、风险统计、版本差异和固定证据，并按严重度、依据类型和处置状态过滤；
- 对 Finding 进行确认、驳回、解决或待跟进处置，保留模型初始结果和追加式处置历史；
- 围绕成功运行开展带证据引用的评审问答，不使用本地模拟结果替代真实失败；
- 展示联网、MCP、模型降级、工具拒绝、审批等待、执行限制和可能过期等真实状态。

### 7.4 AI 模型与路由页

- 查看部署侧已注册的生成式模型来源、模型能力、健康状态和脱敏检测结果；
- 配置默认模型、参数、结构化输出、智能路由、重试、降级与回退顺序，并生成不可变配置版本；
- 展示 `RequirementAnalysisAgent`、Prompt、工具集、MCP 和网络策略版本摘要，不向普通用户开放任意端点、密钥或通用编排；
- 查看运行级实际模型、路由原因、降级、Token、耗时、Agent 回合和工具调用摘要。

## 8. 阶段验收总览

### 8.1 第一阶段验收

第一阶段详细验收场景见 [第一期-项目知识库构建与配置需求文档.md](第一期-项目知识库构建与配置需求文档.md)。总体验收以以下事实为准：

1. 项目资料可以按统一知识资产模型接入、分类、版本化和同步；
2. 未变化内容不会重复产生活动 Chunk 或重复 Embedding；
3. 同步失败、取消或重建失败时，上一个活动索引仍可检索；
4. 检索结果可定位至确定资产版本与原文位置；
5. 配置变更可追踪，影响索引的变更通过异步重建生效；
6. 验收不依赖 AI、Git、测试执行、权限或外部系统能力。

### 8.2 第二阶段验收

第二阶段详细验收场景见 [第二期-需求评审与大模型配置需求文档.md](第二期-需求评审与大模型配置需求文档.md)。总体验收以以下事实为准：

1. 未创建或未选择项目版本时不能进入需求分析；创建 `open` 项目版本后，可将 `ready` 需求资产版本绑定到该版本，且 Review、Run、Finding 和问答只在同一 `projectVersionId` 下可见；
2. 每个评审运行仅启动一个 `RequirementAnalysisAgent` 执行实例，并固定项目版本、需求绑定、需求、索引、检索、模型、Pi Agent Framework/Runtime、Agent 定义、Prompt、工具、MCP、网络、权限和执行限制版本；后续配置变化不修改历史运行；
3. `RequirementAnalysisAgent` 只能调用运行快照内的受控工具，工具循环或 Agent 执行结束不直接代表评审成功；
4. 正式 Finding 通过结构化 Schema、证据映射和一致性校验，有证据结论可打开至固定资产版本和原文位置；
5. 模型、检索、工具、联网、MCP、审批或校验失败均显示真实状态，不回退为模拟结果、缓存答案或空白成功；
6. 人工处置追加保存且不覆盖模型初始输出；重新评审创建独立运行，旧运行和处置继续可查看；
7. 允许范围内的只读和可逆外部操作按策略执行，高风险操作未经逐次确认不得执行，越权、注入和越界网络请求被阻断并记录；
8. 取消、失租约、达到回合/Token/工具/超时限制会停止后续调用，迟到结果不得成为正式 Finding；
9. API、页面和普通日志不返回明文密钥、完整 Authorization、未经脱敏端点或敏感工具参数。

## 9. 依赖与前置条件

### 9.1 第一阶段前置条件

第一阶段仅需要：

- 可用于试点的 Markdown/纯文本项目资料；
- 已确定的资料类型分类和逻辑路径约定；
- 可访问的 Embedding 服务或本地 Embedding 运行环境；
- 受控上传存储或受控本地资料目录；
- 用于验证检索和同步行为的样例资料集。

Git API、CI/CD、测试环境、测试账号、`data-testid`、角色权限策略和外部缺陷系统均不是第一阶段前置条件。

### 9.2 第二阶段前置条件

第二阶段需要：

- Phase 1 可稳定提供 `ready` 需求资产版本、Chunk、活动索引、检索与固定证据定位；
- 项目版本服务可创建、查询和切换 `open/locked/archived` 版本，并能维护项目版本与固定需求资产版本的绑定；
- 至少一个部署侧受控注册、支持结构化输出的生成式模型及明确的数据传输和保留边界；
- 满足锁定 Pi Agent Framework 版本要求的 Node.js 运行环境，以及通过兼容性测试的 `PiAgentRuntimeAdapter`；
- 已创建、发布并启用的唯一 `RequirementAnalysisAgent`，以及它的 Prompt、默认只读工具集、模型配置、权限策略和执行限制版本；
- 如启用联网或 MCP，已注册允许的域名、MCP Server、凭据引用、数据发送范围和安全策略；
- 已定义外部操作风险等级、自动执行边界、人工确认门禁、幂等和补偿责任；
- 包含典型需求、预期 Finding、固定证据、工具调用、攻击样例和人工处置的评审质量验收集。

Git API、代码仓库、CI/CD、测试执行环境、组织级角色权限和外部工单系统不是第二阶段需求评审的必需前置条件；如通过受控 Tool/MCP 接入其中任一能力，只能在已注册、最小授权和专项验收范围内使用。
