## 2. 背景与目标

项目中的需求文档、技术方案、接口文本、数据字典、测试用例、测试报告、缺陷复盘等资料通常由不同人员在不同时间独立维护。若这些资料不能统一记录来源、版本、同步状态和原文定位，后续的 AI 需求分析与技术方案分析就无法可靠地基于项目事实工作。

本期只建设项目知识库基础能力：将 Markdown/纯文本项目资料接入统一的知识资产模型，提供版本化、增量同步、基础检索、索引重建和可观测状态。

### 2.1 本期目标

1. 每个项目拥有一个可配置、可检索的默认项目知识库。
2. 项目资料通过网页/API 受控上传接入并进入持久化异步同步链路；未来 Wiki 等文档接口通过标准连接器复用同一链路。
3. 每份资料具有稳定的知识资产标识、不可变资产版本、来源、逻辑路径和同步状态。
4. 内容变更以 Hash 和稳定 Chunk 标识驱动增量处理，避免无变化内容重复切分或 Embedding。
5. 检索结果可追溯至固定资产版本、标题路径、Chunk 和原文位置。
6. 切分、Embedding 和检索配置可保存、可追踪；影响索引兼容性的变更通过受控异步重建生效。
7. 同步或重建失败、取消时，上一个活动索引持续可用。

### 2.2 本期非目标

本期不实现以下能力，且验收不得依赖它们：

- AI 需求分析、AI 对话、原型图/HTML 原型解析；
- AI 技术方案生成、技术影响分析或方案评审；
- Git、代码仓库、分支、Commit、Pull Request、代码 Diff、静态分析和代码影响分析；
- 测试设计、测试用例/脚本生成、Playwright、测试执行、CI/CD、日志/截图/Trace 采集、根因分析和自愈；
- 用户、角色、ACL、审批流、SSO、组织级审计与多租户治理；
- OpenAPI、Excel、PDF、Word 等专用文件格式解析器；
- 多项目知识库聚合检索与一个项目下多个业务知识库的管理界面。

## 3. 范围与边界

### 3.1 支持的资料格式

| 格式 | 本期支持 | 说明 |
|---|---|---|
| UTF-8 Markdown（`.md`） | 是 | 解析标题层级、正文、代码块、列表和链接，保留原文定位。 |
| UTF-8 纯文本（`.txt`） | 是 | 以文本段落或行区间切分，保留原文定位。 |
| OpenAPI / Swagger | 否 | 后续作为 `api_spec` 的专用解析器接入；本期可用 Markdown/纯文本表达接口说明。 |
| Excel / CSV | 否 | 后续作为数据字典、测试用例等专用解析器接入。 |
| PDF / Word | 否 | 后续接入文本提取与版面定位能力。 |
| 图片、原型包、HTML | 否 | 后续 AI 需求分析阶段单独定义。 |

不因格式暂未支持而限制资料类型：需求、技术方案、接口说明、测试用例和测试报告等均可先以 Markdown/纯文本形式作为知识资产维护。

### 3.2 知识资产边界

以下资料在被人工创建、导入或确认保存为独立项目资料版本后，属于知识资产：

- 版本化需求文档、验收标准和业务规则；
- 技术方案、架构资料；
- API 文本/规范、数据字典；
- 测试规范、测试用例、测试报告；
- 缺陷复盘、问题解决方案、项目术语和其他项目说明。

AI 结论、对话消息、临时草稿与任务中间结果不是自动入库内容。未来如人工将其保存为独立技术方案、报告或其他资料版本，则按本期通用资产接入和同步流程处理。

### 3.3 单一资料域

第一期不设置“分析工作区”与“正式知识库”两套资料域，也不存在“发布到正式知识库”的状态转换。资料是否参与检索只取决于：它是否为独立维护的知识资产版本，以及该版本是否已进入活动索引。

## 4. 术语、对象与状态

### 4.1 对象模型

```text
项目
  └─ 默认项目知识库
       └─ 知识资产
            └─ 资产版本
                 └─ Chunk

默认项目知识库
  └─ 索引版本
       └─ 索引成员（固定关联资产版本、Chunk 与兼容向量）
```

| 对象 | 定义 | 本期规则 |
|---|---|---|
| 项目 | 资料、配置和后续分析的归属边界。 | 一个项目首期创建一个默认项目知识库。 |
| 默认项目知识库 | 项目下资料接入、同步、检索和配置的集合。 | 名称可维护；多知识库能力仅保留数据扩展空间。 |
| 知识资产 | 同一逻辑项目资料的稳定实体。 | 以资产 ID 标识，不以文件名或存储路径作为唯一身份。 |
| 资产版本 | 知识资产的不可变内容快照。 | 内容、来源、逻辑路径、Hash、解析配置和同步结果均可追溯。 |
| Chunk | 一个资产版本的可检索片段。 | 必须有稳定 `chunk_key`、内容 Hash、标题路径和原文定位。 |
| 索引版本 | 在固定配置下可检索的 Chunk 集合。 | 同一时刻只有一个活动索引版本；切换采用原子方式。 |
| 索引成员（`IndexMembership`） | 某个索引版本实际使用的固定 Chunk 及其检索快照。 | 固定关联资产版本、Chunk/`chunk_key`、内容与原文定位、显示名称、资料类型、来源、逻辑路径、Embedding 兼容性和配置快照；重建只能新建成员集合，不得改写旧版本 Chunk 或旧成员快照。 |
| 同步任务 | 接入、解析、切分、Embedding、索引构建或删除处理的异步任务。 | 记录输入范围、状态、进度、尝试次数、错误和配置快照。 |

### 4.2 资料类型

| 值 | 名称 | 示例 |
|---|---|---|
| `requirement` | 需求文档 | 订单退款需求 V2.0 |
| `technical_design` | 技术方案 | 支付重构技术方案 |
| `architecture` | 架构资料 | 服务边界与部署架构 |
| `api_spec` | API 文本/规范 | 订单查询接口说明 |
| `data_dictionary` | 数据字典 | 订单领域字段字典 |
| `test_case` | 测试用例 | 退款流程测试用例 |
| `test_report` | 测试报告 | 版本 V2.0 回归测试报告 |
| `defect_review` | 缺陷复盘 | 重复扣款缺陷复盘 |
| `other` | 其他项目说明 | 项目术语表、运行说明 |

资料类型用于资产分类、检索证据展示和后续 AI 分析语境，不决定解析流程，也不作为浏览或检索筛选条件；同一类型的资料可存在多个知识资产与多个资产版本。

### 4.3 来源类型

| 值 | 名称 | 本期规则 |
|---|---|---|
| `upload` | 受控上传 | 用户通过网页或 API 提交文件内容、资料类型和逻辑路径。 |

Wiki、文档平台、Git、工单系统和其他外部系统仅作为未来 Source Connector 类型预留，不在本期实现；未来连接器必须先固化来源快照，再进入本期定义的异步同步、候选索引和固定证据链路。

### 4.4 资产版本状态

| 状态 | 含义 | 可否被活动索引检索 |
|---|---|---|
| `pending` | 已创建版本，等待同步。 | 否 |
| `syncing` | 正在解析、切分、Embedding 或构建索引。 | 否；旧活动版本仍可检索。 |
| `ready` | 已完成并进入活动索引。 | 是 |
| `failed` | 同步失败，保留错误和可重试信息。 | 否；若存在旧活动版本，旧版本继续可检索。 |
| `deleted` | 来源已确认删除或版本已被逻辑删除。 | 否；历史记录可查看。 |

同步任务状态应至少包括 `queued`、`running`、`succeeded`、`failed`、`cancelled`。资产版本状态与同步任务状态独立：一个失败任务不应覆盖已 `ready` 的旧版本。

## 5. 功能需求

### 5.1 项目与默认项目知识库

#### FR-001：默认知识库创建

- 创建项目时，系统创建一个默认项目知识库。
- 默认知识库必须拥有稳定 ID、名称、所属项目、创建时间、当前活动索引版本和配置版本。
- 首期界面仅管理项目的默认知识库；数据模型可预留未来多知识库扩展字段，但不得以多库管理作为本期交付条件。

#### FR-002：知识库概览

项目知识库页应展示：

- 默认知识库名称和当前索引状态；
- 知识资产总数、已就绪版本数、同步中数、失败数；
- 按资料类型和来源的数量分布；
- 最近成功同步时间、最近失败任务和当前活动索引版本；
- 最近一次索引重建的状态与结果。

### 5.2 知识库配置

#### FR-003：配置草稿与保存

- 配置页必须区分**已保存配置**和**编辑草稿**。
- 草稿与已保存配置不一致时，显示未保存状态；离开页面前应提示存在未保存变更。
- 保存成功后生成新的配置版本并记录保存时间；保存本身不等同于立即修改活动索引。
- 无变化时不允许重复保存，也不生成无意义配置版本。

#### FR-004：可配置项

首期配置至少包含：

| 分类 | 配置项 |
|---|---|
| 接入 | 默认资料类型、逻辑路径规则、上传冲突策略；物理存储目录由系统统一管理 |
| 解析与切分 | 文件编码、Markdown/纯文本解析器版本、Chunk 目标大小、最大大小、重叠、标题层级策略、预处理版本 |
| Embedding | 已注册的来源与模型、向量维度、批处理大小、超时与重试次数；本地来源模型可由页面管理 |
| 检索 | 关键词召回数量、向量召回数量、最终返回数量、相关度阈值、混合检索开关 |

远程 Embedding 来源、Base URL 和 API Key 必须由部署环境注册和注入。知识库配置及浏览器只可选择已注册的来源与模型，不得保存、提交或展示端点、API Key 或其他凭据。

本期配置不包含模型对话参数、AI Agent 提示词、代码仓库、测试环境、用户权限或审批策略。

#### FR-005：配置影响提示

系统应将配置变更分为：

| 变更类型 | 示例 | 生效方式 |
|---|---|---|
| 仅查询行为变更 | 召回数量、最终结果数、相关度阈值 | 保存后用于后续检索，不需要重建索引。 |
| 索引兼容变更 | Chunk 参数、解析器/预处理版本、Embedding Provider/模型/维度 | 保存后明确提示需要重建；未重建前继续使用旧索引和旧配置快照。 |
| 接入行为变更 | 默认资料类型、逻辑路径规则、上传冲突策略 | 保存后用于后续上传，不改写历史资产版本。 |

### 5.3 资料接入

#### FR-006：统一接入链路

网页/API 上传必须使用统一的资产识别、版本创建、同步任务、候选索引和状态记录链路。未来接入 Wiki 等连接器时也必须复用该契约，不得另建不兼容的资料模型或索引流程。

每次接入至少提交或生成：

- 项目与默认项目知识库；
- 来源类型和来源标识；
- 资料类型；
- 显示名称；
- 逻辑路径；
- 文件内容与编码；
- 接入时间和内容 Hash；
- 适用的配置版本。

#### FR-007：受控上传

- 支持上传 UTF-8 `.md` 与 `.txt` 文件。
- 上传时允许选择资料类型、编辑显示名称和指定逻辑路径。
- 服务端不得将用户提交的文件名直接拼接为实际文件系统路径；逻辑路径仅用于知识库内标识和展示。
- 不支持的格式、无法解码的内容、超过限制的文件或解析失败必须明确失败原因，不得伪装为成功。
- 上传成功仅表示已进入同步队列；只有同步完成且索引切换成功后，资产版本才进入 `ready`。

#### FR-008：上传来源与未来连接器边界

- 第一期仅支持网页/API 文件上传，不实现本地目录监听、文件系统扫描或来源对账。
- 上传接入只提交逻辑路径和文件内容，不允许客户端指定服务器物理目录。
- 未来接入 Wiki、文档平台或其他内容接口时，应新增独立 Source Connector，将固定内容快照和标准元数据投递到本期异步同步任务；连接器不得直接写 Chunk、Embedding 或活动索引。
- 未来连接器的认证、增量游标、删除语义、限流、回调和对账规则应在对应阶段单独定义和验收，不视为第一期已交付能力。

#### FR-009：冲突策略

同一项目知识库内，逻辑路径是用于发现同一资料的主要键。重复上传覆盖同一路径时，系统根据已保存的冲突策略处理：

| 策略 | 行为 |
|---|---|
| `reject` | 拒绝新接入并提示已有来源和活动版本。 |
| `new_version` | 将内容视为同一知识资产的新不可变版本；保留原版本和来源记录。 |
| `parallel_asset` | 创建并列知识资产；要求使用不同显示名称，检索结果必须显示来源和逻辑路径以避免混淆。 |

默认策略为 `new_version`。若新内容与当前资产版本内容 Hash 一致，则不创建新资产版本或重复同步任务。

### 5.4 资产、版本与生命周期

#### FR-010：资产识别与版本创建

- 同一逻辑资料更新时保留同一 `assetId`，创建新的不可变 `assetVersionId`。
- 资产版本必须记录内容 Hash、来源、逻辑路径、资料类型、接入时间、解析/切分/Embedding 配置快照和同步结果。
- 资产版本一经 `ready` 不可原地修改；资料更新、类型更正或内容替换必须通过新版本表达。
- 活动版本用于当前活动索引；历史版本允许浏览与证据定位，但不得被“当前最新内容”悄然替代。

#### FR-011：删除、重命名与连续保存

- 用户删除知识资产时，系统创建删除任务；在确认成功切换新索引前，旧活动索引继续服务。
- 删除完成后，受影响版本标为 `deleted`，历史记录保留，不再出现在默认检索结果。
- 同一资料重命名且内容 Hash 不变时，保留资产身份和版本，可更新逻辑路径/展示元数据并记录来源事件；不得重复生成 Embedding。
- 连续或重复上传同一内容时，系统应以内容 Hash 去重，合并可安全合并的任务，避免无效版本和重复向量写入。

### 5.5 同步、切分与索引

#### FR-012：同步任务链路

同步任务至少按以下顺序执行：

```text
发现/上传资料
  → 内容校验与 Hash
  → 识别资产或创建资产版本
  → 解析
  → 结构化切分
  → Chunk 对比与向量复用
  → Embedding
  → 构建候选索引版本
  → 原子切换活动索引
  → 更新资产版本与任务状态
```

任务必须记录任务 ID、触发方式、输入资产/路径范围、开始/结束时间、当前步骤、进度、重试次数、错误摘要和使用的配置快照。

#### FR-013：内容 Hash 与增量同步

- 资产版本内容 Hash 相同则直接复用上一次成功结果，不重新解析、切分或 Embedding。
- 内容变化时，只处理受影响资产版本；不得全库无条件重建。
- Markdown 应使用 AST 或等价结构化解析，以标题层级、段落、列表、代码块等结构形成切分边界；纯文本以段落或行区间形成边界。
- Chunk 必须包含稳定 `chunk_key`、内容 Hash、标题路径、序号和原文定位信息。
- 当 Chunk 内容 Hash 不变时，复用已存在的 Embedding；仅新增或变化的 Chunk 调用 Embedding Provider。
- 切分规则或 Embedding 兼容配置变化时，不得误用旧结果，必须通过 FR-017 的索引重建生成新索引版本。

#### FR-014：失败、重试与取消

- 解析、切分、Embedding、数据库写入或索引构建失败时，任务标记为 `failed` 并记录可理解的错误摘要和失败步骤。
- 可重试错误可由用户手动重试，并按配置限制自动重试次数；每次尝试需记录。
- 用户可取消尚未完成的同步或重建任务；取消后标记为 `cancelled`，清理未提交的候选结果，不修改当前活动索引。
- 失败或取消不得将已 `ready` 的旧版本标记为失败，也不得中断当前检索。

#### FR-015：异步任务恢复

- 同步或重建任务必须先持久化为 `queued`，再由 Worker 领取执行；API 进程重启不得丢失已受理任务。
- 服务启动后应将中断的 `running` 任务恢复为可安全重试的 `queued` 状态，并废弃未提交的候选索引。
- Worker 恢复执行前必须重新校验任务、资产版本、配置版本和当前活动索引，防止陈旧任务覆盖新结果。
- 恢复、重试或重复投递不得创建重复资产版本、重复 Embedding 或无效活动索引。

#### FR-016：活动索引原子切换

- 新同步结果或索引重建结果必须先构建为候选索引版本及其固定索引成员集合。
- 仅在候选成员完整、任务仍有效、输入资产版本和配置快照未陈旧且 Embedding 兼容时，才可使用单次状态更新将其标记为活动索引版本。
- 原子切换前与切换失败时，旧活动索引持续可检索；取消、失败、重启恢复或陈旧任务不得改变旧活动索引。
- 系统必须能展示当前活动索引版本、候选版本、建立时间、固定成员范围、包含的资产版本范围和配置快照。

### 5.6 浏览、检索与定位

#### FR-017：知识资产浏览

知识库页必须支持：

- 按资产版本状态和逻辑路径筛选；资料类型和来源仅作为资产元数据展示；
- 按名称、逻辑路径和内容关键词搜索；
- 查看资产概要、来源、资料类型、当前活动版本、版本历史、内容 Hash、最近成功同步时间和最近错误；
- 查看某个固定资产版本的原始 Markdown/纯文本内容、标题结构、Chunk 列表和同步详情；
- 仅显示或隐藏 `deleted` 资产版本，默认隐藏。

#### FR-018：基础检索

- 支持关键词检索；当 Embedding 可用且已建立活动向量索引时支持纯向量检索和混合检索。
- 关键词模式不依赖 Embedding Provider；混合模式的查询向量不可用时可返回关键词结果，并明确标记向量阶段降级及原因；纯向量模式不可用时必须返回明确的向量不可用状态，不得伪装为无匹配。
- 可选 Reranker 仅对一阶段候选重排；其不可用时保留一阶段排序，并明确标记重排降级及原因。
- 可按逻辑路径缩小检索范围；资料类型和来源仅在结果与证据中展示。
- 每条结果至少显示：
  - 固定索引成员快照中的资产显示名称和资料类型；
  - `assetId`、`assetVersionId` 与 `indexVersionId`；
  - 固定快照中的来源与逻辑路径；
  - 标题路径；
  - `chunkId` 或稳定 `chunk_key`；
  - 行号范围、字符范围或等价原文定位；
  - 匹配摘要、检索方式与相关度信息。
- 打开检索结果必须以返回的 `assetVersionId` 加载该固定版本原文并定位至命中位置，不得仅按 `assetId` 自动跳至同一资产的当前版本。
- 索引成员必须冻结显示名称、资料类型、来源和逻辑路径；资产重命名、移动或类型变更不影响已激活索引的检索证据，只有重建并激活新索引后才使用当前元数据。
- 结果必须清楚区分 `ready` 活动版本与历史版本；默认检索只返回当前活动索引内容。

#### FR-019：检索不可用提示

以下场景应给出明确状态，而非返回误导性空结果：

| 业务码 | 场景 | 响应语义 |
|---|---|---|
| `KB_INITIAL_INDEXING` | 首次建库或相关索引仍在构建。 | 明确展示建立进度；不能以无匹配代替。 |
| `KB_NO_READY_VERSION` | 知识库尚无 `ready` 资产版本。 | 提示上传资料或等待同步。 |
| `KB_NO_ACTIVE_INDEX` | 存在资料但当前没有活动索引。 | 提示索引不可用及下一步。 |
| `KB_VECTOR_UNAVAILABLE` | 纯向量检索的查询向量、Provider 或兼容向量索引不可用。 | 明确该请求未能执行，不返回伪空结果。 |
| `KB_DEGRADED_KEYWORD_ONLY` | 混合检索的向量阶段不可用。 | 返回关键词结果，并携带降级原因。 |
| `KB_DEGRADED_RERANKER_BYPASSED` | Reranker 不可用。 | 返回一阶段排序，并携带重排降级原因。 |
| `KB_FILTER_EMPTY` | 逻辑路径筛选条件排空结果集。 | 提示调整逻辑路径。 |
| `KB_NO_MATCH` | 检索正常执行但没有匹配。 | 提示无匹配并保留查询条件。 |

### 5.7 索引重建

#### FR-020：重建触发与确认

- 当用户保存影响索引兼容性的配置后，系统显示受影响范围、旧活动索引版本和是否立即发起重建。
- 用户确认后创建异步索引重建任务；重建可针对整个默认知识库或明确选定的资产范围。
- 本期不要求静默自动重建，也不在用户未确认时修改活动索引。

#### FR-021：重建过程与结果

- 重建时，系统展示任务进度、处理资产/Chunk 数量、当前步骤、开始时间、预估或已处理数量、错误与取消入口。
- 已存在旧活动索引时，重建期间旧索引继续提供检索。
- 首次建库没有旧索引时，界面明确说明“正在建立首个索引，向量检索暂不可用”。
- 成功后以原子切换更新活动索引；失败或取消后旧索引继续有效。
- 完成后保留新旧索引版本元数据和配置快照，供追溯与问题排查。

### 5.8 可观测性

#### FR-022：任务与错误可见性

知识库页或任务视图应展示：

- 同步与重建任务列表、状态、触发来源、资产/路径范围、开始/完成时间和进度；
- 最近错误、失败步骤、错误摘要、尝试次数和重试入口；
- 当前活动索引版本、候选索引版本、最近切换结果和配置快照；
- 最近成功同步时间、恢复结果和未处理异常。

本期记录的是系统运行和资料处理状态，不构建用户级权限审计、操作审批或合规审计体系。

## 6. 主要交互流程

### 6.1 上传并进入检索

```text
用户选择 Markdown/纯文本文件
  → 选择资料类型、显示名称和逻辑路径
  → 系统校验格式、编码和路径规则
  → 创建或识别知识资产
  → 内容相同：提示已复用已有成功版本
  → 内容变化：创建资产版本并进入同步队列
  → 任务完成并构建候选索引
  → 原子切换成功
  → 版本为 ready，可在检索结果中定位
```

若任一步失败，界面展示失败步骤与可重试动作；旧活动版本和旧索引保持可用。

### 6.2 异步任务与重启恢复

```text
上传 API 固化原文快照并创建 queued 任务
  → Worker 领取任务并建立候选索引
  → 解析、切分与 Embedding
  → 校验任务未取消且输入/配置仍有效
  → 原子切换活动索引
  → 进程中断时恢复 queued 并重新校验
```

服务重启后不得丢失已受理任务；恢复执行必须复用内容 Hash 和已有 Chunk 结果，不因恢复本身全量重复计算 Embedding。

### 6.3 配置保存与重建

```text
用户编辑知识库配置草稿
  → 系统显示未保存及配置影响
  → 保存配置版本
  → 无索引兼容影响：后续查询或接入使用新配置
  → 有索引兼容影响：用户确认重建
  → 使用新配置建立候选索引版本
  → 成功原子切换 / 失败或取消保留旧活动索引
```

### 6.4 检索并查看证据

```text
用户输入问题或关键词并选择筛选条件
  → 系统查询活动索引
  → 返回命中 Chunk、固定资产版本与原文定位
  → 用户打开结果
  → 系统展示该资产版本、标题路径和命中原文
```

## 7. 数据与一致性要求

### 7.1 最小记录字段

| 实体 | 最小字段 |
|---|---|
| 知识资产 | `assetId`、`projectId`、`knowledgeBaseId`、名称、资料类型、当前活动版本、创建/更新时间 |
| 资产版本 | `assetVersionId`、`assetId`、来源、来源标识、逻辑路径、内容 Hash、状态、配置快照、创建/完成时间、错误摘要 |
| Chunk | `chunkId`、`assetVersionId`、稳定 `chunk_key`、内容 Hash、标题路径、序号、原文定位、Embedding 引用 |
| 索引版本 | `indexVersionId`、`knowledgeBaseId`、状态、活动标记、资产版本范围、配置快照、创建/切换时间 |
| 索引成员 | `indexVersionId`、`assetVersionId`、`chunkId`、`chunk_key`、内容 Hash、标题路径、原文定位、显示名称、资料类型、来源、逻辑路径、Embedding 兼容性引用；固定某索引的实际检索范围与证据元数据。 |
| 同步任务 | `jobId`、类型、状态、触发来源、输入范围、进度、尝试次数、错误、开始/结束时间、配置快照 |

### 7.2 一致性规则

1. 资产版本与内容 Hash 一一对应；相同内容不得在同一资产下重复创建版本。
2. 任何检索结果必须保留可访问的资产版本和 Chunk 定位。
3. 候选索引未校验完成前不得成为活动索引。
4. 任务失败、取消或进程重启不得破坏已提交的活动索引。
5. 删除和重命名必须有明确任务或变更记录；不允许仅删除向量却保留可检索资料元数据，或反之。
6. 索引、Chunk 和 Embedding 的生成必须记录适用配置版本，避免不同维度或不同切分规则混用。
7. 每个活动或历史索引必须能通过固定索引成员、配置快照和 Embedding 兼容性还原其实际检索范围与显示名称、资料类型、来源、逻辑路径证据；索引重建不得改写旧资产版本 Chunk 或旧索引成员。

## 8. 验收标准

### AC-001：创建默认项目知识库

**前置条件**：创建一个新项目。

**操作**：进入该项目的知识库页。

**预期结果**：

- 系统存在一个默认项目知识库并显示稳定标识、名称和初始索引状态；
- 未建立首个索引时，界面明确提示向量检索尚不可用；
- 不要求创建多知识库、配置角色或配置权限。

### AC-002：上传不同类型的 Markdown/纯文本资料

**前置条件**：默认项目知识库可用。

**操作**：依次上传需求文档、技术方案、API 说明、测试用例、测试报告各一份 Markdown 或纯文本资料，并分别选择对应资料类型。

**预期结果**：

- 每份资料按统一知识资产模型创建或更新；
- 浏览页可按逻辑路径筛选，并显示资料类型、名称、来源和逻辑路径；
- 不要求 OpenAPI、Excel、PDF、Word 或图片解析。

### AC-003：同内容重复上传不重复处理

**前置条件**：一份 Markdown 资料已同步成功。

**操作**：以相同逻辑路径再次上传内容完全相同的文件。

**预期结果**：

- 系统识别内容 Hash 相同；
- 不新增不可变资产版本，不重复调用 Embedding，不生成无效索引版本；
- 界面说明已复用现有成功结果。

### AC-004：修改资料只增量处理变化 Chunk

**前置条件**：包含多个标题段落的 Markdown 资料已同步成功。

**操作**：只修改其中一个段落并再次接入。

**预期结果**：

- 创建同一知识资产的新资产版本；
- 未变化 Chunk 保留稳定 `chunk_key` 并复用已有 Embedding；
- 仅新增或变化的 Chunk 进入 Embedding；
- 历史版本仍可查看，活动索引切换后新版本用于默认检索。

### AC-005：异步任务与重启恢复

**前置条件**：默认知识库可用，存在可上传的 Markdown/纯文本资料。

**操作**：上传资料并在任务执行前、Embedding 执行中和候选索引完成前检查状态；随后模拟服务在任务 `running` 时重启并恢复任务。

**预期结果**：

- 上传受理后立即生成持久化 `queued` 任务，完成前不切换活动索引；
- Worker 构建完整候选索引并校验成功后，才在条件事务中切换活动索引；
- 任务执行中取消、失败或进程中断时，旧活动索引保持不变；
- 重启后中断任务可恢复执行，废弃候选不会成为活动索引，也不会重复创建资产版本或向量。

### AC-006：同步失败与重试恢复

**前置条件**：存在一个已 `ready` 的资产版本和活动索引。

**操作**：模拟新版本在解析、Embedding 或索引构建阶段失败，然后执行重试。

**预期结果**：

- 新任务与新版本展示 `failed`、失败步骤、错误摘要和尝试次数；
- 已 `ready` 的旧版本与旧活动索引仍可检索；
- 重试成功后才创建候选索引并原子切换；
- 不要求 AI 错误诊断、执行日志分析或权限审批。

### AC-007：基础检索与固定版本定位

**前置条件**：至少存在需求文档、技术方案和测试用例三类已就绪资产。

**操作**：使用关键词、向量或混合检索，并按逻辑路径筛选；打开一个结果；随后重命名或移动对应资产并在重建前后分别检索。

**预期结果**：

- 结果显示固定索引版本、资料类型、来源、逻辑路径、资产 ID、资产版本 ID、标题路径、Chunk 标识和原文定位；相同索引版本在相同查询和筛选条件下可还原其证据成员。
- 打开结果按返回的 `assetVersionId` 显示命中时的固定资产版本，而不是同一资产后续更新的内容。
- 资产重命名或移动后、重建前，活动索引仍返回原有成员的显示名称、资料类型、来源和逻辑路径；重建并激活后，新索引才使用当前元数据。
- 混合检索的查询向量不可用时，仍返回关键词结果并标记 `KB_DEGRADED_KEYWORD_ONLY` 及原因；纯向量模式不可用时，返回 `KB_VECTOR_UNAVAILABLE`，而不是空列表。
- Reranker 不可用时保留一阶段排序，并标记 `KB_DEGRADED_RERANKER_BYPASSED` 及原因。
- `index_building`、无活动索引、无 `ready` 资料、筛选排空和无匹配结果均有区分明确的提示。

### AC-008：保存配置与受控索引重建

**前置条件**：知识库已有活动索引。

**操作**：修改检索阈值后保存；再修改 Chunk 大小或 Embedding 模型并保存，确认发起重建；分别模拟成功、失败和取消。

**预期结果**：

- 检索阈值变更保存后不触发索引重建；
- Chunk 或 Embedding 兼容变更明确提示重建影响，未经确认不修改活动索引；
- 重建期间旧活动索引持续可检索；
- 成功后新索引以原子方式成为活动版本；失败或取消后旧索引持续有效；
- 可查看新旧索引版本与各自配置快照。

### AC-009：AI 派生产物不自动入库

**前置条件**：存在后续阶段产生的分析结论或技术方案草稿样例。

**操作**：查看知识库浏览和检索结果；再将该草稿人工保存为独立技术方案资料版本并执行同步。

**预期结果**：

- 未保存为独立资料版本的 AI 中间产物不出现在知识库资产或默认检索结果中；
- 人工保存后，它以 `technical_design` 或选择的资料类型成为知识资产，并遵循相同版本、同步和检索流程；
- 本期不实现 AI 生成能力，仅验证资料边界。

## 9. 交付边界与后续衔接

第一期交付的是可被后续阶段复用的资料与检索基础，而不是 AI 分析闭环。Phase 2 的 AI 需求分析必须从第一期的固定资产版本、检索范围、配置版本和证据定位读取输入；Phase 3 的技术方案分析必须保存其输入资产版本和引用证据。

后续能力引用一期检索证据时，至少应保存 `knowledgeBaseId`、`indexVersionId`、索引配置快照标识、`assetId`、`assetVersionId`、`chunkId` 或 `chunk_key`、Chunk 内容 Hash、逻辑路径、标题路径、原文定位、检索模式、相关度和降级状态。AI 中间产物、会话消息和临时草稿不会自动进入知识库；只有人工保存为独立资料版本后，才按一期的上传、快照、同步和索引链路处理。

Git、代码分析、测试执行、诊断、自愈、权限治理及专用格式解析器应在后续阶段分别制定需求、数据边界、风险控制和验收标准，不能通过第一期的“预留字段”视为已经交付。

## 10. 前置条件与待确认事项

### 10.1 前置条件

- 一组可用于试点的 UTF-8 Markdown/纯文本项目资料；
- 项目约定的资料类型、逻辑路径和维护责任；
- 受控上传存储或受控本地目录；
- 可访问的 Embedding Provider 或本地 Embedding 运行环境；
- 用于检索评估的样例问题、预期资料和来源定位结果。

### 10.2 待确认事项

1. 首期单文件大小、单项目资料数量、Chunk 数量和保留周期的限制；
2. 默认逻辑路径命名规则，以及重复上传同一路径时是否使用默认 `new_version` 策略；
3. Embedding Provider、模型、向量维度、数据传输与存储边界；
4. Markdown 标题层级、代码块、表格和链接在首期的切分与定位规则；
5. 首期关键词、向量和混合检索的默认参数与检索质量验收样例；
6. OpenAPI、Excel、PDF、Word 等后续格式适配器的优先级；
7. 多知识库、Wiki/Git/外部文档连接器、权限治理的后续阶段排期。
