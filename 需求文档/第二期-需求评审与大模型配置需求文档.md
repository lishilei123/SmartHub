## 2. 背景与目标

一期已建立项目资料的知识资产、不可变资产版本、Chunk、活动索引和固定证据定位能力。当前原型已展示需求列表、分析运行进度、分析概览、原始文档、版本差异、功能树、证据引用、需求问答，以及“AI 模型与路由”配置入口；但上述需求 AI 与模型路由仍使用本地示例数据，不能形成可追溯的真实评审闭环。

本期目标是在一期知识库基础上，将“需求分析”升级为可执行、可复核、可人工处置的**需求评审**能力，并提供支撑该能力的受控大模型配置与路由能力。

### 2.1 本期目标

1. 从确定的需求知识资产版本创建评审，固定输入资料、检索范围、索引版本、模型配置和提示词版本。
2. 通过大模型生成结构化的需求评审结果，识别需求缺口、歧义、冲突、边界、验收风险和测试风险。
3. 每条结论均能打开至固定资料版本的证据位置；缺少证据的内容必须明确标识为待确认或模型推测。
4. 支持人工确认、驳回、解决和补充评论，不覆盖原始 AI 输出，并保留历史评审运行。
5. 支持受控配置生成式大模型、模型参数、路由与降级策略；凭据不进入浏览器和业务数据库明文字段。
6. 真实模型不可用、检索降级、输出结构校验失败或任务取消时，页面必须显示真实状态，不得回退为示例结果或伪造成功。

### 2.2 本期非目标

本期不实现以下能力，且验收不得依赖它们：

- 技术方案生成、架构设计评审、代码影响分析、Git/PR/Commit 集成；
- 测试用例或测试脚本生成、测试执行、CI/CD、日志诊断和自愈；
- 图片、原型图、HTML、PDF、Word、Excel 等专用格式的解析与多模态评审；
- 可编辑的 Prompt 编排平台、Agent 工具调用、联网检索、MCP、自动执行外部操作；
- 完整的组织、角色、ACL、审批流、SSO、合规审计和成本结算体系；
- 将 AI 评审结论、聊天内容自动写回或自动发布为知识库正式资料。

## 3. 范围、术语与原则

### 3.1 核心对象

```text
项目 / 默认知识库
  └─ 需求知识资产
       └─ 不可变资产版本

需求评审
  └─ 评审运行（固定输入与配置快照）
       ├─ 评审结论 / Finding
       │    ├─ 证据快照
       │    └─ 人工处置记录
       └─ 评审问答会话

大模型配置
  └─ 已注册模型来源
       └─ 模型定义
            └─ 评审模型配置版本 / 路由策略
```

| 对象 | 定义与规则 |
|---|---|
| 需求评审 | 对一份需求资产固定版本发起的评审逻辑实体。它不是知识资产，也不替代原需求文档。 |
| 评审运行（`ReviewRun`） | 一次可取消、可追溯的模型执行。每次重新评审都创建新运行，历史运行不可覆盖。 |
| Finding | 可人工处置的结构化评审结论，例如缺口、歧义、冲突、边界或风险。 |
| 证据快照 | Finding 所引用的固定资产版本、Chunk、内容 Hash、标题路径、原文定位和引用摘录。 |
| 大模型来源 | 由部署环境受控注册的生成式模型服务，不等同于一期 Embedding 来源。 |
| 评审模型配置 | 供需求评审使用的模型、参数、路由及降级策略的不可变版本。 |

### 3.2 评审原则

- **固定输入**：每次运行只能使用创建时绑定的需求资产版本、检索范围、活动索引版本和配置版本；不得在运行中悄然改用“当前最新资料”。
- **证据优先**：涉及项目事实的 Finding 必须关联证据；没有足够依据时，结论只能标识为“待确认”或“模型推测”。
- **人机分工**：AI 提供结构化问题、风险和建议，是否采纳由人工决定；系统不得自动修改需求内容或关闭问题。
- **可复现边界**：模型输出具有非确定性，但同一运行必须可还原输入、模型/参数、提示词、检索证据和最终输出。
- **配置分离**：生成式大模型配置与一期 Embedding、切分、索引配置分开管理；调整生成模型不得触发知识库索引重建。

## 4. 用户界面与主要流程

### 4.1 需求评审页面

沿用现有三栏原型布局，并替换所有本地 fixture：

- **左侧需求评审列表**：按项目发布版本或关键字筛选；展示需求名称、需求资产版本、最新评审状态、最新运行时间和未处理 Finding 数量；支持“新建需求评审”。
- **中间评审区**：展示需求基本信息、固定输入快照、运行/取消状态及进度；提供“评审概览、原始文档、版本差异、功能树、证据引用”五个视图。
- **右侧评审问答区**：仅围绕当前评审运行的输入与结论提问；每条回答显示其状态和引用证据。无真实模型或运行未完成时，不展示模拟回答。
- **操作记录**：展示当前评审的创建、运行、取消、人工处置、问答及错误事件；本期不将其定义为组织级合规审计。

### 4.2 创建并运行评审

```text
用户选择需求知识资产
  → 选择 ready 的需求资产版本（默认当前活动版本）
  → 选择或确认逻辑路径检索范围
  → 系统校验活动索引、模型配置和输入长度
  → 创建需求评审与 queued 的评审运行
  → 固定输入、索引、检索、模型及提示词快照
  → 检索关联资料并执行模型评审
  → 校验结构化输出与证据完整性
  → 保存结果、Finding、证据和用量记录
  → 运行 succeeded，等待人工处置
```

若运行失败、取消或校验不通过，保留运行记录、失败步骤和错误摘要；不得生成“分析完成”的示例结果。

### 4.3 人工评审闭环

```text
查看 Finding 与固定证据
  → 确认 / 驳回 / 标记已解决 / 保持待确认
  → 可填写处置说明与补充评论
  → 需求资料更新后创建新的资产版本
  → 人工发起新的评审运行
  → 旧运行保留，并标记为可能过期
```

## 5. 功能需求

### 5.1 需求评审

#### FR-201：从知识资产创建需求评审

- 新建评审时，用户必须从当前项目默认知识库中选择资料类型为 `requirement` 的知识资产及其 `ready` 资产版本。
- 默认选择该资产的当前活动版本；允许选择历史 `ready` 版本以复核历史结论。
- 页面至少展示需求标题、`assetId`、`assetVersionId`、资料逻辑路径、版本号、入库时间和当前活动索引版本。
- 需求尚未入库、无 `ready` 版本、无活动索引或模型配置不可用时，不允许启动运行，并显示可理解的原因和下一步。
- 本期不提供绕过知识库、直接粘贴任意文本后即发起正式评审的入口。

#### FR-202：评审运行快照与状态

- 创建运行时必须固定并保存：`projectId`、`knowledgeBaseId`、需求 `assetId`、`assetVersionId`、输入内容 Hash、`indexVersionId`、逻辑路径检索范围、检索参数与检索结果快照、评审模型配置版本、实际路由结果、提示词模板版本和触发时间。
- 运行状态至少包括 `queued`、`running`、`succeeded`、`failed`、`cancelled`；状态与进度必须由服务端任务真实返回。
- 用户可取消未完成运行；取消后停止后续模型调用，不写入未完成的结构化结论，已保存的步骤日志和错误信息允许查看。
- 同一需求可存在多个历史运行；重新评审必须创建新 `ReviewRun`，不得覆盖历史输入、结论、证据或人工处置。

#### FR-203：评审维度与结构化输出

每次成功运行至少生成以下结构化结果：

| 输出 | 要求 |
|---|---|
| 需求摘要 | 基于输入版本生成，明确标识为 AI 摘要。 |
| 功能树 | 以需求章节或功能域组织功能、业务规则、异常场景和验收要点。 |
| Finding | 按缺口、歧义、冲突、边界、验收风险、测试风险分类，包含问题描述、影响、建议确认项、严重度和结论依据。 |
| 评审统计 | 展示功能项、Finding、待确认项、风险项及有证据项数量；统计必须由本次持久化结果计算。 |
| 版本差异 | 当用户为同一资产选择两个固定版本时，展示真实的版本内容差异和与差异相关的 Finding；无可比较版本时明确提示。 |

- Finding 严重度使用 `blocker`、`high`、`medium`、`low` 四档；页面可映射为阻断、高、中、低风险标签。
- Finding 依据类型使用 `evidence_supported`、`needs_confirmation`、`model_inference` 三种。`evidence_supported` 必须至少有一条有效证据；其余两类必须明确说明未证实原因。
- 服务端必须校验模型输出符合已定义的结构化 Schema。输出不合法、必填字段缺失或证据引用无法解析时，运行标为失败或进入明确的“结果校验失败”步骤，不得将不完整文本伪装为成功结果。

#### FR-204：固定证据与原文定位

- 每条 Finding 可关联零到多条证据；证据快照至少包含 `assetId`、`assetVersionId`、`chunkId` 或 `chunkKey`、Chunk 内容 Hash、标题路径、行号/字符范围、引用摘录、`indexVersionId` 和检索得分/降级状态。
- 评审来源文档本身也必须以固定 `assetVersionId` 打开，不得因该资料后续更新而跳转到最新版本。
- “证据引用”视图按 Finding 展示资料名称、逻辑路径、标题路径和原文定位；点击后打开对应固定版本并高亮命中范围。
- 模型不得将无法映射至固定资料版本的内容表述为项目事实；无证据 Finding 必须使用“待确认”或“模型推测”标签。

#### FR-205：Finding 人工处置

- Finding 初始状态为 `open`，人工可更新为 `confirmed`、`dismissed`、`resolved` 或 `needs_follow_up`。
- 人工处置必须追加保存状态、处置说明、操作者显示信息和时间；不得改写模型最初生成的标题、描述、严重度、依据类型及证据快照。
- `dismissed`、`resolved` 和 `confirmed` 均应要求填写或确认处置说明；允许为空的临时保存仅使用 `open` 或 `needs_follow_up`。
- 页面支持按严重度、依据类型和处置状态过滤 Finding，并在需求列表与概览中展示未处理数量。
- 本期不实现多级审批、指派、催办、消息通知或跨项目工单同步。

#### FR-206：评审问答

- 问答必须绑定一个已成功完成的评审运行，输入范围仅包括该运行的需求资产版本、固定检索证据、结构化评审结果和会话历史。
- 每轮回答保存问题、回答、实际模型配置版本、执行时间、用量摘要和引用证据；回答中的项目事实遵循 FR-204 的证据规则。
- 当模型不可用、请求超时、证据不足或输出校验失败时，显示明确失败原因；不得使用本地模拟答案代替。
- 问答会话是 AI 派生产物，不自动进入知识库；后续如需沉淀，由人工另行保存为独立资料版本。

#### FR-207：过期标记与历史查看

- 当评审输入资产产生新的活动版本，或该评审所用索引版本不再是当前活动版本时，历史评审显示“可能过期”，但仍可查看其完整快照。
- 过期提示不得自动改变 Finding 状态，也不得自动关闭或重新运行评审。
- 用户可在同一需求资产的固定版本之间发起比较；比较对象、差异结果和所选运行必须清楚区分。

#### FR-208：输入长度与检索降级

- 启动前，系统应根据实际路由模型的上下文窗口、输出预留和固定需求内容评估输入是否可执行。
- 超出本期支持上限时，系统必须拒绝启动并说明需求版本过长及可选处理方式；不得静默截断需求原文后仍标记为“完整评审”。
- 检索发生关键词降级或 Reranker 绕过时，可按已保存策略继续运行，但运行、Finding 证据和页面必须显示降级状态及原因。
- 无活动索引、检索完全不可用或需求输入版本不可读取时，运行必须失败，不得仅依赖模型记忆生成结论。

### 5.2 大模型配置与路由

#### FR-301：生成式模型来源管理

- 系统将一期的 Embedding/检索配置与本期生成式大模型配置分为独立模块。知识库索引继续使用一期配置；修改本期模型配置不触发索引重建。
- 首期生成式调用仅支持部署环境注册的 OpenAI-compatible Chat Completions 协议或等价的受控文本生成适配器；接入其他协议须新增适配器并单独验收。
- 模型来源至少包含稳定 ID、显示名称、协议类型、启用状态、部署侧端点引用、凭据引用、可用模型列表和健康状态。
- 浏览器、需求评审 API 响应、运行快照和业务数据库不得保存或返回 Base URL、API Key、Token 等明文凭据。部署环境负责安全注入端点和密钥。
- 本期不提供用户在浏览器中新增任意端点、录入密钥或探测任意第三方服务的能力。

#### FR-302：模型定义与能力声明

每个可选生成模型至少维护：

- `modelId`、显示名称、所属模型来源、启用状态；
- 最大上下文长度、最大输出 Token、支持的语言或文本能力声明；
- 是否支持结构化 JSON 输出；本期需求评审仅允许选择支持结构化输出的模型；
- 默认超时、最大重试次数和健康状态；
- 最近一次连通性检测时间、检测结果与脱敏错误摘要。

模型不可用、被停用、能力不满足或超出输入上限时，不得成为新运行的路由目标；历史运行仍保存原实际模型标识与配置快照。

#### FR-303：评审模型配置版本

- “AI 模型与路由”页面提供需求评审模型配置草稿、未保存提示、保存和版本历史查看能力。
- 配置至少包含：默认模型、温度、最大输出 Token、请求超时、重试次数、结构化输出模式、智能路由开关、降级开关及候选回退模型顺序。
- 保存后生成新的不可变配置版本；无变化不生成冗余版本。
- 配置保存仅影响后续创建的评审运行；已创建或正在执行的运行继续使用其固定配置快照。
- 系统内置的需求评审提示词以受控模板版本标识参与快照。本期只展示版本与摘要，不提供页面编辑、发布或 Agent 工具权限配置。

#### FR-304：模型路由、重试与降级

- 智能路由关闭时，所有新运行使用当前默认模型；开启时，路由必须根据已保存的模型能力、上下文长度、启用状态和健康状态选择模型，并记录实际选择理由。
- 主模型发生可重试错误时，先按配置重试；重试耗尽后仅当“允许降级”已开启且候选模型满足输入与结构化输出能力时才切换回退模型。
- 发生模型降级时，运行页面、用量记录和最终结果必须展示实际模型与降级原因；不得将回退模型结果标识为主模型结果。
- 无可用模型时，运行进入 `failed` 并保留错误摘要；不得回退为固定示例、缓存答案或未验证旧结果。

#### FR-305：模型检测与安全反馈

- 配置页支持对已注册模型执行受控连通性与结构化输出检测，检测请求使用部署侧凭据，且不使用项目需求资料作为测试内容。
- 检测结果至少返回可用/不可用、模型能力、检测时间和脱敏错误摘要；不得回显请求头、密钥、完整端点或服务端原始堆栈。
- 前端保存或检测期间显示真实加载、成功和失败状态；保存失败不得更新页面“已保存”版本。

#### FR-306：运行可观测性

- 每次模型调用至少记录运行/会话 ID、配置版本、实际模型、路由与降级结果、开始/结束时间、耗时、输入/输出 Token（服务提供时）、重试次数、结果状态和脱敏错误摘要。
- 页面提供评审运行级别的执行概览；本期不提供组织级账单、价格计算、配额分配或审计导出。
- 运行日志不得记录完整 API Key、Authorization 请求头或未经脱敏的敏感配置值。

## 6. 数据与一致性要求

### 6.1 最小记录字段

| 实体 | 最小字段 |
|---|---|
| 需求评审 | `reviewId`、`projectId`、`knowledgeBaseId`、需求 `assetId`、创建时间、最新运行 ID、过期标记 |
| 评审运行 | `runId`、`reviewId`、状态、进度、需求 `assetVersionId`、输入内容 Hash、`indexVersionId`、检索范围与结果快照、模型配置版本、实际路由结果、提示词版本、开始/结束时间、错误摘要 |
| Finding | `findingId`、`runId`、分类、严重度、依据类型、标题、描述、影响、建议确认项、初始生成内容、处置状态、创建时间 |
| Finding 证据 | `evidenceId`、`findingId`、`assetId`、`assetVersionId`、`chunkId/chunkKey`、内容 Hash、标题路径、原文定位、引用摘录、检索分数、降级状态 |
| Finding 处置 | `actionId`、`findingId`、前后状态、处置说明、操作者显示信息、操作时间 |
| 评审问答消息 | `messageId`、`runId`、角色、内容、引用证据、模型配置版本、实际模型、用量摘要、创建时间、失败摘要 |
| 生成式模型配置版本 | `configVersionId`、配置内容快照、默认模型、路由/降级策略、参数、提示词版本、创建时间、状态 |
| 模型调用记录 | `callId`、`runId/messageId`、实际模型、路由原因、耗时、Token、重试、状态、脱敏错误摘要 |

### 6.2 一致性规则

1. 评审运行一经创建，输入资产版本、检索证据、模型配置和提示词版本不可原地修改。
2. 成功运行的每个 `evidence_supported` Finding 至少有一条可打开的固定证据；证据失效时必须在查看时明确提示，而非跳转到最新资料。
3. 人工处置只能追加，不得篡改模型初始输出或历史处置记录。
4. 新需求资产版本、索引重建或模型配置更新不修改历史运行；仅影响后续运行，并触发相应的可能过期标记。
5. 运行完成前的候选输出不可作为正式 Finding 展示或参与统计。
6. 配置、模型或检索异常不得被转换为“无问题”或空白成功结论。
7. 一期的资产、资产版本、Chunk、索引版本和证据稳定标识必须继续复用，不得另建不可关联的评审资料容器。

## 7. 验收标准

### AC-201：基于固定需求版本创建评审

**前置条件**：知识库存在至少两份 `ready` 的需求资产版本，且存在活动索引和可用评审模型配置。

**操作**：选择其中一个历史需求资产版本创建评审并启动。

**预期结果**：

- 运行保存选定的 `assetVersionId`、内容 Hash、`indexVersionId`、检索范围和模型配置版本；
- 之后上传该需求的新版本，不改变历史运行打开的原文和证据；
- 页面不允许用未入库文本或无 `ready` 版本启动正式评审。

### AC-202：结构化 Finding 与证据定位

**前置条件**：需求文档中存在可识别的业务规则、缺失项和关联资料。

**操作**：完成一次评审，打开评审概览和证据引用，点击一个有依据 Finding。

**预期结果**：

- Finding 展示分类、严重度、依据类型、问题、影响和建议确认项；
- 有依据 Finding 至少关联一条固定资产版本、标题路径和原文位置；
- 点击后按 `assetVersionId` 打开原文并定位命中范围；
- 无依据结果显式标记为待确认或模型推测，不伪装为已证实事实。

### AC-203：人工处置不覆盖历史结果

**前置条件**：存在成功运行和至少一个 `open` Finding。

**操作**：依次将 Finding 标记为确认、驳回或已解决，并填写处置说明；随后重新运行评审。

**预期结果**：

- 每次处置都有独立记录、时间和说明；
- 原始 AI Finding 与证据不被改写；
- 新运行具有新的运行 ID 和独立结果，旧运行及人工处置仍可查看。

### AC-204：取消、失败与检索降级状态

**前置条件**：可触发运行中的取消、模型调用失败和混合检索向关键词检索降级。

**操作**：分别取消运行、模拟模型不可用，并执行发生检索降级的评审。

**预期结果**：

- 取消运行标记为 `cancelled`，不产生未完成的正式 Finding；
- 模型不可用时标记为 `failed` 并显示脱敏错误摘要，不展示本地模拟结果；
- 检索降级时，运行与相关证据清楚展示降级原因；无活动索引时不允许仅依赖模型启动评审。

### AC-205：模型配置版本与受控路由

**前置条件**：部署环境已注册至少两个支持结构化输出的模型，其中一个可作为回退模型。

**操作**：保存新的需求评审模型配置，启动运行；再使主模型不可用并启动另一运行。

**预期结果**：

- 保存产生新的模型配置版本，已存在运行不受影响；
- 智能路由关闭时使用默认模型；开启且主模型失败时，只有在允许降级、回退模型满足能力时才切换；
- 页面和调用记录显示实际模型、重试次数和降级原因；
- 不返回或保存 API Key、完整端点和 Authorization 请求头。

### AC-206：模型检测与输出校验

**前置条件**：存在一个可用模型和一个返回不符合结构化 Schema 的测试模型响应。

**操作**：对已注册模型执行连通性检测，并使用异常响应模型启动评审。

**预期结果**：

- 检测展示可用性、结构化输出能力、检测时间和脱敏错误；
- 异常响应导致运行显示结果校验失败，不生成成功统计和可处置 Finding；
- 页面不提供任意端点和明文密钥的输入或回显。

## 8. 交付边界与后续衔接

本期交付的是以知识库证据为基础的需求评审闭环和受控生成式模型配置，不是需求文档编辑、自动决策或工程执行平台。

- AI 评审结果和问答仅为派生产物；只有人工将其编辑并保存为独立资料版本后，才可按一期统一资产流程进入知识库。
- Phase 3 技术方案分析应复用本期已确认的 Finding、人工处置和固定证据，但必须创建独立的方案运行与输入快照。
- Git、测试、权限、审批、成本治理及 Prompt/Agent 平台应在后续阶段分别定义，不能因本期预留字段或原型入口而视为已交付。

## 9. 前置条件与待确认事项

### 9.1 前置条件

- 一期知识库可稳定提供需求资产版本、Chunk、活动索引和固定证据定位；
- 至少一组可用于试点评审的 Markdown/纯文本需求资料及关联业务规则资料；
- 部署环境已注册至少一个支持结构化输出的受控生成式模型；
- 已确认本期允许发送至模型服务的项目资料范围与数据边界；
- 已准备典型需求、预期问题、证据来源和人工处置样例作为评审质量验收集。

### 9.2 待确认事项

1. Finding 的严重度定义、默认处置时限及是否需要后续对接工单系统；
2. 单次需求评审支持的最大输入长度，以及超长需求的后续分段评审策略；
3. 首批可注册的生成模型来源、模型清单、上下文窗口、数据传输边界和保留策略；
4. 关键词降级时是否允许继续生成评审，以及对结论可信度的展示规则；
5. 需求评审提示词模板的维护责任、发布流程和变更评审机制；
6. 试点质量验收集、Finding 召回率/证据准确率等度量口径，以及人工复核责任人；
7. 后续何时引入成员权限、审批、成本配额、Prompt/Agent 编排和多模态资料解析。
