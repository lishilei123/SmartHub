## 2. 背景与目标

一期已建立项目资料的知识资产、不可变资产版本、Chunk、活动索引和固定证据定位能力。当前原型已展示需求列表、分析运行进度、分析概览、原始文档、版本差异、功能树、证据引用、需求问答，以及“AI 模型与路由”配置入口；但上述需求 AI 与模型路由仍使用本地示例数据，不能形成可追溯的真实评审闭环。

本期目标是在一期知识库基础上，正式启用基于 Pi Agent Framework 的首个业务 Agent——`RequirementAnalysisAgent`（需求分析 Agent），将“需求分析”升级为可执行、可复核、可人工处置的**需求评审**能力，并提供支撑该 Agent 的大模型配置、路由、Prompt、工具与执行治理。该 Agent 可调用受控工具、联网检索和 MCP 适配器完成多步证据分析，但正式运行状态、输入快照、权限边界、结果校验与人工处置仍由 SmartHub 业务流程控制。

### 2.1 本期目标

1. 以已创建的项目版本作为需求分析的强制工作空间，从该版本绑定的确定需求知识资产版本创建评审，固定项目版本、需求绑定、输入资料、检索范围、索引版本、模型配置和提示词版本。
2. 通过大模型生成结构化的需求评审结果，识别需求缺口、歧义、冲突、边界、验收风险和测试风险。
3. 每条结论均能打开至固定资料版本的证据位置；缺少证据的内容必须明确标识为待确认或模型推测。
4. 支持人工确认、驳回、解决和补充评论，不覆盖原始 AI 输出，并保留历史评审运行。
5. 支持受控配置生成式大模型、模型参数、路由与降级策略；凭据不进入浏览器和业务数据库明文字段。
6. 真实模型不可用、检索降级、输出结构校验失败或任务取消时，页面必须显示真实状态，不得回退为示例结果或伪造成功。
7. 基于 Pi Agent Framework 创建并启用一个版本化的 `RequirementAnalysisAgent`，由该 Agent 编排需求评审内部的检索、证据读取、引用校验与结构化结果提交，并固定 Agent、Prompt、工具集、MCP 与网络策略版本。
8. 支持按工具风险和运行策略受控调用联网、MCP 及外部操作；未授权、高风险或需要人工确认的操作不得由模型绕过策略直接执行。

### 2.2 本期非目标

本期不实现以下能力，且验收不得依赖它们：

- 技术方案生成、架构设计评审、代码影响分析、Git/PR/Commit 集成；
- 测试用例或测试脚本生成、测试执行、CI/CD、日志诊断和自愈；
- 图片、原型图、HTML、PDF、Word、Excel 等专用格式的解析与多模态评审；
- 需求原文的富文本编辑、AI 改写、划词重写、User Story 转换、建议自动应用或自动替换；
- Jira、PingCode、Webhook 等外部项目管理系统的推送、同步和跨项目工单联动；
- 面向用户的通用可视化 Prompt/Agent 编排平台、任意 Agent 创建与发布、任意工具或 MCP 服务注册；
- 多 Agent 协作、Agent 间任务委派与汇总；第二期每次需求评审仅运行一个 `RequirementAnalysisAgent`；
- 允许用户在浏览器中配置任意联网端点、密钥、Shell、代码执行环境或不受控外部操作；
- 完整的组织、角色、ACL、审批流、SSO、合规审计和成本结算体系；
- 多项目创建、项目选择器、项目切换、跨项目资源共享和跨项目分析；当前平台实例固定服务一个系统初始化项目；
- 将 AI 评审结论、聊天内容自动写回或自动发布为知识库正式资料。

上述非目标限定的是本期**不交付通用 Agent 创建平台和多 Agent 协作能力**，不影响本期以 Pi Agent Framework 为统一底座创建并启用 `RequirementAnalysisAgent`，也不禁止该 Agent 依赖受控 Prompt、内部工具、联网检索、MCP 适配器和外部操作执行器。所有依赖必须预注册、版本化、最小授权、可取消、可审计，并受本文件 5.3 节约束。

## 3. 范围、术语与原则

### 3.1 核心对象

```text
平台实例（单项目）
  └─ 当前项目（系统初始化）
       ├─ 默认知识库（项目级共享）
       │    └─ 需求知识资产
       │         └─ 不可变资产版本
       └─ 项目版本（项目空间隔离边界）
            └─ 项目版本需求绑定
                 └─ 引用一个不可变需求资产版本

需求评审（归属项目版本与需求绑定）
  └─ 评审运行（固定输入与配置快照）
       ├─ 评审结论 / Finding
       │    ├─ 证据快照
       │    └─ 人工处置记录
       └─ 评审问答会话

大模型配置
  └─ 已注册模型来源
       └─ 模型定义
            └─ 评审模型配置版本 / 路由策略

Pi Agent Framework
  └─ RequirementAnalysisAgent（第二期唯一业务 Agent）
       └─ Agent 定义版本
            ├─ Prompt 模板版本
            ├─ 工具集与权限策略版本
            ├─ MCP / 网络策略版本
            └─ 执行限制与外部操作策略
```

| 对象 | 定义与规则 |
|---|---|
| 项目版本（`ProjectVersion`） | 项目空间业务数据的强制隔离边界，状态为 `open`、`locked` 或 `archived`。只有 `open` 版本允许新增/替换需求绑定和发起新评审。 |
| 项目版本需求绑定 | 当前项目版本对一个固定需求 `assetVersionId` 的引用关系，不复制知识资产内容；知识资产更新不得自动改变既有绑定。 |
| 需求评审 | 在一个确定项目版本中，基于一条有效需求绑定发起的评审逻辑实体。它不是知识资产，也不替代原需求文档。 |
| 评审运行（`ReviewRun`） | 一次可取消、可追溯的 `RequirementAnalysisAgent` 业务运行。每次重新评审都创建新运行，历史运行不可覆盖。 |
| Finding | 可人工处置的结构化评审结论，例如缺口、歧义、冲突、边界或风险。 |
| 证据快照 | Finding 所引用的固定资产版本、Chunk、内容 Hash、标题路径、原文定位和引用摘录。 |
| 大模型来源 | 由部署环境受控注册的生成式模型服务，不等同于一期 Embedding 来源。 |
| 评审模型配置 | 供需求评审使用的模型、参数、路由及降级策略的不可变版本。 |
| Pi Agent Framework | SmartHub Phase 2～Phase 4 共用的 Agent 框架，负责业务 Agent 的模型交互、消息状态、工具循环、事件和取消传播。 |
| `RequirementAnalysisAgent` | 第二期创建并启用的唯一业务 Agent，负责一次需求评审中的多轮推理、检索、证据读取、引用校验、结构化结果提交和评审问答。 |
| Agent 定义版本 | Agent 的系统指令、Prompt 引用、允许工具、执行限制和安全策略组成的不可变版本。 |
| 受控工具 | 由服务端注册并具有输入/输出 Schema、权限、风险等级、超时、幂等性与数据策略的确定性能力。 |
| MCP 适配器 | 将受信 MCP Server 暴露的能力转换为 SmartHub 统一工具描述的接入层；MCP 本身不绕过工具权限、参数校验和审计。 |
| Agent 执行 | 绑定一个 `ReviewRun` 的内部多轮执行实例；其消息与工具事件是过程记录，不替代 `ReviewRun` 业务状态。 |

### 3.2 评审原则

- **单项目上下文**：平台不展示项目选择器，所有页面自动使用系统初始化的唯一项目；用户进入项目空间后只创建、选择和切换项目版本。
- **项目版本前置**：未创建或未选择项目版本时不得进入需求分析工作台。项目知识库与系统配置共享，Review、Run、Finding、问答及其统计必须按 `projectVersionId` 隔离。
- **固定输入**：每次运行只能使用创建时绑定的需求资产版本、检索范围、活动索引版本和配置版本；不得在运行中悄然改用“当前最新资料”。
- **证据优先**：涉及项目事实的 Finding 必须关联证据；没有足够依据时，结论只能标识为“待确认”或“模型推测”。
- **人机分工**：AI 提供结构化问题、风险和建议，是否采纳由人工决定；系统不得自动修改需求内容或关闭问题。
- **可复现边界**：模型输出具有非确定性，但同一运行必须可还原输入、模型/参数、提示词、检索证据和最终输出。
- **配置分离**：生成式大模型配置与一期 Embedding、切分、索引配置分开管理；调整生成模型不得触发知识库索引重建。
- **统一框架、分期创建 Agent**：Pi Agent Framework 是跨阶段统一底座；第二期启用一个需求分析 Agent，第三期创建独立技术方案分析 Agent，第四期由多个测试 Agent 协作。每个 Agent 具有独立定义、Prompt、工具、权限和版本。
- **业务流程外层、Agent 执行内层**：排队、租约、取消、路由、重试、降级、Schema 校验、正式落库和人工处置由 SmartHub 控制；Agent 只在单次受控执行内部决定如何调用已授权工具。
- **最小权限**：工具、MCP、网络和外部操作默认关闭，按 Agent 定义版本显式开放；只读能力与写操作分级管理，高风险操作必须经过人工确认。
- **不信任工具内容**：知识库、网页、MCP 与外部系统返回内容均视为不可信数据，不得作为系统指令执行；服务端必须防止提示词注入改变工具权限或运行策略。

## 4. 用户界面与主要流程

### 4.1 需求评审页面

沿用现有三栏原型并升级为“顶部操作区 + 三栏评审工作台”，替换所有本地 fixture。页面以项目版本隔离、固定版本评审、证据定位和人工处置为核心，不提供需求原文编辑能力。未创建或未选择项目版本时，页面只显示版本说明与“新建项目版本”主操作，不加载需求列表和评审工作台。

#### 4.1.1 顶部操作区

- 展示返回入口、当前项目版本、需求名称、需求资产版本、当前运行 ID、运行状态、进度、检索/模型降级状态和固定输入快照入口；项目发布版本与需求资产版本必须使用不同标签和文案展示。
- 根据运行状态提供“开始评审、取消运行、重新评审”操作；已成功运行不得原地覆盖，重新评审必须创建新的 `ReviewRun`。
- 支持复制或导出 Markdown 评审报告；报告必须携带需求资产版本、运行 ID、生成时间和证据引用，不得被表述为已经修改或发布的正式需求文档。
- 不提供“保存原文版本”“AI 改写”“自动应用”等会修改固定输入的操作。需求变化必须先通过一期知识库形成新的 `ready` 资产版本，再创建新评审运行。

#### 4.1.2 左侧需求评审列表

- 项目版本由项目空间顶部版本选择器确定；列表只查询当前 `projectVersionId` 下的需求绑定、评审和统计，不提供跨版本混合列表。
- 按关键字、运行状态和未处理 Finding 数量筛选。
- 展示需求名称、需求资产版本、最新评审状态、最新运行时间、未处理 Finding 数量和“可能过期”标记。
- 支持“上传并加入当前版本”和“从知识库加入”。上传接受 `.md`、`.txt` 和 `.zip`，复用一期入库任务；ZIP 仅作为批量封装，包内 Markdown/纯文本进入解析，图片作为附件保留，其他格式跳过并反馈。
- 支持“新建需求评审”，并在同一项目版本和需求绑定下切换历史评审运行；跨两个项目版本比较必须显式选择双方，不得混入默认历史。
- 列表允许折叠；折叠只改变当前页面布局，不改变评审状态。

#### 4.1.3 中间评审工作区

中间区域展示需求基本信息、固定输入快照、真实运行进度，并提供以下五个视图：

| 视图 | 页面能力 |
|---|---|
| 评审概览 | 展示功能项、Finding、阻断/高风险、待确认、有证据和无证据数量；按类型、严重度、依据类型和人工处置状态过滤结构化 Finding。 |
| 原始文档 | 只读打开本次运行绑定的固定资产版本；Markdown 自动生成章节大纲，支持目录跳转、滚动章节同步、证据范围高亮和引用段落到问答。 |
| 版本差异 | 比较同一资产的两个固定版本，展示真实新增、删除、修改内容及受差异影响的 Finding；不得展示虚构 Diff。 |
| 功能树 | 按需求章节或功能域展示功能、业务规则、异常场景、边界和验收要点；点击节点过滤关联 Finding 并定位原文章节。 |
| 证据引用 | 按 Finding 展示资料、逻辑路径、固定版本、标题路径、原文位置、引用摘录、检索得分和降级状态。 |

#### 4.1.4 右侧评审问答区

- 仅围绕当前成功评审运行的固定需求版本、检索证据快照、结构化结果、会话历史及已授权只读工具提问。
- 支持从 Finding 或原文选区发起追问；引用原文时保存固定 `assetVersionId` 和原文定位，而不是复制一个会失去来源的可编辑上下文。
- 每条回答显示状态、证据引用、实际模型、重试/降级摘要和生成时间；失败时显示真实原因，不展示模拟回答。
- 问答区允许折叠；折叠不清空会话。问答不得修改 Finding 状态或需求原文。

#### 4.1.5 操作与历史记录

- 展示当前评审的创建、运行、取消、人工处置、问答及错误事件；本期不将其定义为组织级合规审计。
- 页面必须区分需求资产版本历史、评审运行历史和 Finding 人工处置历史，不能使用一个含义模糊的“版本/历史”列表混合展示。

### 4.2 创建并运行评审

```text
用户进入项目空间
  → 选择已存在项目版本；若不存在则先新建 open 项目版本
  → 上传 Markdown/纯文本/ZIP 并等待入库 ready，或从知识库选择 ready 的需求资产版本
  → 将固定需求资产版本绑定到当前 projectVersionId
  → 用户选择当前项目版本中的有效需求绑定
  → 选择或确认逻辑路径检索范围
  → 系统校验活动索引、模型配置、Agent 定义、工具权限和输入长度
  → 创建需求评审与 queued 的评审运行
  → 固定项目版本、需求绑定、输入、索引、检索、模型、Agent、Prompt、工具集、MCP 与网络策略快照
  → RequirementAnalysisAgent 调用已授权工具检索关联资料、读取原文并校验证据
  → Agent 在回合、Token、工具调用和超时限制内提交候选结构化结果
  → 校验结构化输出与证据完整性
  → 保存结果、Finding、证据和用量记录
  → 运行 succeeded，等待人工处置
```

若运行失败、取消或校验不通过，保留运行记录、失败步骤和错误摘要；不得生成“分析完成”的示例结果。

### 4.3 Agent 内部执行流程

```text
Review Worker 取得运行租约
  → 装载固定 ReviewRun、Agent、Prompt、模型、工具与权限快照
  → Pi Agent Framework 启动 RequirementAnalysisAgent 执行
  → RequirementAnalysisAgent 按需调用知识库检索、证据读取、引用校验、联网或 MCP 工具
  → Tool Runtime 校验参数、身份、权限、风险、超时、网络策略和审批条件
  → 工具结果脱敏、限长并记录后返回 Agent
  → Agent 调用 submit_review_result 提交候选结果
  → SmartHub 独立执行 Schema、证据与一致性校验
  → 校验通过后原子保存正式 Finding；否则运行失败并保留过程记录
```

- `ReviewRun` 是业务事实来源；Pi Agent Framework 的内存状态、消息历史或 Agent 结束事件不得直接代表运行成功。
- Agent 不得直接写入 Finding、处置记录、知识资产或活动索引；`submit_review_result` 仅提交候选结果，由 SmartHub 校验并落库。
- Worker 失去租约、用户取消、达到执行限制或服务端关闭时，必须向模型请求和正在执行的工具传播取消信号。

### 4.4 人工评审闭环

```text
查看 Finding 与固定证据
  → 确认 / 驳回 / 标记已解决 / 保持待确认
  → 可填写处置说明与补充评论
  → 需求资料更新后创建新的资产版本
  → 人工发起新的评审运行
  → 旧运行保留，并标记为可能过期
```

### 4.5 Finding、原文与问答联动

```text
点击 Finding 或证据
  → 按 evidence.assetVersionId 打开固定原文
  → 滚动到 evidence.location
  → 高亮证据范围并保持 Finding 选中状态

点击原文高亮范围或功能树节点
  → 展示关联 Finding
  → 可定位证据、执行人工处置或引用到评审问答
```

- 双向联动只使用正式保存的 Finding 与 Evidence 关系；无证据 Finding 不得伪造原文高亮。
- 高亮、当前 Tab、折叠状态和筛选条件属于前端视图状态，不得改写固定原文、Finding 初始输出或证据快照。
- 当证据版本不存在、无权限或定位失败时，页面保留 Finding 上下文并显示明确错误，不得自动跳转到资产最新版本。

## 5. 功能需求

### 5.1 需求评审

#### FR-201：在项目版本中绑定需求并创建评审

- 每个需求评审必须显式归属一个已存在的 `projectVersionId`。未创建或未选择项目版本时，不加载需求分析数据、不允许新建评审或启动运行，并提供“新建项目版本”主操作。
- 项目版本状态至少包括 `open`、`locked`、`archived`；只有 `open` 版本允许新增/替换需求绑定和启动新运行，`locked`、`archived` 版本只允许查看历史数据。
- 版本管理提供独立“删除版本”操作并要求二次确认。删除时移除该版本的需求绑定，但不删除全局知识库中的资产和固定资产版本；版本仍被其他版本作为继承来源时禁止删除。正式 ReviewRun 持久化上线后，存在需保留审计记录的版本禁止物理删除，只允许归档。
- 新建项目版本时可选择基于一个已有项目版本继承其有效需求绑定；继承只创建指向同一固定 `assetVersionId` 的新绑定，不复制 Review、Run、Finding、处置记录或问答会话。
- 继承后的 `open` 版本可独立新增、替换或移除需求绑定；这些修改不得回写来源版本。替换绑定必须由用户明确选择新的 `ready assetVersionId`，页面继续按原固定版本展示，不能因为知识库出现新版本而自动漂移。
- 用户可从当前项目默认知识库选择资料类型为 `requirement` 的 `ready` 资产版本并加入当前项目版本。默认可推荐该资产的活动版本，但最终必须保存用户确认的固定 `assetVersionId`。
- 评审页支持上传 `.md`、`.txt` 和 `.zip`。上传、格式校验、解包、内容解析、索引和失败反馈复用一期知识库流程；资产版本达到 `ready` 后，系统在同一业务操作中将其绑定到发起上传时的 `projectVersionId`。ZIP 中图片作为附件保留但不参与本期正文分析，其他不支持格式跳过并逐项反馈。
- 页面至少展示项目版本、需求标题、`requirementBindingId`、`assetId`、`assetVersionId`、资料逻辑路径、需求资产版本号、入库时间和当前活动索引版本，并明确区分“项目版本”和“需求文档版本”。
- 需求尚未入库、绑定不存在/失效、资产版本非 `ready`、无活动索引或模型配置不可用时，不允许启动运行，并显示可理解的原因和下一步。
- 切换项目版本时，前端必须取消旧版本的在途读取请求并清空旧视图数据，再按新 `projectVersionId` 重新加载；服务端已运行任务继续归属原项目版本，不因页面切换迁移或取消。
- 本期不提供绕过知识库、直接粘贴任意文本后即发起正式评审的入口。

#### FR-202：评审运行快照与状态

- 创建运行时必须固定并保存：服务端派生的唯一 `projectId`、前端显式选择的 `projectVersionId`、`requirementBindingId`、`knowledgeBaseId`、需求 `assetId`、`assetVersionId`、输入内容 Hash、`indexVersionId`、逻辑路径检索范围、检索参数与检索结果快照、评审模型配置版本、实际路由结果、`RequirementAnalysisAgent` 稳定 ID、Agent 类型与定义版本、Pi Agent Framework/Runtime 版本、提示词模板版本、工具集版本、MCP 配置版本、网络策略版本、权限快照、执行限制和触发时间。客户端不得提交或切换任意 `projectId`。
- 运行状态至少包括 `queued`、`running`、`succeeded`、`failed`、`cancelled`；状态与进度必须由服务端任务真实返回。
- 用户可取消未完成运行；取消后停止后续模型调用和工具调用，不写入未完成的结构化结论，已保存的 Agent 回合、工具调用、步骤日志和错误信息允许查看。
- 同一需求可存在多个历史运行；重新评审必须创建新 `ReviewRun`，不得覆盖历史输入、结论、证据或人工处置。
- 页面进度至少区分“校验输入、固定运行快照、检索关联资料、读取证据、AI 分析、引用校验、结构化结果校验、保存正式结果、等待人工评审”；每一步状态必须来自服务端运行事件或持久化状态。

#### FR-203：评审维度与结构化输出

每次成功运行至少生成以下结构化结果：

| 输出 | 要求 |
|---|---|
| 需求摘要 | 基于输入版本生成，明确标识为 AI 摘要。 |
| 功能树 | 以需求章节或功能域组织功能、业务规则、异常场景和验收要点。 |
| Finding | 按缺口、歧义、冲突、边界、验收风险、测试风险分类，包含问题描述、影响、建议确认项、严重度和结论依据。 |
| 评审统计 | 展示功能项、Finding、待确认项、风险项及有证据项数量；统计必须由本次持久化结果计算。 |
| 版本差异 | 当用户为同一资产选择两个固定版本时，展示真实的版本内容差异和与差异相关的 Finding；无可比较版本时明确提示。 |

- Finding 严重度使用 `blocker`、`high`、`medium`、`low` 四档；页面可映射为阻断、高、中、低风险标签。
- Finding 依据类型使用 `evidence_supported`、`needs_confirmation`、`model_inference` 三种。`evidence_supported` 必须至少有一条有效证据；其余两类必须明确说明未证实原因。
- 服务端必须校验模型输出符合已定义的结构化 Schema。输出不合法、必填字段缺失或证据引用无法解析时，运行标为失败或进入明确的“结果校验失败”步骤，不得将不完整文本伪装为成功结果。
- Agent 只能通过受控结果提交工具产生候选结果；模型正文、部分流式内容或普通工具返回不得直接成为正式 Finding。
- 评审概览必须按持久化正式结果计算统计，并支持按 Finding 类型、严重度、依据类型和人工处置状态组合过滤；前端不得根据流式文本自行推算正式统计。

#### FR-204：固定证据与原文定位

- 每条 Finding 可关联零到多条证据；证据快照至少包含 `assetId`、`assetVersionId`、`chunkId` 或 `chunkKey`、Chunk 内容 Hash、标题路径、行号/字符范围、引用摘录、`indexVersionId` 和检索得分/降级状态。
- 评审来源文档本身也必须以固定 `assetVersionId` 打开，不得因该资料后续更新而跳转到最新版本。
- “证据引用”视图按 Finding 展示资料名称、逻辑路径、标题路径和原文定位；点击后打开对应固定版本并高亮命中范围。
- 原始文档视图对 Markdown 生成章节大纲，支持目录与滚动位置同步；大纲仅用于导航，不替代固定证据定位。
- 页面支持从 Finding/证据定位原文，并从已高亮原文范围反查正式关联的 Finding；无正式 Evidence 关系时不得创建伪关联。
- 模型不得将无法映射至固定资料版本的内容表述为项目事实；无证据 Finding 必须使用“待确认”或“模型推测”标签。
- Agent 通过联网或 MCP 获得的内容不能自动成为项目事实；只有能够映射至本次允许证据范围并通过引用校验的内容才能标记为 `evidence_supported`，其他内容必须标记为外部参考、待确认或模型推测。

#### FR-205：Finding 人工处置

- Finding 初始状态为 `open`，人工可更新为 `confirmed`、`dismissed`、`resolved` 或 `needs_follow_up`。
- 人工处置必须追加保存状态、处置说明、操作者显示信息和时间；不得改写模型最初生成的标题、描述、严重度、依据类型及证据快照。
- `dismissed`、`resolved` 和 `confirmed` 均应要求填写或确认处置说明；允许为空的临时保存仅使用 `open` 或 `needs_follow_up`。
- 页面支持按严重度、依据类型和处置状态过滤 Finding，并在需求列表与概览中展示未处理数量。
- Finding 卡片至少提供查看证据、定位原文、确认、驳回、标记已解决、保持待确认、补充处置说明和针对此问题追问；所有处置仍遵循追加写规则。
- 本期不实现多级审批、指派、催办、消息通知或跨项目工单同步。

#### FR-206：评审问答

- 问答必须绑定一个已成功完成的评审运行，由同一 `RequirementAnalysisAgent` 的问答模式执行；输入范围仅包括该运行的需求资产版本、固定检索证据、结构化评审结果、会话历史及 Agent 定义允许访问的只读工具。
- 每轮回答保存问题、回答、实际模型配置版本、Agent/Prompt/工具集版本、执行时间、用量摘要、工具调用摘要和引用证据；回答中的项目事实遵循 FR-204 的证据规则。
- 当模型不可用、请求超时、证据不足或输出校验失败时，显示明确失败原因；不得使用本地模拟答案代替。
- 问答会话是 AI 派生产物，不自动进入知识库；后续如需沉淀，由人工另行保存为独立资料版本。
- 用户可引用选中的 Finding 或固定原文范围追问；消息引用至少保存 `assetVersionId`、标题路径、行号/字符范围和摘录 Hash，并在发送与历史查看时重新校验读取权限。

#### FR-207：过期标记与历史查看

- 当评审输入资产产生新的活动版本，或该评审所用索引版本不再是当前活动版本时，历史评审显示“可能过期”，但仍可查看其完整快照。
- 过期提示不得自动改变 Finding 状态，也不得自动关闭或重新运行评审。
- 用户可在同一需求资产的固定版本之间发起比较；跨两个项目版本比较时必须显式选择两个 `projectVersionId` 及其绑定，比较对象、差异结果和所选运行必须清楚区分。
- 页面分别展示需求资产版本历史、当前项目版本内的评审运行历史和 Finding 处置历史；历史运行切换后，所有概览、原文、证据与问答必须同时绑定所选 `projectVersionId` 与 `runId`，不得混用其他项目版本或最新运行数据。

#### FR-208：输入长度与检索降级

- 启动前，系统应根据实际路由模型的上下文窗口、输出预留和固定需求内容评估输入是否可执行。
- 超出本期支持上限时，系统必须拒绝启动并说明需求版本过长及可选处理方式；不得静默截断需求原文后仍标记为“完整评审”。
- 检索发生关键词降级或 Reranker 绕过时，可按已保存策略继续运行，但运行、Finding 证据和页面必须显示降级状态及原因。
- 无活动索引、检索完全不可用或需求输入版本不可读取时，运行必须失败，不得仅依赖模型记忆生成结论。

#### FR-209：评审工作台与报告导出

- 页面采用“顶部操作区 + 左侧评审列表 + 中间评审工作区 + 右侧评审问答区”；左右两栏可折叠，中间工作区始终保留最小可读宽度。
- 页面 URL 或等价路由状态应能稳定表达 `projectVersionId`、`reviewId`、`runId`、当前视图及可选 `findingId/evidenceId`，推荐 `/versions/{projectVersionId}/requirement-reviews/{reviewId}/runs/{runId}`；唯一项目由服务端会话确定，不进入用户可切换路由。刷新或分享链接后不得回退到默认/最新项目版本或最新运行。
- 运行中页面通过增量事件或轮询展示真实阶段；连接中断后以服务端状态恢复，不得继续播放本地假进度。
- 原始文档、证据和历史运行均为只读；本期不提供 AI 改写、自动应用、富文本编辑、User Story 转换或“保存原文版本”。
- 支持复制或导出 Markdown 评审报告。导出内容至少包含固定输入摘要、运行快照标识、正式统计、Finding、证据引用、降级说明和人工处置状态；默认不导出问答全文和敏感执行日志。
- 窄屏下允许将左侧列表和右侧问答收纳为抽屉，但不得隐藏运行状态、证据依据类型和人工处置入口。
- 每个 Agent 定义必须设置最大回合数、最大工具调用数、单工具结果长度、总 Token 预算和总执行超时；达到限制时运行必须停止并显示真实原因，不得无限循环或以部分结果标记成功。

### 5.2 大模型配置与路由

#### FR-301：生成式模型来源管理

- 系统将一期的 Embedding/检索配置与本期生成式大模型配置分为独立模块。知识库索引继续使用一期配置；修改本期模型配置不触发索引重建。
- 首期生成式调用仅支持部署环境注册的 OpenAI-compatible Chat Completions 协议或等价的受控文本生成适配器；接入其他协议须新增适配器并单独验收。
- 模型来源至少包含稳定 ID、显示名称、协议类型、启用状态、部署侧端点引用、凭据引用、可用模型列表和健康状态。
- 浏览器、需求评审 API 响应、运行快照和业务数据库不得保存或返回 Base URL、API Key、Token 等明文凭据。部署环境负责安全注入端点和密钥。
- 本期不提供用户在浏览器中新增任意端点、录入密钥或探测任意第三方服务的能力。

#### FR-302：模型定义与能力声明

每个可选生成模型至少维护：

- `modelId`、显示名称、所属模型来源、启用状态；
- 最大上下文长度、最大输出 Token、支持的语言或文本能力声明；
- 是否支持结构化 JSON 输出；本期需求评审仅允许选择支持结构化输出的模型；
- 默认超时、最大重试次数和健康状态；
- 最近一次连通性检测时间、检测结果与脱敏错误摘要。

模型不可用、被停用、能力不满足或超出输入上限时，不得成为新运行的路由目标；历史运行仍保存原实际模型标识与配置快照。

#### FR-303：评审模型配置版本

- “AI 模型与路由”页面提供需求评审模型配置草稿、未保存提示、保存和版本历史查看能力。
- 配置至少包含：默认模型、温度、最大输出 Token、请求超时、重试次数、结构化输出模式、智能路由开关、降级开关及候选回退模型顺序。
- 保存后生成新的不可变配置版本；无变化不生成冗余版本。
- 配置保存仅影响后续创建的评审运行；已创建或正在执行的运行继续使用其固定配置快照。
- 系统内置的需求评审提示词以受控模板版本标识参与快照。本期配置页只展示 Agent、Prompt、工具集、MCP 与网络策略的版本和摘要，不提供面向普通用户的通用编辑、发布或任意工具授权能力；其版本由部署侧受控配置发布。

#### FR-304：模型路由、重试与降级

- 智能路由关闭时，所有新运行使用当前默认模型；开启时，路由必须根据已保存的模型能力、上下文长度、启用状态和健康状态选择模型，并记录实际选择理由。
- 主模型发生可重试错误时，先按配置重试；重试耗尽后仅当“允许降级”已开启且候选模型满足输入与结构化输出能力时才切换回退模型。
- 发生模型降级时，运行页面、用量记录和最终结果必须展示实际模型与降级原因；不得将回退模型结果标识为主模型结果。
- 无可用模型时，运行进入 `failed` 并保留错误摘要；不得回退为固定示例、缓存答案或未验证旧结果。
- 模型路由、重试与降级由 SmartHub 在 Agent 执行边界外控制；Agent 不得自行修改模型、密钥或回退顺序。发生模型切换时应创建可追踪的调用阶段，并继续遵守同一运行的固定输入、Agent、Prompt、工具与权限快照。

#### FR-305：模型检测与安全反馈

- 配置页支持对已注册模型执行受控连通性与结构化输出检测，检测请求使用部署侧凭据，且不使用项目需求资料作为测试内容。
- 检测结果至少返回可用/不可用、模型能力、检测时间和脱敏错误摘要；不得回显请求头、密钥、完整端点或服务端原始堆栈。
- 前端保存或检测期间显示真实加载、成功和失败状态；保存失败不得更新页面“已保存”版本。

#### FR-306：运行可观测性

- 每次模型调用至少记录运行/会话 ID、Agent 执行 ID、配置版本、实际模型、路由与降级结果、开始/结束时间、耗时、输入/输出 Token（服务提供时）、重试次数、结果状态和脱敏错误摘要。
- 页面提供评审运行级别的执行概览；本期不提供组织级账单、价格计算、配额分配或审计导出。
- 运行日志不得记录完整 API Key、Authorization 请求头或未经脱敏的敏感配置值。

### 5.3 Pi Agent Framework、业务 Agent 与工具治理

#### FR-401：Pi Agent Framework 与需求分析 Agent

- SmartHub 从第二期开始统一采用 Pi Agent Framework 作为业务 Agent 框架；第二期必须创建、发布并启用一个 `RequirementAnalysisAgent`，每个 `ReviewRun` 仅启动该 Agent 的一个执行实例，不引入多 Agent 协作。
- `RequirementAnalysisAgent` 是可版本化业务对象，而不是临时 Prompt 调用：它必须具有稳定 `agentId`、类型 `requirement_analysis`、定义版本、Prompt、模型配置、工具集、权限、MCP/网络策略和执行限制。
- Pi Agent Framework 负责单个业务 Agent 的模型交互、消息状态、工具循环、流式事件和取消传播；PostgreSQL Job/Worker 继续负责排队、租约、恢复、重试、取消与最终业务状态。
- SmartHub 可通过内部 `PiAgentRuntimeAdapter` 封装框架 SDK 和生命周期，但不得将 Pi Agent Framework 抽象成任意可替换框架而丢失统一 Agent 语义；业务数据层不得直接使用框架内存消息作为事实来源。
- Pi Agent Framework 依赖必须锁定明确版本并经过兼容性测试后升级；Framework、Runtime 与 Agent 定义版本必须进入运行快照。
- Agent 执行只有在结果提交完成且 SmartHub Schema、证据和一致性校验全部通过后，才能推动 `ReviewRun` 进入 `succeeded`。

#### FR-402：Agent、Prompt 与工具集版本

- `RequirementAnalysisAgent` 定义至少包含稳定 ID、类型 `requirement_analysis`、不可变版本、系统指令摘要、Prompt 模板版本、模型配置引用、允许工具集、MCP/网络策略、执行限制、外部操作策略、创建时间和启用状态。
- Prompt、Agent 定义、工具描述或权限策略发生变化时必须生成新版本；已创建运行继续使用原固定版本。
- 本期允许部署管理员通过代码或受控部署配置维护这些版本，但不交付通用可视化编排、任意发布和普通用户授权界面。
- 文档、网页和工具返回内容不得覆盖系统指令、Agent 定义或权限策略；发现疑似提示词注入时应记录安全事件并按策略拒绝、降级或继续只读执行。

#### FR-403：统一工具注册与执行

- 所有 Agent 工具必须先转换为统一 Tool Descriptor，至少声明：稳定名称与版本、`builtin/local/http/mcp/sandbox` 类型、输入/输出 Schema、所需权限、`read/write-reversible/write-high-risk` 风险等级、超时、幂等性、重试策略、结果大小限制和数据发送范围。
- Tool Runtime 必须在每次执行前校验工具是否属于运行快照、参数是否符合 Schema、调用身份与权限是否有效、目标是否在允许范围、是否需要审批以及取消信号是否生效。
- `RequirementAnalysisAgent` 的默认工具集至少包含固定需求版本读取、知识库检索、Chunk/原文读取、证据定位校验和候选结果提交；这些内部工具不得绕过一期资产、版本、活动索引和证据稳定标识。
- Shell、任意代码执行、宿主文件系统访问和未注册动态工具默认禁用；确有需要时必须进入隔离沙箱并单独验收。

#### FR-404：联网检索与 MCP

- 联网检索只能访问部署侧允许的协议、域名、端口和数据范围；默认阻止内网地址、云元数据地址、回环地址、文件协议、重定向越界和凭据转发，防止 SSRF 与数据外泄。
- MCP Server 必须由部署环境预注册并固定配置版本；发现的 Tool 必须转换为统一 Tool Descriptor 后才能使用，MCP 名称或返回内容不能自动获得权限。
- 发送到联网或 MCP 工具的数据必须最小化并记录数据策略；密钥由服务端注入，浏览器、模型上下文、工具参数快照和普通日志不得包含明文凭据。
- 联网或 MCP 内容必须保留来源、获取时间、工具版本和必要的响应摘要；若不能映射为项目固定证据，只能作为外部参考或待确认信息展示。

#### FR-405：外部操作与人工确认

- `read` 工具可在运行权限快照允许时自动执行；`write-reversible` 工具只有在 Agent 定义明确允许、目标受限、具备幂等键和补偿方式时才可自动执行或按策略请求确认；`write-high-risk` 工具必须逐次人工确认，不得静默执行。
- 人工确认是运行时操作门禁，不等同于本期交付完整组织审批流；确认记录至少包含运行、工具、参数摘要、风险、确认人显示信息、确认时间、有效期和最终结果。
- 模型不得拆分参数、改用其他工具或通过联网/MCP 间接调用来规避审批。审批拒绝、超时或参数变化后必须阻止执行，并将结果返回 Agent。
- 外部操作不得自动修改需求原文、正式知识资产、Finding 人工处置或活动索引；如后续确需开放，必须新增专项需求、权限与验收标准。

#### FR-406：Agent 执行可观测性与终止

- 每次 Agent 执行至少记录 `agentExecutionId`、`runId/messageId`、`agentId`、Agent 类型与定义版本、Pi Agent Framework/Runtime/Prompt/工具集/MCP/网络策略版本、开始/结束时间、回合数、工具调用数、Token 汇总、终止原因和脱敏错误摘要。
- 每次工具调用至少记录工具名称与版本、调用 ID、参数摘要、风险、权限/审批结果、开始/结束时间、耗时、结果状态、结果摘要、重试和脱敏错误；敏感参数和大结果只保存受控引用或 Hash。
- 页面执行概览应按“模型调用—工具执行—结果提交—服务端校验”展示真实步骤，可隐藏模型私有推理，但不得伪造进度。
- 用户取消、Worker 失租约、达到回合/Token/工具/超时限制或高风险操作未获批准时，必须终止后续执行；已完成过程记录保留，候选结果不得作为正式 Finding。

### 6.2 一致性规则

1. 未提供有效 `projectVersionId` 和当前版本有效 `requirementBindingId` 时不得创建 Review 或 ReviewRun；创建时项目版本必须为 `open`。
2. 评审运行一经创建，项目版本、需求绑定、输入资产版本、检索证据、模型配置、Agent、Runtime、Prompt、工具集、MCP、网络策略、权限和执行限制快照不可原地修改。
3. 同一项目版本、同一需求资产最多有一条有效绑定；替换需求版本时创建新绑定并将旧绑定标为 `superseded`，历史 Review 与 Run 继续引用旧绑定。
4. 新项目版本继承时只复制有效需求绑定，不复制 Review、Run、Finding、处置记录、问答和统计；被继承版本后续变更不联动修改新版本。
5. 知识资产新增活动版本不自动更新任何项目版本需求绑定，必须由用户在 `open` 项目版本中显式替换。
6. 所有查询、统计、导出和问答必须先校验 `projectVersionId`，不得将不同项目版本数据混合或以默认/最新版本兜底。
7. 成功运行的每个 `evidence_supported` Finding 至少有一条可打开的固定证据；证据失效时必须在查看时明确提示，而非跳转到最新资料。
8. 人工处置只能追加，不得篡改模型初始输出或历史处置记录。
9. 新需求资产版本、索引重建，或模型、Agent、Prompt、工具、MCP 与网络策略更新不修改历史运行；仅影响后续运行，并触发相应提示。
10. 运行完成前的候选输出不可作为正式 Finding 展示或参与统计。
11. 配置、模型、Agent、工具或检索异常不得被转换为“无问题”或空白成功结论。
12. 一期的资产、资产版本、Chunk、索引版本和证据稳定标识必须继续复用，不得另建不可关联的评审资料容器。
13. Pi Agent Framework 的 Agent 结束事件、普通文本输出或工具循环结束不等于评审成功；只有服务端校验并原子保存正式结果后才能写入 `succeeded`。
14. 每次工具调用必须匹配运行固定的 Tool Descriptor 与权限快照；运行中发现的新 MCP 工具、参数或目标不得自动扩大权限。
15. 写操作必须具备幂等键或明确补偿策略；需审批的操作只有在参数未变化且确认有效时才能执行，确认记录不得被 Agent 改写。
16. 联网、MCP 和外部系统内容不得改变系统 Prompt、权限、路由和终止条件；其事实属性由固定证据映射和服务端校验决定。
17. 取消、失租约或达到执行限制后产生的迟到模型/工具结果必须丢弃或仅作为过程记录保存，不得进入正式结果。
18. 视图路由、筛选、高亮和折叠状态不属于业务事实；前端不得用这些临时状态修改固定快照、Finding、Evidence 或人工处置历史。
19. 报告导出必须读取指定 `projectVersionId` 与 `runId` 的正式持久化结果并重新校验权限；不得导出其他项目版本、候选结果、明文凭据或未脱敏工具日志。

## 7. 验收标准

### AC-201：项目版本前置、需求绑定与固定版本评审

**前置条件**：项目尚未创建项目版本；知识库存在至少两份 `ready` 的需求资产版本，并准备一个含 Markdown、纯文本、图片和不支持格式的 ZIP；系统存在活动索引、可用评审模型配置和已发布启用的 `RequirementAnalysisAgent` 及其 Prompt/工具集版本。

**操作**：进入需求分析，创建 `open` 项目版本；分别从知识库加入一个历史需求版本、上传 ZIP 并等待解析完成；选择一个绑定创建评审并启动，再锁定该项目版本并尝试创建新运行。

**预期结果**：

- 创建版本前只显示“新建项目版本”引导，无法加载需求列表或启动分析；
- 加入既有需求后形成当前项目版本的有效绑定；ZIP 中支持的正文文件按一期流程入库并在 `ready` 后绑定，图片仅保留为附件，不支持格式被跳过并反馈；
- 运行保存选定的 `projectVersionId`、`requirementBindingId`、`assetVersionId`、内容 Hash、`indexVersionId`、检索范围，以及模型、`RequirementAnalysisAgent`、Pi Agent Framework/Runtime、Prompt、工具集、MCP、网络与权限策略版本；
- 之后上传该需求的新版本，不改变历史运行打开的原文和证据；
- 页面不允许用未入库文本、无有效绑定或无 `ready` 版本启动正式评审；项目版本锁定后不可再新增绑定或运行，但历史数据可查看。

### AC-202：结构化 Finding 与证据定位

**前置条件**：需求文档中存在可识别的业务规则、缺失项和关联资料。

**操作**：完成一次评审，打开评审概览和证据引用，点击一个有依据 Finding。

**预期结果**：

- Finding 展示分类、严重度、依据类型、问题、影响和建议确认项；
- 有依据 Finding 至少关联一条固定资产版本、标题路径和原文位置；
- 点击后按 `assetVersionId` 打开原文并定位命中范围；
- 无依据结果显式标记为待确认或模型推测，不伪装为已证实事实。

### AC-203：人工处置不覆盖历史结果

**前置条件**：存在成功运行和至少一个 `open` Finding。

**操作**：依次将 Finding 标记为确认、驳回或已解决，并填写处置说明；随后重新运行评审。

**预期结果**：

- 每次处置都有独立记录、时间和说明；
- 原始 AI Finding 与证据不被改写；
- 新运行具有新的运行 ID 和独立结果，旧运行及人工处置仍可查看。

### AC-204：取消、失败与检索降级状态

**前置条件**：可触发运行中的取消、模型调用失败和混合检索向关键词检索降级。

**操作**：分别取消运行、模拟模型不可用，并执行发生检索降级的评审。

**预期结果**：

- 取消运行会向模型与工具传播取消信号并标记为 `cancelled`，迟到结果不得产生正式 Finding；
- 模型不可用时标记为 `failed` 并显示脱敏错误摘要，不展示本地模拟结果；
- 检索降级时，运行与相关证据清楚展示降级原因；无活动索引时不允许仅依赖模型启动评审。

### AC-205：模型配置版本与受控路由

**前置条件**：部署环境已注册至少两个支持结构化输出的模型，其中一个可作为回退模型。

**操作**：保存新的需求评审模型配置，启动运行；再使主模型不可用并启动另一运行。

**预期结果**：

- 保存产生新的模型配置版本，已存在运行不受影响；
- 智能路由关闭时使用默认模型；开启且主模型失败时，只有在允许降级、回退模型满足能力时才切换；
- 页面和调用记录显示实际模型、重试次数和降级原因；
- 不返回或保存 API Key、完整端点和 Authorization 请求头。

### AC-206：模型检测与输出校验

**前置条件**：存在一个可用模型和一个返回不符合结构化 Schema 的测试模型响应。

**操作**：对已注册模型执行连通性检测，并使用异常响应模型启动评审。

**预期结果**：

- 检测展示可用性、结构化输出能力、检测时间和脱敏错误；
- 异常响应导致运行显示结果校验失败，不生成成功统计和可处置 Finding；
- 页面不提供任意端点和明文密钥的输入或回显。

### AC-207：RequirementAnalysisAgent 工具循环与版本快照

**前置条件**：一份需求需要检索至少两份关联资料才能完成评审，已启用的 `RequirementAnalysisAgent` 配置了知识库检索、原文读取、引用校验和候选结果提交工具。

**操作**：启动评审，观察 Agent 多轮调用工具并提交候选结构化结果；运行期间发布新的 Prompt 或工具集版本。

**预期结果**：

- 每个 `ReviewRun` 只启动一个 `RequirementAnalysisAgent` 执行实例，且只能调用运行快照中允许的工具；所有调用均经过 Schema、权限、超时和结果限制校验；
- 运行保存 `agentId`、Agent 类型与定义版本，以及 Pi Agent Framework/Runtime、Prompt、工具集、MCP、网络策略、权限和执行限制版本；新发布版本不影响本次运行；
- 页面可查看真实工具执行步骤、状态和脱敏摘要，工具结果中的指令不能扩大 Agent 权限；
- Agent 结束事件或候选结果不直接代表成功，只有服务端 Schema 与证据校验通过后才保存正式 Finding。

### AC-208：联网、MCP 与外部操作安全边界

**前置条件**：存在允许的联网/MCP 只读工具、一个越界目标、一个可逆写工具、一个高风险写工具及包含提示词注入内容的模拟响应。

**操作**：让 Agent 分别调用上述工具，并测试无确认、确认后参数变化、审批拒绝和取消运行。

**预期结果**：

- 允许范围内的只读调用可自动执行，越界网络、未注册 MCP 工具和参数 Schema 不合法的调用被拒绝；
- 提示词注入内容作为不可信数据处理，不能修改系统 Prompt、工具集、路由或审批策略；
- 可逆写操作遵循配置的自动执行或确认策略，高风险写操作未逐次确认不得执行；参数变化后原确认失效；
- 取消、拒绝或超时会终止后续调用，调用、风险、审批、结果与脱敏错误均有记录，敏感凭据不进入页面或普通日志。

### AC-209：三栏评审工作台与双向证据联动

**前置条件**：两个项目版本各自绑定需求并存在成功评审运行；同一需求存在两个固定资产版本、至少一个有证据 Finding、一个无证据 Finding 和一段可引用原文。

**操作**：切换项目版本与历史运行，按严重度过滤 Finding，依次点击有证据 Finding、原文高亮、功能树节点和证据引用；引用该原文发起追问，并导出 Markdown 评审报告。

**预期结果**：

- 左侧列表、中间五个视图和右侧问答始终绑定所选 `projectVersionId` 与 `runId`，刷新页面后仍恢复同一作用域；切换版本时不混入上一版本或最新版本数据；
- 有证据 Finding 打开指定 `assetVersionId` 并滚动高亮，原文高亮和功能树节点能反查正式关联 Finding；无证据 Finding 不生成伪高亮；
- 引用追问保存固定原文位置，回答展示引用、实际模型和降级状态，且不能修改原文或 Finding 状态；
- 运行阶段来自服务端，断线恢复后与服务端一致；
- 导出报告包含固定输入、正式 Finding、证据、降级和处置状态，不包含候选输出、问答全文、明文凭据或未脱敏日志；
- 页面不存在 AI 改写、自动应用、测试用例生成、PDF/Word/Excel/HTML 等专用格式解析和 Jira/PingCode 推送入口；ZIP 仅作为 Markdown/纯文本及附件的批量上传封装。

## 8. 交付边界与后续衔接

本期交付的是基于 Pi Agent Framework 创建并启用首个业务 Agent——`RequirementAnalysisAgent`，由其驱动以知识库证据为基础的需求评审闭环；同时交付该 Agent 所需的生成式模型配置和工具治理。本期不是通用 Agent 创建平台、多 Agent 协作平台、需求文档编辑或工程执行平台。

- AI 评审结果和问答仅为派生产物；只有人工将其编辑并保存为独立资料版本后，才可按一期统一资产流程进入知识库。
- Pi Agent Framework 是 Phase 2～Phase 4 的统一 Agent 框架；本期交付 `RequirementAnalysisAgent` 及其运行、版本、权限、安全与观测边界，但不交付普通用户任意创建 Agent、注册工具、配置端点或发布编排的通用平台。
- Phase 3 应在同一 Pi Agent Framework 上创建独立的 `TechnicalSolutionAnalysisAgent`，复用本期已确认的 Finding、人工处置、固定证据和 Tool Runtime，但使用独立 Agent 定义、Prompt、工具集、方案运行与输入快照。
- Phase 4 应在同一框架上创建测试需求分析、测试用例设计、测试脚本编写和测试执行等多个业务 Agent，并由显式多 Agent 工作流管理任务分解、产物交接、审批与汇总；第二期不提前实现该多 Agent 能力。
- Git、测试、组织级权限、完整审批、成本治理及通用 Prompt/Agent 管理界面应在后续阶段分别定义；底层已有相关 Tool/MCP 适配能力不代表这些产品能力已经交付。

## 9. 前置条件与待确认事项

### 9.1 前置条件

- 一期知识库可稳定提供需求资产版本、Chunk、活动索引和固定证据定位；
- 项目版本服务可创建、查询、锁定、归档和受控删除版本，并维护项目版本需求绑定；至少可创建一个 `open` 项目版本；
- 至少一组可用于试点评审的 Markdown/纯文本需求资料及关联业务规则资料；
- 部署环境已注册至少一个支持结构化输出的受控生成式模型；
- 服务端 Node.js 版本满足所锁定 Pi Agent Framework 版本要求，并已完成 `PiAgentRuntimeAdapter` 集成与兼容性测试；
- 已创建、发布并启用唯一的 `RequirementAnalysisAgent`，以及它的 Prompt、默认只读工具集、执行限制和权限策略版本；
- 如启用联网或 MCP，已完成 Server 注册、网络白名单、凭据托管、数据边界和提示词注入防护评审；
- 已定义外部操作风险等级、自动执行边界、人工确认门禁、幂等与补偿要求；
- 已确认本期允许发送至模型服务、联网与 MCP 工具的项目资料范围和保留策略；
- 已准备典型需求、预期问题、证据来源、工具调用、安全攻击样例和人工处置样例作为评审质量验收集。

### 9.2 待确认事项

1. Finding 的严重度定义、默认处置时限及是否需要后续对接工单系统；
2. 单次需求评审支持的最大输入长度，以及超长需求的后续分段评审策略；
3. 首批可注册的生成模型来源、模型清单、上下文窗口、数据传输边界和保留策略；
4. 关键词降级时是否允许继续生成评审，以及对结论可信度的展示规则；
5. `RequirementAnalysisAgent`、Prompt、工具集和安全策略的维护责任、发布流程、版本审批与回滚机制；
6. 最大 Agent 回合、工具调用、Token、单工具结果、总超时和并发配额；
7. 首批联网域名、MCP Server、允许发送的数据范围及外部内容引用规则；
8. `write-reversible` 自动执行边界、`write-high-risk` 逐次确认方式、确认人范围和补偿责任；
9. 试点质量验收集、Finding 召回率、证据准确率、工具成功率和安全拒绝准确率等度量口径，以及人工复核责任人；
10. 后续何时交付成员权限、组织级审批、成本配额、通用 Prompt/Agent 可视化编排和多模态资料解析。
