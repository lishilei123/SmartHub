## 2. 背景与目标

一期已建立项目资料的知识资产、不可变资产版本、Chunk、活动索引和固定证据定位能力。当前原型已展示需求列表、分析运行进度、分析概览、原始文档、版本差异、功能树、证据引用、需求问答，以及“AI 模型与路由”配置入口；但上述需求 AI 与模型路由仍使用本地示例数据，不能形成可追溯的真实评审闭环。

本期目标是在一期知识库基础上，将“需求分析”升级为可执行、可复核、可人工处置的**需求评审**能力，并提供支撑该能力的受控大模型配置、路由与 Agent 执行能力。评审可在服务端依赖版本化 Prompt、受控工具、联网检索和 MCP 适配器完成多步证据检索，但正式运行状态、输入快照、权限边界、结果校验与人工处置仍由 SmartHub 确定性业务流程控制。

### 2.1 本期目标

1. 从确定的需求知识资产版本创建评审，固定输入资料、检索范围、索引版本、模型配置和提示词版本。
2. 通过大模型生成结构化的需求评审结果，识别需求缺口、歧义、冲突、边界、验收风险和测试风险。
3. 每条结论均能打开至固定资料版本的证据位置；缺少证据的内容必须明确标识为待确认或模型推测。
4. 支持人工确认、驳回、解决和补充评论，不覆盖原始 AI 输出，并保留历史评审运行。
5. 支持受控配置生成式大模型、模型参数、路由与降级策略；凭据不进入浏览器和业务数据库明文字段。
6. 真实模型不可用、检索降级、输出结构校验失败或任务取消时，页面必须显示真实状态，不得回退为示例结果或伪造成功。
7. 使用受控 Agent Runtime 编排需求评审内部的检索、证据读取、引用校验与结构化结果提交，并固定 Agent、Prompt、工具集、MCP 与网络策略版本。
8. 支持按工具风险和运行策略受控调用联网、MCP 及外部操作；未授权、高风险或需要人工确认的操作不得由模型绕过策略直接执行。

### 2.2 本期非目标

本期不实现以下能力，且验收不得依赖它们：

- 技术方案生成、架构设计评审、代码影响分析、Git/PR/Commit 集成；
- 测试用例或测试脚本生成、测试执行、CI/CD、日志诊断和自愈；
- 图片、原型图、HTML、PDF、Word、Excel 等专用格式的解析与多模态评审；
- 面向用户的通用可视化 Prompt/Agent 编排平台、任意 Agent 创建与发布、任意工具或 MCP 服务注册；
- 允许用户在浏览器中配置任意联网端点、密钥、Shell、代码执行环境或不受控外部操作；
- 完整的组织、角色、ACL、审批流、SSO、合规审计和成本结算体系；
- 将 AI 评审结论、聊天内容自动写回或自动发布为知识库正式资料。

上述非目标限定的是本期**不交付通用平台产品能力**，不禁止需求评审在服务端依赖 Pi Agent Core 或兼容 Agent Runtime、受控 Prompt、内部工具、联网检索、MCP 适配器和外部操作执行器。此类依赖必须由部署环境预注册、版本化、最小授权、可取消、可审计，并受本文件 5.3 节约束。

## 3. 范围、术语与原则

### 3.1 核心对象

```text
项目 / 默认知识库
  └─ 需求知识资产
       └─ 不可变资产版本

需求评审
  └─ 评审运行（固定输入与配置快照）
       ├─ 评审结论 / Finding
       │    ├─ 证据快照
       │    └─ 人工处置记录
       └─ 评审问答会话

大模型配置
  └─ 已注册模型来源
       └─ 模型定义
            └─ 评审模型配置版本 / 路由策略

受控评审 Agent
  └─ Agent 定义版本
       ├─ Prompt 模板版本
       ├─ 工具集与权限策略版本
       ├─ MCP / 网络策略版本
       └─ 执行限制与外部操作策略
```

| 对象 | 定义与规则 |
|---|---|
| 需求评审 | 对一份需求资产固定版本发起的评审逻辑实体。它不是知识资产，也不替代原需求文档。 |
| 评审运行（`ReviewRun`） | 一次可取消、可追溯的模型执行。每次重新评审都创建新运行，历史运行不可覆盖。 |
| Finding | 可人工处置的结构化评审结论，例如缺口、歧义、冲突、边界或风险。 |
| 证据快照 | Finding 所引用的固定资产版本、Chunk、内容 Hash、标题路径、原文定位和引用摘录。 |
| 大模型来源 | 由部署环境受控注册的生成式模型服务，不等同于一期 Embedding 来源。 |
| 评审模型配置 | 供需求评审使用的模型、参数、路由及降级策略的不可变版本。 |
| 评审 Agent | 在一次评审运行内部执行多轮推理、检索、证据读取、引用校验和结果提交的受控 Agent；首期服务端运行时采用 Pi Agent Core，并通过 SmartHub 内部接口隔离具体框架。 |
| Agent 定义版本 | Agent 的系统指令、Prompt 引用、允许工具、执行限制和安全策略组成的不可变版本。 |
| 受控工具 | 由服务端注册并具有输入/输出 Schema、权限、风险等级、超时、幂等性与数据策略的确定性能力。 |
| MCP 适配器 | 将受信 MCP Server 暴露的能力转换为 SmartHub 统一工具描述的接入层；MCP 本身不绕过工具权限、参数校验和审计。 |
| Agent 执行 | 绑定一个 `ReviewRun` 的内部多轮执行实例；其消息与工具事件是过程记录，不替代 `ReviewRun` 业务状态。 |

### 3.2 评审原则

- **固定输入**：每次运行只能使用创建时绑定的需求资产版本、检索范围、活动索引版本和配置版本；不得在运行中悄然改用“当前最新资料”。
- **证据优先**：涉及项目事实的 Finding 必须关联证据；没有足够依据时，结论只能标识为“待确认”或“模型推测”。
- **人机分工**：AI 提供结构化问题、风险和建议，是否采纳由人工决定；系统不得自动修改需求内容或关闭问题。
- **可复现边界**：模型输出具有非确定性，但同一运行必须可还原输入、模型/参数、提示词、检索证据和最终输出。
- **配置分离**：生成式大模型配置与一期 Embedding、切分、索引配置分开管理；调整生成模型不得触发知识库索引重建。
- **确定性外层、Agent 内层**：排队、租约、取消、路由、重试、降级、Schema 校验、正式落库和人工处置由 SmartHub 控制；Agent 只在单次受控执行内部决定如何调用已授权工具。
- **最小权限**：工具、MCP、网络和外部操作默认关闭，按 Agent 定义版本显式开放；只读能力与写操作分级管理，高风险操作必须经过人工确认。
- **不信任工具内容**：知识库、网页、MCP 与外部系统返回内容均视为不可信数据，不得作为系统指令执行；服务端必须防止提示词注入改变工具权限或运行策略。

## 4. 用户界面与主要流程

### 4.1 需求评审页面

沿用现有三栏原型布局，并替换所有本地 fixture：

- **左侧需求评审列表**：按项目发布版本或关键字筛选；展示需求名称、需求资产版本、最新评审状态、最新运行时间和未处理 Finding 数量；支持“新建需求评审”。
- **中间评审区**：展示需求基本信息、固定输入快照、运行/取消状态及进度；提供“评审概览、原始文档、版本差异、功能树、证据引用”五个视图。
- **右侧评审问答区**：仅围绕当前评审运行的输入与结论提问；每条回答显示其状态和引用证据。无真实模型或运行未完成时，不展示模拟回答。
- **操作记录**：展示当前评审的创建、运行、取消、人工处置、问答及错误事件；本期不将其定义为组织级合规审计。

### 4.2 创建并运行评审

```text
用户选择需求知识资产
  → 选择 ready 的需求资产版本（默认当前活动版本）
  → 选择或确认逻辑路径检索范围
  → 系统校验活动索引、模型配置、Agent 定义、工具权限和输入长度
  → 创建需求评审与 queued 的评审运行
  → 固定输入、索引、检索、模型、Agent、Prompt、工具集、MCP 与网络策略快照
  → 评审 Agent 调用已授权工具检索关联资料、读取原文并校验证据
  → Agent 在回合、Token、工具调用和超时限制内提交候选结构化结果
  → 校验结构化输出与证据完整性
  → 保存结果、Finding、证据和用量记录
  → 运行 succeeded，等待人工处置
```

若运行失败、取消或校验不通过，保留运行记录、失败步骤和错误摘要；不得生成“分析完成”的示例结果。

### 4.3 Agent 内部执行流程

```text
Review Worker 取得运行租约
  → 装载固定 ReviewRun、Agent、Prompt、模型、工具与权限快照
  → Pi Agent Core 启动受控 Agent 执行
  → Agent 按需调用知识库检索、证据读取、引用校验、联网或 MCP 工具
  → Tool Runtime 校验参数、身份、权限、风险、超时、网络策略和审批条件
  → 工具结果脱敏、限长并记录后返回 Agent
  → Agent 调用 submit_review_result 提交候选结果
  → SmartHub 独立执行 Schema、证据与一致性校验
  → 校验通过后原子保存正式 Finding；否则运行失败并保留过程记录
```

- `ReviewRun` 是业务事实来源；Pi Agent Core 的内存状态、消息历史或结束事件不得直接代表运行成功。
- Agent 不得直接写入 Finding、处置记录、知识资产或活动索引；`submit_review_result` 仅提交候选结果，由 SmartHub 校验并落库。
- Worker 失去租约、用户取消、达到执行限制或服务端关闭时，必须向模型请求和正在执行的工具传播取消信号。

### 4.4 人工评审闭环

```text
查看 Finding 与固定证据
  → 确认 / 驳回 / 标记已解决 / 保持待确认
  → 可填写处置说明与补充评论
  → 需求资料更新后创建新的资产版本
  → 人工发起新的评审运行
  → 旧运行保留，并标记为可能过期
```

## 5. 功能需求

### 5.1 需求评审

#### FR-201：从知识资产创建需求评审

- 新建评审时，用户必须从当前项目默认知识库中选择资料类型为 `requirement` 的知识资产及其 `ready` 资产版本。
- 默认选择该资产的当前活动版本；允许选择历史 `ready` 版本以复核历史结论。
- 页面至少展示需求标题、`assetId`、`assetVersionId`、资料逻辑路径、版本号、入库时间和当前活动索引版本。
- 需求尚未入库、无 `ready` 版本、无活动索引或模型配置不可用时，不允许启动运行，并显示可理解的原因和下一步。
- 本期不提供绕过知识库、直接粘贴任意文本后即发起正式评审的入口。

#### FR-202：评审运行快照与状态

- 创建运行时必须固定并保存：`projectId`、`knowledgeBaseId`、需求 `assetId`、`assetVersionId`、输入内容 Hash、`indexVersionId`、逻辑路径检索范围、检索参数与检索结果快照、评审模型配置版本、实际路由结果、Agent 定义版本、Agent Runtime 版本、提示词模板版本、工具集版本、MCP 配置版本、网络策略版本、权限快照、执行限制和触发时间。
- 运行状态至少包括 `queued`、`running`、`succeeded`、`failed`、`cancelled`；状态与进度必须由服务端任务真实返回。
- 用户可取消未完成运行；取消后停止后续模型调用和工具调用，不写入未完成的结构化结论，已保存的 Agent 回合、工具调用、步骤日志和错误信息允许查看。
- 同一需求可存在多个历史运行；重新评审必须创建新 `ReviewRun`，不得覆盖历史输入、结论、证据或人工处置。

#### FR-203：评审维度与结构化输出

每次成功运行至少生成以下结构化结果：

| 输出 | 要求 |
|---|---|
| 需求摘要 | 基于输入版本生成，明确标识为 AI 摘要。 |
| 功能树 | 以需求章节或功能域组织功能、业务规则、异常场景和验收要点。 |
| Finding | 按缺口、歧义、冲突、边界、验收风险、测试风险分类，包含问题描述、影响、建议确认项、严重度和结论依据。 |
| 评审统计 | 展示功能项、Finding、待确认项、风险项及有证据项数量；统计必须由本次持久化结果计算。 |
| 版本差异 | 当用户为同一资产选择两个固定版本时，展示真实的版本内容差异和与差异相关的 Finding；无可比较版本时明确提示。 |

- Finding 严重度使用 `blocker`、`high`、`medium`、`low` 四档；页面可映射为阻断、高、中、低风险标签。
- Finding 依据类型使用 `evidence_supported`、`needs_confirmation`、`model_inference` 三种。`evidence_supported` 必须至少有一条有效证据；其余两类必须明确说明未证实原因。
- 服务端必须校验模型输出符合已定义的结构化 Schema。输出不合法、必填字段缺失或证据引用无法解析时，运行标为失败或进入明确的“结果校验失败”步骤，不得将不完整文本伪装为成功结果。
- Agent 只能通过受控结果提交工具产生候选结果；模型正文、部分流式内容或普通工具返回不得直接成为正式 Finding。

#### FR-204：固定证据与原文定位

- 每条 Finding 可关联零到多条证据；证据快照至少包含 `assetId`、`assetVersionId`、`chunkId` 或 `chunkKey`、Chunk 内容 Hash、标题路径、行号/字符范围、引用摘录、`indexVersionId` 和检索得分/降级状态。
- 评审来源文档本身也必须以固定 `assetVersionId` 打开，不得因该资料后续更新而跳转到最新版本。
- “证据引用”视图按 Finding 展示资料名称、逻辑路径、标题路径和原文定位；点击后打开对应固定版本并高亮命中范围。
- 模型不得将无法映射至固定资料版本的内容表述为项目事实；无证据 Finding 必须使用“待确认”或“模型推测”标签。
- Agent 通过联网或 MCP 获得的内容不能自动成为项目事实；只有能够映射至本次允许证据范围并通过引用校验的内容才能标记为 `evidence_supported`，其他内容必须标记为外部参考、待确认或模型推测。

#### FR-205：Finding 人工处置

- Finding 初始状态为 `open`，人工可更新为 `confirmed`、`dismissed`、`resolved` 或 `needs_follow_up`。
- 人工处置必须追加保存状态、处置说明、操作者显示信息和时间；不得改写模型最初生成的标题、描述、严重度、依据类型及证据快照。
- `dismissed`、`resolved` 和 `confirmed` 均应要求填写或确认处置说明；允许为空的临时保存仅使用 `open` 或 `needs_follow_up`。
- 页面支持按严重度、依据类型和处置状态过滤 Finding，并在需求列表与概览中展示未处理数量。
- 本期不实现多级审批、指派、催办、消息通知或跨项目工单同步。

#### FR-206：评审问答

- 问答必须绑定一个已成功完成的评审运行，输入范围仅包括该运行的需求资产版本、固定检索证据、结构化评审结果、会话历史及问答 Agent 定义允许访问的只读工具。
- 每轮回答保存问题、回答、实际模型配置版本、Agent/Prompt/工具集版本、执行时间、用量摘要、工具调用摘要和引用证据；回答中的项目事实遵循 FR-204 的证据规则。
- 当模型不可用、请求超时、证据不足或输出校验失败时，显示明确失败原因；不得使用本地模拟答案代替。
- 问答会话是 AI 派生产物，不自动进入知识库；后续如需沉淀，由人工另行保存为独立资料版本。

#### FR-207：过期标记与历史查看

- 当评审输入资产产生新的活动版本，或该评审所用索引版本不再是当前活动版本时，历史评审显示“可能过期”，但仍可查看其完整快照。
- 过期提示不得自动改变 Finding 状态，也不得自动关闭或重新运行评审。
- 用户可在同一需求资产的固定版本之间发起比较；比较对象、差异结果和所选运行必须清楚区分。

#### FR-208：输入长度与检索降级

- 启动前，系统应根据实际路由模型的上下文窗口、输出预留和固定需求内容评估输入是否可执行。
- 超出本期支持上限时，系统必须拒绝启动并说明需求版本过长及可选处理方式；不得静默截断需求原文后仍标记为“完整评审”。
- 检索发生关键词降级或 Reranker 绕过时，可按已保存策略继续运行，但运行、Finding 证据和页面必须显示降级状态及原因。
- 无活动索引、检索完全不可用或需求输入版本不可读取时，运行必须失败，不得仅依赖模型记忆生成结论。
- 每个 Agent 定义必须设置最大回合数、最大工具调用数、单工具结果长度、总 Token 预算和总执行超时；达到限制时运行必须停止并显示真实原因，不得无限循环或以部分结果标记成功。

### 5.2 大模型配置与路由

#### FR-301：生成式模型来源管理

- 系统将一期的 Embedding/检索配置与本期生成式大模型配置分为独立模块。知识库索引继续使用一期配置；修改本期模型配置不触发索引重建。
- 首期生成式调用仅支持部署环境注册的 OpenAI-compatible Chat Completions 协议或等价的受控文本生成适配器；接入其他协议须新增适配器并单独验收。
- 模型来源至少包含稳定 ID、显示名称、协议类型、启用状态、部署侧端点引用、凭据引用、可用模型列表和健康状态。
- 浏览器、需求评审 API 响应、运行快照和业务数据库不得保存或返回 Base URL、API Key、Token 等明文凭据。部署环境负责安全注入端点和密钥。
- 本期不提供用户在浏览器中新增任意端点、录入密钥或探测任意第三方服务的能力。

#### FR-302：模型定义与能力声明

每个可选生成模型至少维护：

- `modelId`、显示名称、所属模型来源、启用状态；
- 最大上下文长度、最大输出 Token、支持的语言或文本能力声明；
- 是否支持结构化 JSON 输出；本期需求评审仅允许选择支持结构化输出的模型；
- 默认超时、最大重试次数和健康状态；
- 最近一次连通性检测时间、检测结果与脱敏错误摘要。

模型不可用、被停用、能力不满足或超出输入上限时，不得成为新运行的路由目标；历史运行仍保存原实际模型标识与配置快照。

#### FR-303：评审模型配置版本

- “AI 模型与路由”页面提供需求评审模型配置草稿、未保存提示、保存和版本历史查看能力。
- 配置至少包含：默认模型、温度、最大输出 Token、请求超时、重试次数、结构化输出模式、智能路由开关、降级开关及候选回退模型顺序。
- 保存后生成新的不可变配置版本；无变化不生成冗余版本。
- 配置保存仅影响后续创建的评审运行；已创建或正在执行的运行继续使用其固定配置快照。
- 系统内置的需求评审提示词以受控模板版本标识参与快照。本期配置页只展示 Agent、Prompt、工具集、MCP 与网络策略的版本和摘要，不提供面向普通用户的通用编辑、发布或任意工具授权能力；其版本由部署侧受控配置发布。

#### FR-304：模型路由、重试与降级

- 智能路由关闭时，所有新运行使用当前默认模型；开启时，路由必须根据已保存的模型能力、上下文长度、启用状态和健康状态选择模型，并记录实际选择理由。
- 主模型发生可重试错误时，先按配置重试；重试耗尽后仅当“允许降级”已开启且候选模型满足输入与结构化输出能力时才切换回退模型。
- 发生模型降级时，运行页面、用量记录和最终结果必须展示实际模型与降级原因；不得将回退模型结果标识为主模型结果。
- 无可用模型时，运行进入 `failed` 并保留错误摘要；不得回退为固定示例、缓存答案或未验证旧结果。
- 模型路由、重试与降级由 SmartHub 在 Agent 执行边界外控制；Agent 不得自行修改模型、密钥或回退顺序。发生模型切换时应创建可追踪的调用阶段，并继续遵守同一运行的固定输入、Agent、Prompt、工具与权限快照。

#### FR-305：模型检测与安全反馈

- 配置页支持对已注册模型执行受控连通性与结构化输出检测，检测请求使用部署侧凭据，且不使用项目需求资料作为测试内容。
- 检测结果至少返回可用/不可用、模型能力、检测时间和脱敏错误摘要；不得回显请求头、密钥、完整端点或服务端原始堆栈。
- 前端保存或检测期间显示真实加载、成功和失败状态；保存失败不得更新页面“已保存”版本。

#### FR-306：运行可观测性

- 每次模型调用至少记录运行/会话 ID、Agent 执行 ID、配置版本、实际模型、路由与降级结果、开始/结束时间、耗时、输入/输出 Token（服务提供时）、重试次数、结果状态和脱敏错误摘要。
- 页面提供评审运行级别的执行概览；本期不提供组织级账单、价格计算、配额分配或审计导出。
- 运行日志不得记录完整 API Key、Authorization 请求头或未经脱敏的敏感配置值。

### 5.3 受控 Agent、工具、联网与 MCP

#### FR-401：受控 Agent Runtime

- 首期服务端评审运行采用 Pi Agent Core 作为受控 Agent Runtime，并通过 SmartHub `ReviewAgentRuntime` 接口适配，业务领域层不得直接依赖框架消息类型或以内存会话作为事实来源。
- Pi Agent Core 负责单次运行内部的模型交互、工具循环、流式事件和取消传播；PostgreSQL Job/Worker 继续负责排队、租约、恢复、重试、取消与最终状态。
- 依赖必须锁定明确版本并经过兼容性测试后升级；Agent Runtime 版本必须进入运行快照。
- Agent 执行只有在结果提交完成且 SmartHub Schema、证据和一致性校验全部通过后，才能推动 `ReviewRun` 进入 `succeeded`。

#### FR-402：Agent、Prompt 与工具集版本

- 每个评审 Agent 定义至少包含稳定 ID、不可变版本、系统指令摘要、Prompt 模板版本、模型配置引用、允许工具集、MCP/网络策略、执行限制、外部操作策略、创建时间和启用状态。
- Prompt、Agent 定义、工具描述或权限策略发生变化时必须生成新版本；已创建运行继续使用原固定版本。
- 本期允许部署管理员通过代码或受控部署配置维护这些版本，但不交付通用可视化编排、任意发布和普通用户授权界面。
- 文档、网页和工具返回内容不得覆盖系统指令、Agent 定义或权限策略；发现疑似提示词注入时应记录安全事件并按策略拒绝、降级或继续只读执行。

#### FR-403：统一工具注册与执行

- 所有 Agent 工具必须先转换为统一 Tool Descriptor，至少声明：稳定名称与版本、`builtin/local/http/mcp/sandbox` 类型、输入/输出 Schema、所需权限、`read/write-reversible/write-high-risk` 风险等级、超时、幂等性、重试策略、结果大小限制和数据发送范围。
- Tool Runtime 必须在每次执行前校验工具是否属于运行快照、参数是否符合 Schema、调用身份与权限是否有效、目标是否在允许范围、是否需要审批以及取消信号是否生效。
- 首期评审 Agent 的默认工具集至少包含固定需求版本读取、知识库检索、Chunk/原文读取、证据定位校验和候选结果提交；这些内部工具不得绕过一期资产、版本、活动索引和证据稳定标识。
- Shell、任意代码执行、宿主文件系统访问和未注册动态工具默认禁用；确有需要时必须进入隔离沙箱并单独验收。

#### FR-404：联网检索与 MCP

- 联网检索只能访问部署侧允许的协议、域名、端口和数据范围；默认阻止内网地址、云元数据地址、回环地址、文件协议、重定向越界和凭据转发，防止 SSRF 与数据外泄。
- MCP Server 必须由部署环境预注册并固定配置版本；发现的 Tool 必须转换为统一 Tool Descriptor 后才能使用，MCP 名称或返回内容不能自动获得权限。
- 发送到联网或 MCP 工具的数据必须最小化并记录数据策略；密钥由服务端注入，浏览器、模型上下文、工具参数快照和普通日志不得包含明文凭据。
- 联网或 MCP 内容必须保留来源、获取时间、工具版本和必要的响应摘要；若不能映射为项目固定证据，只能作为外部参考或待确认信息展示。

#### FR-405：外部操作与人工确认

- `read` 工具可在运行权限快照允许时自动执行；`write-reversible` 工具只有在 Agent 定义明确允许、目标受限、具备幂等键和补偿方式时才可自动执行或按策略请求确认；`write-high-risk` 工具必须逐次人工确认，不得静默执行。
- 人工确认是运行时操作门禁，不等同于本期交付完整组织审批流；确认记录至少包含运行、工具、参数摘要、风险、确认人显示信息、确认时间、有效期和最终结果。
- 模型不得拆分参数、改用其他工具或通过联网/MCP 间接调用来规避审批。审批拒绝、超时或参数变化后必须阻止执行，并将结果返回 Agent。
- 外部操作不得自动修改需求原文、正式知识资产、Finding 人工处置或活动索引；如后续确需开放，必须新增专项需求、权限与验收标准。

#### FR-406：Agent 执行可观测性与终止

- 每次 Agent 执行至少记录 `agentExecutionId`、`runId/messageId`、Agent/Runtime/Prompt/工具集/MCP/网络策略版本、开始/结束时间、回合数、工具调用数、Token 汇总、终止原因和脱敏错误摘要。
- 每次工具调用至少记录工具名称与版本、调用 ID、参数摘要、风险、权限/审批结果、开始/结束时间、耗时、结果状态、结果摘要、重试和脱敏错误；敏感参数和大结果只保存受控引用或 Hash。
- 页面执行概览应按“模型调用—工具执行—结果提交—服务端校验”展示真实步骤，可隐藏模型私有推理，但不得伪造进度。
- 用户取消、Worker 失租约、达到回合/Token/工具/超时限制或高风险操作未获批准时，必须终止后续执行；已完成过程记录保留，候选结果不得作为正式 Finding。

### 6.2 一致性规则

1. 评审运行一经创建，输入资产版本、检索证据、模型配置、Agent、Runtime、Prompt、工具集、MCP、网络策略、权限和执行限制快照不可原地修改。
2. 成功运行的每个 `evidence_supported` Finding 至少有一条可打开的固定证据；证据失效时必须在查看时明确提示，而非跳转到最新资料。
3. 人工处置只能追加，不得篡改模型初始输出或历史处置记录。
4. 新需求资产版本、索引重建，或模型、Agent、Prompt、工具、MCP 与网络策略更新不修改历史运行；仅影响后续运行，并触发相应提示。
5. 运行完成前的候选输出不可作为正式 Finding 展示或参与统计。
6. 配置、模型、Agent、工具或检索异常不得被转换为“无问题”或空白成功结论。
7. 一期的资产、资产版本、Chunk、索引版本和证据稳定标识必须继续复用，不得另建不可关联的评审资料容器。
8. Pi Agent Core 的 `agent_end`、普通文本输出或工具循环结束不等于评审成功；只有服务端校验并原子保存正式结果后才能写入 `succeeded`。
9. 每次工具调用必须匹配运行固定的 Tool Descriptor 与权限快照；运行中发现的新 MCP 工具、参数或目标不得自动扩大权限。
10. 写操作必须具备幂等键或明确补偿策略；需审批的操作只有在参数未变化且确认有效时才能执行，确认记录不得被 Agent 改写。
11. 联网、MCP 和外部系统内容不得改变系统 Prompt、权限、路由和终止条件；其事实属性由固定证据映射和服务端校验决定。
12. 取消、失租约或达到执行限制后产生的迟到模型/工具结果必须丢弃或仅作为过程记录保存，不得进入正式结果。

## 7. 验收标准

### AC-201：基于固定需求版本创建评审

**前置条件**：知识库存在至少两份 `ready` 的需求资产版本，且存在活动索引、可用评审模型配置和已发布的评审 Agent/Prompt/工具集版本。

**操作**：选择其中一个历史需求资产版本创建评审并启动。

**预期结果**：

- 运行保存选定的 `assetVersionId`、内容 Hash、`indexVersionId`、检索范围，以及模型、Agent、Runtime、Prompt、工具集、MCP、网络与权限策略版本；
- 之后上传该需求的新版本，不改变历史运行打开的原文和证据；
- 页面不允许用未入库文本或无 `ready` 版本启动正式评审。

### AC-202：结构化 Finding 与证据定位

**前置条件**：需求文档中存在可识别的业务规则、缺失项和关联资料。

**操作**：完成一次评审，打开评审概览和证据引用，点击一个有依据 Finding。

**预期结果**：

- Finding 展示分类、严重度、依据类型、问题、影响和建议确认项；
- 有依据 Finding 至少关联一条固定资产版本、标题路径和原文位置；
- 点击后按 `assetVersionId` 打开原文并定位命中范围；
- 无依据结果显式标记为待确认或模型推测，不伪装为已证实事实。

### AC-203：人工处置不覆盖历史结果

**前置条件**：存在成功运行和至少一个 `open` Finding。

**操作**：依次将 Finding 标记为确认、驳回或已解决，并填写处置说明；随后重新运行评审。

**预期结果**：

- 每次处置都有独立记录、时间和说明；
- 原始 AI Finding 与证据不被改写；
- 新运行具有新的运行 ID 和独立结果，旧运行及人工处置仍可查看。

### AC-204：取消、失败与检索降级状态

**前置条件**：可触发运行中的取消、模型调用失败和混合检索向关键词检索降级。

**操作**：分别取消运行、模拟模型不可用，并执行发生检索降级的评审。

**预期结果**：

- 取消运行会向模型与工具传播取消信号并标记为 `cancelled`，迟到结果不得产生正式 Finding；
- 模型不可用时标记为 `failed` 并显示脱敏错误摘要，不展示本地模拟结果；
- 检索降级时，运行与相关证据清楚展示降级原因；无活动索引时不允许仅依赖模型启动评审。

### AC-205：模型配置版本与受控路由

**前置条件**：部署环境已注册至少两个支持结构化输出的模型，其中一个可作为回退模型。

**操作**：保存新的需求评审模型配置，启动运行；再使主模型不可用并启动另一运行。

**预期结果**：

- 保存产生新的模型配置版本，已存在运行不受影响；
- 智能路由关闭时使用默认模型；开启且主模型失败时，只有在允许降级、回退模型满足能力时才切换；
- 页面和调用记录显示实际模型、重试次数和降级原因；
- 不返回或保存 API Key、完整端点和 Authorization 请求头。

### AC-206：模型检测与输出校验

**前置条件**：存在一个可用模型和一个返回不符合结构化 Schema 的测试模型响应。

**操作**：对已注册模型执行连通性检测，并使用异常响应模型启动评审。

**预期结果**：

- 检测展示可用性、结构化输出能力、检测时间和脱敏错误；
- 异常响应导致运行显示结果校验失败，不生成成功统计和可处置 Finding；
- 页面不提供任意端点和明文密钥的输入或回显。

### AC-207：受控 Agent 工具循环与版本快照

**前置条件**：一份需求需要检索至少两份关联资料才能完成评审，评审 Agent 已配置知识库检索、原文读取、引用校验和候选结果提交工具。

**操作**：启动评审，观察 Agent 多轮调用工具并提交候选结构化结果；运行期间发布新的 Prompt 或工具集版本。

**预期结果**：

- Agent 只能调用运行快照中允许的工具，所有调用均经过 Schema、权限、超时和结果限制校验；
- 运行保存 Agent、Runtime、Prompt、工具集、MCP、网络策略、权限和执行限制版本，新发布版本不影响本次运行；
- 页面可查看真实工具执行步骤、状态和脱敏摘要，工具结果中的指令不能扩大 Agent 权限；
- `agent_end` 或候选结果不直接代表成功，只有服务端 Schema 与证据校验通过后才保存正式 Finding。

### AC-208：联网、MCP 与外部操作安全边界

**前置条件**：存在允许的联网/MCP 只读工具、一个越界目标、一个可逆写工具、一个高风险写工具及包含提示词注入内容的模拟响应。

**操作**：让 Agent 分别调用上述工具，并测试无确认、确认后参数变化、审批拒绝和取消运行。

**预期结果**：

- 允许范围内的只读调用可自动执行，越界网络、未注册 MCP 工具和参数 Schema 不合法的调用被拒绝；
- 提示词注入内容作为不可信数据处理，不能修改系统 Prompt、工具集、路由或审批策略；
- 可逆写操作遵循配置的自动执行或确认策略，高风险写操作未逐次确认不得执行；参数变化后原确认失效；
- 取消、拒绝或超时会终止后续调用，调用、风险、审批、结果与脱敏错误均有记录，敏感凭据不进入页面或普通日志。

## 8. 交付边界与后续衔接

本期交付的是以知识库证据为基础、由受控 Agent Runtime 驱动的需求评审闭环和生成式模型配置，不是面向用户的通用 Agent 编排、需求文档编辑、自动决策或工程执行平台。

- AI 评审结果和问答仅为派生产物；只有人工将其编辑并保存为独立资料版本后，才可按一期统一资产流程进入知识库。
- Pi Agent Core、受控 Prompt、Tool Runtime、联网与 MCP 是评审实现依赖；本期交付其运行、版本、权限、安全与观测边界，但不交付普通用户任意创建 Agent、注册工具、配置端点或发布编排的通用平台。
- Phase 3 技术方案分析应复用本期已确认的 Finding、人工处置、固定证据以及受控 Agent/Tool Runtime，但必须创建独立的方案运行与输入快照。
- Git、测试、组织级权限、完整审批、成本治理及通用 Prompt/Agent 管理界面应在后续阶段分别定义；底层已有相关 Tool/MCP 适配能力不代表这些产品能力已经交付。

## 9. 前置条件与待确认事项

### 9.1 前置条件

- 一期知识库可稳定提供需求资产版本、Chunk、活动索引和固定证据定位；
- 至少一组可用于试点评审的 Markdown/纯文本需求资料及关联业务规则资料；
- 部署环境已注册至少一个支持结构化输出的受控生成式模型；
- 服务端 Node.js 版本满足所锁定 Pi Agent Core 版本要求，并已完成 `ReviewAgentRuntime` 适配与兼容性测试；
- 已发布首个评审 Agent、Prompt、默认只读工具集、执行限制和权限策略版本；
- 如启用联网或 MCP，已完成 Server 注册、网络白名单、凭据托管、数据边界和提示词注入防护评审；
- 已定义外部操作风险等级、自动执行边界、人工确认门禁、幂等与补偿要求；
- 已确认本期允许发送至模型服务、联网与 MCP 工具的项目资料范围和保留策略；
- 已准备典型需求、预期问题、证据来源、工具调用、安全攻击样例和人工处置样例作为评审质量验收集。

### 9.2 待确认事项

1. Finding 的严重度定义、默认处置时限及是否需要后续对接工单系统；
2. 单次需求评审支持的最大输入长度，以及超长需求的后续分段评审策略；
3. 首批可注册的生成模型来源、模型清单、上下文窗口、数据传输边界和保留策略；
4. 关键词降级时是否允许继续生成评审，以及对结论可信度的展示规则；
5. 评审 Agent、Prompt、工具集和安全策略的维护责任、发布流程、版本审批与回滚机制；
6. 最大 Agent 回合、工具调用、Token、单工具结果、总超时和并发配额；
7. 首批联网域名、MCP Server、允许发送的数据范围及外部内容引用规则；
8. `write-reversible` 自动执行边界、`write-high-risk` 逐次确认方式、确认人范围和补偿责任；
9. 试点质量验收集、Finding 召回率、证据准确率、工具成功率和安全拒绝准确率等度量口径，以及人工复核责任人；
10. 后续何时交付成员权限、组织级审批、成本配额、通用 Prompt/Agent 可视化编排和多模态资料解析。
