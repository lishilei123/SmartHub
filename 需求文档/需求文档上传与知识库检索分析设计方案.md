# SmartHub 需求文档上传与知识库检索分析设计方案

## 1. 业务目标

用户上传一份 Markdown 需求文档后，SmartHub 自动完成：

1. 保存并解析上传的需求文档；
2. 从需求中提取功能点、约束、业务规则和待确认问题；
3. 针对每个需求点检索现有项目知识库；
4. 找出相关历史方案、接口、数据模型、业务规则和潜在冲突；
5. 生成一份带知识库来源引用的设计方案；
6. 保存分析结果，支持查看、重新分析和人工确认。

该功能不是简单地把整份需求文档交给大模型总结。核心要求是“先结构化需求，再逐项检索，再基于证据形成方案”，以降低遗漏和幻觉。

## 2. 与知识库同步方案的关系

本功能复用《SmartHub Markdown 知识库增量向量同步方案》中的基础能力：

- Upload API：接收和保存 Markdown 文件；
- Markdown AST Chunker：按标题与语义结构切分；
- Embedding Provider：生成查询向量；
- PostgreSQL + pgvector：存储和检索知识库 Chunk；
- `sync_jobs`/Worker：执行异步任务、重试和状态管理；
- 混合检索：结合向量相似度和全文关键词；
- 来源引用：返回文件路径、标题链、行号和 Chunk ID。

需要注意：上传的需求分析文件与正式知识库文档属于不同用途。需求文件默认进入“分析工作区”，不应在分析完成前自动成为正式知识库内容，否则它可能检索到自身并造成证据污染。只有用户明确点击“发布到知识库”后，才进入知识库同步流程。

## 3. 总体流程

```text
上传需求 Markdown
        │
        ▼
文件校验与持久化
        │
        ▼
需求结构化解析
  ├─ 背景/目标
  ├─ 功能需求
  ├─ 非功能需求
  ├─ 业务规则
  ├─ 数据与接口约束
  └─ 待确认问题
        │
        ▼
为每个需求点生成检索查询
        │
        ▼
知识库混合检索 + 权限过滤
        │
        ▼
去重、融合、Rerank、证据分组
        │
        ▼
需求覆盖/冲突/缺口分析
        │
        ▼
生成带引用的设计方案
        │
        ▼
人工确认 ──> 导出方案 / 发布到知识库 / 创建开发任务
```

## 4. 角色和使用场景

### 4.1 主要角色

- 产品经理：上传需求并查看需求完整性、冲突和待确认项；
- 架构师/开发人员：查看受影响模块、接口、数据模型和实施建议；
- 项目管理员：选择允许检索的知识库、管理权限和模型配置；
- 审核人员：确认方案后发布到正式知识库。

### 4.2 典型用例

1. 上传新的功能需求，系统检索类似历史需求和已有实现；
2. 识别新需求与现有业务规则、接口契约之间的冲突；
3. 分析会影响的模块、数据库表、API 和部署配置；
4. 根据项目既有规范生成统一格式的技术设计；
5. 对修改后的需求重新分析，仅处理变化的需求项。

## 5. 功能需求

### 5.1 上传需求文档

建议接口：

```http
POST /api/requirement-analyses
Content-Type: multipart/form-data

file: requirement.md
attachments: [login.png, user-list.webp]
attachmentManifest: [{"file":"login.png","logicalPath":"assets/login.png"}]
prototypePackage: prototype.zip
prototypeEntry: index.html
knowledgeBaseIds: ["kb-project", "kb-architecture"]
analysisTemplate: "technical-design-v1"
```

接口处理：

- 校验身份以及目标知识库的读取权限；
- 主文档第一阶段支持 UTF-8 Markdown，附件支持 PNG、JPEG 和 WebP 页面原型图；
- 可选上传 HTML/CSS/JavaScript 交互原型 ZIP 包；远程原型使用独立的受控 URL 接口登记；
- 校验主文档、附件数量、单文件及总大小、图片真实格式、像素和文件名；
- 流式计算 SHA-256 并保存原文件；
- 创建分析记录和异步任务；
- 返回 `202 Accepted`，不在 HTTP 请求中等待完整分析。

主文档和附件构成一个不可分割的“需求包”。服务端应生成对象 Key 保存文件，客户端文件名和相对路径仅作为展示及引用映射，不能直接拼接为服务器路径。推荐限制每个需求包最多 50 张图片、单图 10 MB、总大小 100 MB，具体值可配置。

返回示例：

```json
{
  "analysisId": "ra_01J...",
  "jobId": "job_01J...",
  "status": "queued"
}
```

### 5.2 需求结构化解析

先通过 Markdown AST 获取标题、段落、列表、表格和代码块，再由规则与模型结合提取需求项。每个需求项至少包含：

```json
{
  "requirementKey": "REQ-FUNC-001",
  "type": "functional",
  "title": "上传 Markdown 需求文档",
  "description": "用户可上传需求文档并发起分析",
  "acceptanceCriteria": ["返回分析任务 ID"],
  "constraints": ["仅允许读取有权限的知识库"],
  "keywords": ["上传", "Markdown", "异步任务"],
  "source": {
    "startLine": 12,
    "endLine": 18,
    "headingPath": ["文件上传"]
  }
}
```

需求类型建议包括：

- `business`：业务目标与规则；
- `functional`：功能行为；
- `non_functional`：性能、安全、可用性等；
- `data`：数据字段、生命周期和一致性；
- `integration`：接口及外部系统；
- `constraint`：技术、合规或部署限制；
- `question`：信息不足、歧义或需要确认的内容。

模型输出必须经过 JSON Schema 校验。不符合结构时允许有限次数的修复重试，不能直接把未校验结果写入正式分析表。

### 5.3 页面原型图解析与图文融合

系统先建立 Markdown 图片引用和上传附件的映射，再对实际被引用或人工绑定的图片进行解析：

1. 解码图片并校验真实格式，生成安全缩略图；
2. OCR 提取可见文字及坐标；
3. 多模态模型识别页面、区域、控件、状态和候选交互；
4. 将控件与 OCR 文本、图片区域及对应需求章节关联；
5. 输出结构化页面模型；
6. 比较文字需求和图片模型，生成一致、冲突、缺失和不确定项；
7. 将人工确认后的页面模型提供给测试点和 UI 用例生成环节。

结构化页面模型示例：

```json
{
  "pageKey": "PAGE-LOGIN",
  "name": "登录页",
  "attachmentId": "att_01J...",
  "states": ["default", "validation_error", "loading"],
  "elements": [
    {
      "elementKey": "username",
      "role": "textbox",
      "label": "用户名",
      "bounds": {"x": 120, "y": 180, "width": 320, "height": 40},
      "candidateRules": ["required"],
      "confidence": 0.92
    }
  ],
  "sourceRegion": {"x": 0, "y": 0, "width": 1440, "height": 900}
}
```

图文融合必须遵循：

- 图片中的可见文字属于直接证据，模型推断的点击效果或校验规则属于候选结论；
- 图片未展示某状态，不等于系统不存在该状态；
- 图文冲突时生成 `needs_confirmation=true` 的 finding；
- 每个图片结论引用 `attachment_id + attachment_version + region`；
- 低清晰度、严重裁剪或识别置信度不足时提示用户更换图片或人工标注；
- 原型图中出现的真实姓名、手机号和账号等信息在送入外部模型前脱敏。

### 5.4 HTML 交互原型解析

HTML 原型应作为可执行输入处理。首期接收包含 `index.html` 及相对静态资源的 ZIP 包；不能接受任意服务器目录或直接在平台主进程中打开 HTML。后续支持远程 URL 时，应只允许项目管理员配置的域名，并通过独立抓取服务访问。

处理流程：

```text
原型 ZIP
  → 安全解压与文件清单校验
  → 创建一次性隔离环境
  → 本地只读静态服务器
  → Playwright 隔离浏览器加载
  → 采集 DOM/Accessibility Tree/截图/控制台
  → 在预算内执行低风险交互
  → 生成页面状态图与交互轨迹
  → 与文字需求、静态原型图进行一致性分析
```

需要采集：

- 页面 URL、标题、路由和视口；
- DOM 结构、ARIA role/name、label、placeholder 和可见文本；
- 表单字段、按钮、链接、菜单、弹窗、表格和分页；
- 初始、加载、空数据、校验失败、成功等可达状态；
- 每次操作前后的 DOM 差异、截图和网络请求摘要；
- JavaScript 错误、资源加载失败和可访问性问题候选；
- 用户录制或指定的关键业务流程。

自动探索必须有硬边界：最大页面数、最大操作步数、单页超时、总任务超时和最大产物大小。默认只允许点击导航、展开、切换和在明确的测试表单中输入合成数据。删除、提交订单、支付、授权、文件上传及任何无法判断副作用的操作不得自动执行。

隔离要求：

- 使用一次性容器或等价的强隔离执行环境，并以非特权用户运行；
- 禁止访问宿主文件系统、云元数据地址、内网网段和平台控制面；
- 默认阻断所有外部网络；确需远程资源时使用显式域名白名单和响应大小限制；
- 设置 CPU、内存、进程数、磁盘和执行时间配额；
- 禁止浏览器下载、弹出外部应用和持久化凭据；
- 原型脚本、DOM 文本和网络响应均视为不可信内容，不得成为 Agent 系统指令；
- 执行结束销毁容器、浏览器上下文、Cookie 和临时数据。

HTML 原型证据引用采用 `prototype_id + version + page_id + state_id + element_id/action_id`，并关联对应 DOM 快照与截图。系统必须区分“实际观察到的交互结果”和“AI 推测但尚未执行的候选行为”。

### 5.5 知识库检索

不能只用整份需求文档进行一次检索。应针对每个需求项生成多个互补查询：

- 语义查询：需求描述和验收标准；
- 关键词查询：模块名、接口名、表名、错误码和业务术语；
- 约束查询：性能、安全、权限、部署等非功能条件；
- 关系查询：可能受影响的上下游组件。

推荐检索过程：

1. 对查询进行 Embedding；
2. pgvector 语义召回 Top 30；
3. PostgreSQL 全文检索召回 Top 30；
4. 按知识库权限、文档状态、标签和项目范围过滤；
5. 使用 RRF 合并；
6. 按 `document_id + chunk_key` 去重；
7. 可选 reranker 重排；
8. 为每个需求项保留 Top 5～10 条证据。

检索结果必须记录当时使用的 `document_id`、`document_version` 和 `chunk_id`。否则知识库发生更新后，用户无法还原生成方案时所依据的内容。

### 5.6 相关需求分析

系统针对每个需求项输出以下判断：

| 分析类型 | 说明 |
|---|---|
| 已有能力 | 知识库中是否已有相同或可复用实现 |
| 相关需求 | 与哪些历史需求、设计或规范相关 |
| 影响范围 | 可能影响的模块、接口、表、任务和部署项 |
| 一致性 | 是否符合现有架构和业务规则 |
| 冲突 | 是否与已有规则、字段语义或接口契约冲突 |
| 缺口 | 哪些内容无法从需求和知识库确定 |
| 风险 | 安全、性能、迁移、兼容性和运维风险 |
| 建议 | 复用、修改、新增或废弃的技术建议 |

每一项结论都应包含：

- `conclusion`：结论；
- `confidence`：`high`、`medium` 或 `low`；
- `evidenceIds`：支持结论的证据；
- `reasoningSummary`：简短依据；
- `needsConfirmation`：是否需要人工确认。

没有证据时必须明确标记“知识库中未找到依据”，不能把模型常识伪装成项目事实。

### 5.7 生成设计方案

默认输出结构：

1. 需求概述；
2. 需求范围与非目标；
3. 现有系统关联分析；
4. 需求覆盖矩阵；
5. 总体架构；
6. 模块设计；
7. 数据模型与迁移；
8. API 与事件设计；
9. 权限和安全；
10. 异常、重试和一致性；
11. 兼容性与上线方案；
12. 测试及验收标准；
13. 风险和待确认问题；
14. 知识库引用。

方案中的项目事实必须带引用，例如：

```text
现有知识库已使用 document.active_version 控制活动版本。
[来源：SmartHub Markdown 知识库增量向量同步方案.md > 数据模型，版本 3，Chunk abc123]
```

### 5.8 Markdown 在线查看与编辑

需求主文档、生成的设计方案以及用户有权限修改的相关 Markdown 均可在浏览器中查看和编辑。编辑器提供源码、预览和分屏模式，并支持目录、行号、搜索、语法高亮、图片相对引用和未保存提示。

保存过程：

```text
浏览器提交 content + baseVersion + contentHash
  → 权限与 CSRF 校验
  → Markdown/附件引用校验
  → 比较当前版本与 baseVersion
  → 无冲突：创建不可变新版本
  → 更新文档 current_version
  → 标记旧分析可能过期
  → 正式知识库文档另行创建增量同步任务
```

并发编辑使用乐观锁。若 `baseVersion` 已落后，接口返回 `409 Conflict` 以及基准、服务端当前版本和用户草稿的比较信息，由用户选择合并，不执行最后写入者覆盖。自动保存只保存个人草稿；明确点击保存后才创建共享文档版本。

恢复历史版本不是移动指针，而是复制历史内容创建一个新版本，确保审计链连续。AI 修改也必须先生成 unified diff 或结构化 Patch，用户预览并确认后才能进入同一保存流程。

### 5.9 一键需求 AI 分析

需求详情页提供“一键 AI 分析”。服务端触发时创建 `analysis_snapshot`，固定以下输入：

- Markdown 文档版本及内容 Hash；
- 图片附件和 HTML 原型版本；
- 目标知识库集合及可见文档版本；
- 分析模板、模型、检索和预处理版本；
- 发起人的权限快照及项目范围。

任务依次执行解析、原型融合、检索、分析和方案生成。API 立即返回任务 ID，前端通过 SSE 或轮询显示阶段进度。相同输入指纹已有运行中任务时直接返回现有任务；已有成功结果时提示复用或明确选择重新生成，避免误触重复消耗模型费用。

分析过程中用户仍可编辑文档，但分析结果始终指向快照版本。完成时若当前文档已更新，界面标记“结果基于旧版本”，并提供对新版本重新分析入口。

### 5.10 上下文 AI 对话

每个需求分析可建立多个对话会话。创建会话时指定可用上下文范围，后续每轮仅在该范围和用户实时权限内检索：

- 当前需求或用户指定版本；
- 结构化需求项、findings 和生成方案；
- 图片区域、HTML 原型状态与交互轨迹；
- 选定知识库、代码变更、用例和执行结果。

每轮对话过程为：问题分类 → 权限过滤 → 混合检索 → 上下文裁剪 → 模型回答 → 引用校验 → 保存消息。回答中的项目事实必须引用 evidence ID；无法验证时明确说明“未找到项目依据”。

支持的对话操作包括解释需求、发现遗漏、比较版本、生成验收标准、提出测试点、分析冲突以及建议修改 Markdown。任何产生副作用的操作都只生成操作建议或 Patch，用户确认后再调用对应业务接口。

会话保存标题、创建人、模型、消息、引用快照和工具调用摘要。删除采用权限校验和保留策略；导出时包含引用清单。历史消息可以保留文本，但再次展示引用原文时必须重新检查权限，权限撤销后不得返回受限内容。

## 6. 数据模型

```sql
CREATE TYPE analysis_status AS ENUM (
  'queued', 'parsing', 'retrieving', 'analyzing',
  'generating', 'ready', 'failed', 'cancelled'
);

CREATE TABLE requirement_analyses (
  id                    uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  owner_id              uuid NOT NULL,
  source_uri            text NOT NULL,
  original_filename     text NOT NULL,
  source_hash           char(64) NOT NULL,
  current_document_version integer NOT NULL DEFAULT 1,
  analysis_stale        boolean NOT NULL DEFAULT false,
  status                analysis_status NOT NULL DEFAULT 'queued',
  template_version      text NOT NULL,
  parser_version        text NOT NULL,
  model_name            text NOT NULL,
  progress              integer NOT NULL DEFAULT 0 CHECK (progress BETWEEN 0 AND 100),
  current_stage         text,
  last_error            text,
  created_at            timestamptz NOT NULL DEFAULT now(),
  updated_at            timestamptz NOT NULL DEFAULT now(),
  completed_at          timestamptz
);

CREATE TABLE requirement_document_versions (
  id                    uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  analysis_id           uuid NOT NULL REFERENCES requirement_analyses(id) ON DELETE CASCADE,
  version               integer NOT NULL,
  content_markdown      text NOT NULL,
  content_hash          char(64) NOT NULL,
  change_summary        text,
  created_by            uuid NOT NULL,
  created_at            timestamptz NOT NULL DEFAULT now(),
  UNIQUE (analysis_id, version)
);

CREATE TABLE requirement_drafts (
  analysis_id           uuid NOT NULL REFERENCES requirement_analyses(id) ON DELETE CASCADE,
  user_id               uuid NOT NULL,
  base_version          integer NOT NULL,
  content_markdown      text NOT NULL,
  updated_at            timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (analysis_id, user_id)
);

CREATE TABLE requirement_analysis_kbs (
  analysis_id           uuid NOT NULL REFERENCES requirement_analyses(id) ON DELETE CASCADE,
  knowledge_base_id     uuid NOT NULL REFERENCES knowledge_bases(id),
  PRIMARY KEY (analysis_id, knowledge_base_id)
);

CREATE TABLE requirement_attachments (
  id                    uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  analysis_id           uuid NOT NULL REFERENCES requirement_analyses(id) ON DELETE CASCADE,
  attachment_version    integer NOT NULL DEFAULT 1,
  original_filename     text NOT NULL,
  logical_path          text NOT NULL,
  object_uri            text NOT NULL,
  media_type            text NOT NULL,
  content_hash          char(64) NOT NULL,
  byte_size             bigint NOT NULL,
  width                 integer,
  height                integer,
  parse_status          text NOT NULL DEFAULT 'queued',
  ocr_result            jsonb,
  page_model            jsonb,
  created_at            timestamptz NOT NULL DEFAULT now(),
  UNIQUE (analysis_id, logical_path, attachment_version)
);

CREATE TABLE requirement_prototypes (
  id                    uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  analysis_id           uuid NOT NULL REFERENCES requirement_analyses(id) ON DELETE CASCADE,
  version               integer NOT NULL DEFAULT 1,
  source_type           text NOT NULL CHECK (source_type IN ('zip', 'url')),
```

## 7. 分析任务状态机

```text
queued
  → parsing
  → retrieving
  → analyzing
  → generating
  → ready

任意执行阶段 → failed
任意未完成阶段 → cancelled
```

各阶段应保存检查点：

- `parsing` 完成后持久化需求项；
- `retrieving` 完成后持久化证据快照；
- `analyzing` 完成后持久化 findings；
- `generating` 完成后保存最终 Markdown。

Worker 重启时从最后一个完整检查点继续，避免再次调用所有模型。重复执行必须以 `analysis_id + requirement_key + chunk_id` 保证幂等。

## 8. 增量重新分析

用户修改并重新上传需求文档时：

1. 对新的结构化需求项计算 `source_hash`；
2. 与上一分析版本按稳定 `requirement_key` 和 Hash 比较；
3. 未变化项可以复用检索证据和分析结果；
4. 新增或变化项重新检索与分析；
5. 删除项从新方案中移除；
6. 若知识库版本发生变化，可选择“仅刷新过期证据”或“全部重新分析”。

复用分析结果必须同时满足：

- 需求项 Hash 相同；
- 目标知识库集合相同；
- 引用的文档版本仍然有效；
- 检索配置、分析模板和模型版本兼容。

## 10. Prompt 与输出控制

分析模型输入应分层组织：

```text
系统约束
  - 只能将 Evidence 中内容视为项目事实
  - 无依据必须标记未知
  - 每个关键结论必须引用 evidenceId

当前需求项
  - 结构化需求内容

检索证据
  - evidenceId、来源、版本、正文摘录

输出 Schema
  - findings、risks、questions、recommendations
```

安全要求：

- 上传文档和知识库内容都属于不可信数据；其中出现的“忽略系统指令”等内容不能改变系统规则；
- 不允许模型自行扩大知识库访问范围；
- 模型看到的是通过服务端权限过滤后的证据；
- 限制单次上下文大小，超长证据先按需求项聚类和压缩；
- 所有结构化输出进行 Schema 校验和引用存在性校验；
- 引用摘录不得指向任务执行后才出现的文档版本。

## 11. 前端交互

### 11.1 上传页

- 上传或拖放 Markdown 主文档及 PNG/JPEG/WebP 页面原型图；
- 可选上传 HTML 原型 ZIP、指定入口文件，或由管理员登记受控原型 URL；
- 展示需求包目录和 Markdown 图片引用映射，对未引用图片提供人工绑定；
- 支持图片预览、删除、排序以及选择对应需求章节；
- 选择一个或多个目标知识库；
- 选择分析模板；
- 展示文件校验结果和预计处理范围；
- 提交后进入分析详情页。

### 11.2 分析详情页

- 顶部显示阶段、百分比、耗时和失败原因；
- 顶部提供“一键 AI 分析”，允许选择模板、知识库和范围，并显示重复任务提示；
- Markdown 支持源码、渲染、分屏编辑、个人草稿、保存和离开提醒；
- 支持查看版本历史、比较 Diff、恢复为新版本和处理并发冲突；
- 左侧展示结构化需求树；
- 中间展示关联需求、影响范围、冲突和缺口；
- 右侧展示知识库和原型图证据，可跳转原文或高亮图片区域；
- 支持在原型图上框选控件、修正名称/类型并确认页面模型；
- 支持查看 HTML 原型页面状态图、交互轨迹、DOM/截图证据和未探索路径；
- 最终方案支持预览、下载、复制和版本对比；
- 人工确认待定项后可重新生成方案；
- 只有具备权限的用户才能发布到正式知识库。

### 11.3 AI 对话面板

- 可从需求详情页右侧打开，支持新建、重命名、切换、删除和导出会话；
- 默认关联当前需求版本，用户可附加需求项、原型区域、方案章节或知识库范围；
- 支持选择 Markdown 文本后直接发起“解释、完善、生成测试点、建议修改”；
- 流式展示回答，并在句段旁显示可点击引用；
- 点击文档引用定位到版本和行/Chunk，图片引用高亮区域，HTML 引用打开对应状态；
- 修改建议以 Diff 展示，用户确认后进入文档保存流程；
- 明确展示模型、上下文范围、是否使用知识库以及无证据提示；
- 会话期间权限变化时重新校验，隐藏已失去权限的证据内容。

## 12. 异常处理

| 异常 | 处理方式 |
|---|---|
| Markdown 无有效需求内容 | 标记失败并提示调整文档结构 |
| 图片格式伪造或解码失败 | 拒绝该附件并返回明确错误，不交给模型处理 |
| Markdown 引用的图片缺失 | 标记需求包不完整，提示补传或移除引用 |
| 原型图无法可靠识别 | 保留原图并转人工标注，不阻塞纯文本需求分析 |
| 图文描述冲突 | 生成待确认项，同时保留两方证据 |
| ZIP 路径穿越、压缩炸弹或非法链接 | 拒绝整个原型包并记录安全事件 |
| HTML 脚本超时或资源失控 | 终止隔离环境，保留已采集状态并转人工检查 |
| 原型依赖被阻断的外部资源 | 展示资源清单，由管理员决定是否加入临时白名单 |
| 自动探索遇到高风险动作 | 不执行，记录为候选路径并等待人工录制或批准 |
| 指定知识库为空 | 允许继续，但明确标记无项目证据 |
| 检索无结果 | 生成缺口和待确认项，不伪造结论 |
| Embedding/模型限流 | 指数退避，保留阶段检查点 |
| 某个需求项分析失败 | 单项重试，其他需求项继续 |
| 知识库在分析中更新 | 本次使用已记录的版本快照，下次提示刷新 |
| 用户取消 | 停止领取新步骤，保留或按策略清理中间结果 |
| 发布失败 | 分析方案仍保留，知识库同步任务可独立重试 |
| Markdown 保存版本冲突 | 返回 `409` 和三方对比信息，保留用户草稿供合并 |
| Markdown 附件引用失效 | 阻止正式保存或允许保存为草稿，并列出失效路径 |
| 重复点击一键分析 | 返回相同输入指纹的运行中任务或提示复用已有结果 |
| 分析期间文档被修改 | 继续基于快照执行，完成后明确标记结果已过期 |
| AI 对话引用权限被撤销 | 不再返回原文，并提示引用当前不可访问 |
| 对话模型不可用 | 保留消息草稿并允许重试，不影响文档查看和编辑 |

## 13. 可观测性

至少记录：

- 每个阶段耗时、成功率和重试次数；
- 解析出的需求项数量及类型分布；
- 原型图数量、OCR/多模态耗时、控件识别数和人工修正率；
- HTML 原型页面/状态/动作数量、探索覆盖率、阻断资源和沙箱终止次数；
- 每个需求项的检索查询数、召回数和最终证据数；
- 无证据需求项比例、低置信度结论比例；
- Token、Embedding 和生成模型费用；
- 人工采纳、修改、驳回和重新生成比例；
- 从上传到可查看方案的 P50/P95 总耗时。
- Markdown 保存成功率、版本冲突率和草稿恢复率；
- 一键分析重复任务复用率、取消率和各阶段耗时；
- AI 对话首 Token/总响应耗时、引用有效率、建议采纳率和单会话成本。

日志携带 `analysis_id`、`job_id`、`requirement_item_id` 和 `trace_id`，但不记录完整需求正文、敏感证据或密钥。

## 14. 验收标准

### 14.1 功能验收

- 上传有效 Markdown 后返回 `analysisId` 和 `jobId`；
- 可在同一请求中上传多张页面原型图，并正确建立 Markdown 引用映射；
- 能从原型图提取页面、控件、可见文字、候选状态和候选校验规则；
- 图文冲突被标记为待确认项，不静默覆盖任何一方；
- 图片分析结论可定位到具体附件版本和矩形区域；
- 原型图解析失败时，纯文本需求分析仍可完成；
- 可安全上传包含多页面及本地资源的 HTML 原型 ZIP 并指定入口；
- 能生成 HTML 原型的页面状态、控件和低风险交互轨迹；
- HTML 证据可定位到具体原型版本、页面状态、元素和动作；
- 高风险交互不会被自动执行，外部网络和宿主资源默认不可访问；
- 恶意 ZIP、无限循环脚本和超额资源使用能被拒绝或终止；
- 能从文档中提取功能、非功能、数据、接口和待确认需求；
- 每个需求项分别执行知识库检索；
- 生成方案包含需求覆盖、相关资料、冲突、缺口、风险和实施建议；
- 项目事实均可通过引用跳转到具体文档版本和 Chunk；
- 检索无结果时明确提示依据不足；
- 无知识库权限的用户无法读取对应证据；
- 任务失败后可从检查点重试；
- 修改一个需求项后只重新分析受影响项；
- 发布操作经过人工确认，并通过知识库同步任务入库。
- 能在线查看、预览和编辑 Markdown，正式保存后产生不可变新版本；
- 两人基于同一版本并发修改时不会发生静默覆盖；
- 能查看版本 Diff，并将历史内容恢复为新的当前版本；
- 点击一次即可创建完整 AI 分析任务并查看实时阶段进度；
- 相同输入的连续点击不会无提示创建重复计费任务；
- 分析结果固定引用触发时的文档、附件和知识库快照；
- 能围绕当前需求进行多轮 AI 对话，并查看每项项目事实的引用；
- AI 对话提出的 Markdown 修改以 Diff 展示，未经确认不改变文档；
- 用户权限撤销后不能通过历史对话重新获得受限证据原文；

### 14.2 质量验收

- 引用有效率达到 100%，不存在无法解析的 evidenceId；
- 需求项覆盖率大于 95%，未提取项能通过评测集发现；
- 高置信度项目事实的引用支持率达到 100%；
- 对明确冲突测试集的识别召回率达到预先设定目标；
- 同一输入、同一证据快照和同一配置下，输出结构保持稳定；
- 人工审核可以区分“知识库事实”“模型建议”和“待确认内容”。

## 15. 分阶段实施

### Phase 1：最小可用闭环

- Markdown 需求上传和异步任务；
- 页面原型图附件上传、OCR/多模态解析、图文融合和人工校正；
- HTML 原型 ZIP 上传、沙箱渲染、DOM/可访问性采集和有限交互探索；
- AST 解析与结构化需求提取；
- 按需求项执行混合检索；
- 保存证据快照；
- 生成带引用的 Markdown 设计方案；
- 状态查询、失败重试和方案下载。
- Markdown 在线查看、编辑、草稿、版本历史和 Diff；
- 一键 AI 分析、输入快照、进度展示与任务去重；
- 需求上下文 AI 对话、流式回答、引用和修改建议确认。

### Phase 2：分析质量

- Reranker、查询扩展和证据去重；
- 需求覆盖矩阵、冲突与缺口分析；
- 人工确认项和重新生成；
- 固定评测集与引用质量监控；
- 增量重新分析。

### Phase 3：流程集成

- 批量需求和跨需求多文件包；
- 方案审核、版本对比与正式知识库发布；
- 对接项目管理系统创建任务；
- 多模板、多项目和多租户策略；
- 分析结果反馈闭环与质量优化。

## 16. 推荐实现边界

第一阶段主文档只支持 Markdown，同时支持 PNG、JPEG、WebP 页面原型图和自包含 HTML 原型 ZIP，先把“需求项提取 → 图文/交互融合 → 逐项检索 → 证据快照 → 带引用方案”做可靠。Word、PDF、Figma 直连、需要登录的远程原型和复杂跨系统交互可后续增加，它们只应扩展文档解析层，不应改变后续检索和分析主流程。

最终形成两类明确的数据流：

```text
知识文档上传/本地修改
  → 同步到正式知识库

需求文档上传
  → 临时分析工作区
  → 检索正式知识库
  → 生成设计方案
  → 人工审核后可选择发布到正式知识库
```

这种隔离可以避免未确认需求污染知识库，同时让生成方案具有可追溯的项目依据。
