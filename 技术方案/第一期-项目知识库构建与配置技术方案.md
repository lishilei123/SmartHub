## 2. 方案目标、范围与边界

### 2.1 方案目标

本方案将项目内独立维护的 Markdown/纯文本资料纳入统一知识资产模型，实现来源可追溯、内容版本不可变、同步增量化、检索可定位和配置可控。方案为后续 AI 分析提供可信输入与证据边界，但一期不建设 AI 分析、AI 方案生成或对话能力。

实施结果应满足以下目标：

1. 项目创建时自动拥有唯一默认知识库，并能够查看资料、任务、配置和活动索引状态。
2. 网页/API 上传先固化资料快照和持久化任务，再由 Worker 异步执行资产识别、切分、Embedding 和索引发布；未来 Wiki 等连接器复用相同契约。
3. 每次内容变化形成不可变资产版本；检索和证据读取始终指向确定的 `assetVersionId` 与原文位置。
4. 通过整体内容 Hash、稳定 Chunk 标识和兼容性指纹复用无变化处理结果。
5. 新结果先构建候选索引，校验通过后原子切换；同步、重建、取消或 Worker 恢复不影响既有活动索引。
6. 配置变更可追溯，只有影响索引兼容性的变更才要求用户确认并发起异步重建。

### 2.2 一期交付范围

| 能力域 | 一期交付 |
|---|---|
| 知识库 | 项目默认知识库自动创建、概览、活动索引状态与配置。 |
| 资料格式 | UTF-8 Markdown（`.md`）和 UTF-8 纯文本（`.txt`）。 |
| 接入来源 | 网页/API 受控上传；未来 Wiki 等文档系统通过标准 Source Connector 扩展。 |
| 资料管理 | 稳定知识资产、不可变资产版本、资料类型、来源绑定、逻辑路径、版本与原文浏览。 |
| 同步索引 | 内容校验、Hash 去重、结构化切分、增量 Embedding、候选索引、活动索引切换。 |
| 查询能力 | 关键词检索；Embedding 与活动向量索引可用时的向量/混合检索；资料类型、来源与路径筛选。 |
| 配置与运维 | 配置草稿/版本、影响分析、受控重建、任务进度、错误、重试、取消与中断恢复。 |

### 2.3 边界约束

一期不实现 AI 分析或生成、Git/代码资料接入、测试执行、产品级 ACL/SSO、跨项目检索、多知识库管理，亦不实现 OpenAPI、Excel、PDF、Word 等专用格式解析器。完整非目标以需求文档“2.2 本期非目标”为准。

资料只有在被人工创建、导入或确认保存为独立项目资料版本后才会成为知识资产；AI 中间产物、会话消息和临时草稿不自动入库。后续若保存为独立资料，应复用本方案的统一接入链路。

## 3. 总体技术方案

### 3.1 核心实现路径

采用“**统一接入、固定快照、异步增量处理、候选索引构建、原子切换、固定版本证据检索**”的技术路径：

```text
网页/API 上传
        │
        ▼
文件校验、服务端快照、来源标准化、资产识别
        │
        ▼
持久化 SyncJob（受理后异步执行）
        │
        ▼
Worker：解析 → 稳定切分 → Hash/兼容性匹配 → 向量复用或生成
        │
        ▼
候选 AssetVersion / Chunk / IndexVersion 构建与完整性校验
        │
        ▼
条件事务激活新索引 → 默认检索读取固定成员 → 原文按固定版本定位
```

上传 API 仅负责校验、固化服务端可读的不可变资料快照和创建/合并持久化任务，不在请求内解析、切分、生成 Embedding 或更新活动索引。同步 Worker 处理任务并在候选校验后切换索引。未来 Wiki 等连接器也只允许投递固定快照和标准任务。

### 3.2 核心原则

1. **资产稳定、版本不可变**：文件名和路径仅是来源信息；资产以 `assetId` 表示，内容变化以新的 `assetVersionId` 表示。
2. **任务至少一次、处理必须幂等**：重复上传、任务重复投递、重试和中断恢复不能产生重复版本、重复 Chunk 或重复向量。
3. **候选优先于覆盖**：任何可能影响默认检索范围的处理均须先写入候选，再校验和切换。
4. **活动索引是唯一默认真相**：默认检索只读取活动 `IndexVersion` 的固定成员，禁止按数据库中“最新资产版本”拼装结果。
5. **配置快照隔离**：编辑草稿、已保存配置与活动索引实际使用的配置必须分离，避免活动索引内混入不兼容结果。
6. **检索必须返回固定证据**：返回资产版本、Chunk、标题路径和原文位置；打开结果不得自动跳转为资产最新版本。

### 3.3 组件职责与边界

| 组件 | 主要职责 | 明确不负责 |
|---|---|---|
| 项目与知识库服务 | 项目创建时初始化默认知识库；提供概览、统计、活动索引状态。 | 解析资料、执行同步任务。 |
| 配置服务 | 维护草稿、已保存配置、影响分类、待重建状态和配置快照。 | 直接修改活动索引内容。 |
| 上传接入适配器 | 校验文件、生成服务端资料快照、支持幂等受理。 | 在请求线程执行解析或模型调用。 |
| Source Connector 扩展边界 | 未来接入 Wiki 等文档接口，将固定内容快照和标准元数据投递到同步任务。 | 在一期直接写入 Chunk、Embedding 或活动索引。 |
| 资产识别与快照服务 | 规范化路径、处理冲突、识别资产/版本、维护来源绑定。 | 执行长时间同步任务。 |
| 任务存储与 Worker | 领取任务、租约心跳、重试、取消、幂等提交和恢复。 | 取代领域服务的业务规则。 |
| 解析与切分器 | 解析 Markdown/TXT，生成结构、定位、Chunk 与 Hash。 | 决定资料是否允许接入。 |
| Embedding Provider 适配器 | 批量向量化、健康检查、错误分类与模型元数据封装。 | 将 Provider 专有细节暴露给前端。 |
| 候选索引服务 | 建立固定成员范围、完整性校验、原子激活与延后回收。 | 对未完成候选索引提供默认检索。 |
| 检索与证据服务 | 在活动索引内检索、过滤、排序，读取固定版本原文。 | 将命中结果替换为资产最新版本。 |

前端、上传 API 与未来 Source Connector 不得直接写入 Chunk、Embedding 或活动索引指针。Worker 只读取服务端生成的不可变资料快照，不信任客户端传入的物理存储路径。

## 4. 数据与存储方案

### 4.1 逻辑对象

| 对象 | 保存内容 | 关键约束 |
|---|---|---|
| `KnowledgeBase` | 项目归属、名称、当前配置版本、活动索引版本。 | 一期每项目一个默认库；活动索引仅指向通过校验的版本。 |
| `KnowledgeAsset` | 稳定资产标识、显示名称、资料类型、当前活动版本。 | 路径和文件名不是永久身份。 |
| `AssetVersion` | 来源、逻辑路径、整体内容 Hash、原文快照引用、状态、配置快照、错误与时间。 | 同资产相同 Hash 不重复创建；`ready` 后不可原地修改。 |
| `SourceBinding` | 来源类型、来源标识、规范化路径和资产关联。 | 一期记录上传来源；为未来连接器游标与外部 ID 预留，不替代资产身份。 |
| `Chunk` | `chunkKey`、内容 Hash、标题路径、原文位置、Token 数、Embedding 关联。 | 仅归属固定资产版本，定位必须可复现。 |
| `EmbeddingRecord` | 规范化输入 Hash、兼容性指纹、Provider、模型、维度、向量状态。 | 只有输入和兼容性指纹都一致时才允许复用。 |
| `IndexVersion` | 候选/活动/历史状态、固定成员范围、配置快照、激活时间、错误。 | 同一知识库同一时刻只能有一个活动索引。 |
| `IndexMembership` | 索引版本与资产版本/Chunk 的固定关联。 | 使检索范围、审计和历史查询可复现。 |
| `SyncJob` | 任务类型、输入范围、状态、步骤、进度、尝试数、租约、错误、配置快照。 | 统一表达同步、删除和重建。 |
| `ConfigDraft` / `ConfigVersion` | 草稿或已保存配置、版本号、影响分类、待重建标识。 | 草稿不覆盖保存版本；活动索引始终保留实际使用的配置快照。 |

### 4.2 存储职责划分

| 数据类别 | 推荐初版方案 | 原因与约束 |
|---|---|---|
| 原始资料快照 | 受控对象存储或受控本地目录。 | 保存不可变原文引用；对象键或受控路径必须由服务端生成。 |
| 事务元数据、配置、任务、索引成员 | 关系数据库。 | 支持唯一约束、状态机、条件更新和活动索引原子切换。 |
| 关键词与向量检索 | 关系数据库的全文检索和向量扩展能力。 | 初版降低多套基础设施复杂度；规模评测后可演进专用检索或向量服务。 |
| 候选与回收记录 | 与元数据同一持久化系统，异步清理。 | 失败或取消时避免破坏活动资料和索引。 |

PostgreSQL 16+、pgvector、Node.js/TypeScript、对象存储及特定 Markdown AST 库均是当前推荐初版，不是本方案预先批准的强制选型。当前 PostgreSQL 实现按实体增量写入，事务内读取数据库最新状态并通过数据库锁串行化条件切换，禁止全库 `TRUNCATE + 重写` 作为业务写路径。

### 4.3 一致性不变量

1. 同一 `assetId` 下，相同整体内容 Hash 至多对应一个资产版本。
2. `ready` 资产版本、其 Chunk 与原文定位均不可修改；内容、类型或解析结果更正以新版本表达。
3. `KnowledgeBase.activeIndexVersionId` 只能指向已完成完整性校验的活动索引。
4. 候选索引的成员、Chunk、Embedding 和配置快照必须兼容；不同模型维度或预处理/切分版本不得混入同一活动索引。
5. 失败、取消、重复投递、Worker 崩溃或乱序事件不得改变已激活索引。
6. 删除资料仅能在不含该资料的新候选索引激活后影响默认检索范围；历史版本按保留策略可读。
7. Worker 提交前必须确认任务未取消、输入版本与配置快照未被更新任务取代，防止陈旧任务覆盖新结果。

## 5. 接入、同步与索引发布

### 5.1 统一接入契约

上传来源标准化为下列信息后进入同一链路；未来 Source Connector 必须遵守相同契约：

- `projectId`、`knowledgeBaseId`；
- `sourceType`：一期固定为 `upload`；未来连接器使用独立枚举值；
- `sourceIdentifier`：一期为服务端对象键；未来可为 Wiki 页面 ID 等稳定外部标识；
- `assetType`、`displayName`；
- `logicalPath` 及用于比较的 `canonicalPath`；
- 不可变资料快照引用、整体 `contentHash`；
- 接入时适用的已保存 `configVersion`；
- 任务类型：`upsert`、`delete` 或 `rebuild`。

`canonicalPath` 使用统一的 `/` 分隔符，并按部署平台规则标准化比较；Windows 环境保留原始逻辑路径供展示。逻辑路径用于发现和冲突处理，不作为资产永久身份。

### 5.2 上传处理流程

1. 服务端校验扩展名、内容、UTF-8、大小、逻辑路径和 `Idempotency-Key`，不信任客户端 MIME、文件名或物理路径。
2. 文件以流式方式写入服务端受控暂存位置并计算整体 Hash；提交后生成不可变快照引用。
3. 根据知识库、规范化路径、冲突策略与 Hash 识别资产和版本。
4. 同资产内容未变化时复用已成功版本，返回 `200 OK` 和复用状态；不创建版本、任务、Embedding 或索引版本。
5. 新资料或内容变化时创建或合并 `upsert` 任务，返回 `202 Accepted`、`assetId`、`assetVersionId` 和 `jobId`。
6. 只有候选构建、校验及索引激活成功后，资产版本才转换为 `ready`。

默认冲突策略建议为 `new_version`：同逻辑资料更新时保留 `assetId` 并新建不可变版本；`reject` 与 `parallel_asset` 是否开放应由配置与产品规则共同确认。

### 5.3 未来 Source Connector 扩展边界

一期不实现本地目录监听、扫描或来源对账。未来 Wiki、文档平台等连接器应作为独立适配器建设：连接器负责认证、增量游标、限流和固定来源快照，只向现有上传/同步边界提交 `sourceType`、稳定外部 ID、逻辑路径、内容 Hash 和不可变快照引用；Worker、资产版本、候选索引和检索证据链保持复用。连接器的删除、回调、全量校准和冲突优先级需在对应阶段单独设计与验收。

### 5.4 Worker 增量同步流程

```text
1. 领取可执行任务，建立租约和心跳；对知识库与规范化路径或资产范围加互斥保护。
2. 读取上传时已固化的固定资料快照。
3. 计算整体内容 Hash；与当前成功版本一致时更新来源观察信息并结束。
4. Markdown 按 AST 或等价结构解析；TXT 按段落、行区间或句子边界切分。
5. 输出标题路径、原文位置、chunkKey、内容 Hash、Token 数和兼容性指纹。
6. 与上一个成功版本匹配 Chunk；未变化且兼容的 Chunk 复用 Embedding，仅对新增/变化 Chunk 批量调用 Provider。
7. 写入候选资产版本、Chunk、Embedding 关联、候选 IndexVersion 和 IndexMembership。
8. 校验固定成员范围、Chunk 数量、Embedding 状态、配置兼容性、任务有效性和取消状态。
9. 在单个条件事务中切换资产活动版本、知识库活动索引、索引状态、资产版本状态和任务状态。
10. 延后异步回收临时对象、失败候选和失去引用的数据；不在激活前删除旧活动内容。
```

对可重试错误（限流、超时、临时服务错误）按最大尝试次数、指数退避和抖动执行重试；非法编码、文件超限、格式不支持和路径不合法等确定性错误直接失败。用户取消仅清理未提交候选，不回滚或删除旧活动索引。租约超时的 `running` 任务必须可被安全接管。

### 5.5 增量与稳定切分规则

#### 资料级去重

相同资产的整体 `contentHash` 未变化时，直接复用最近成功结果，不再次解析、切分、调用 Embedding 或生成索引。

#### Chunk 级复用

内容变化时，使用稳定 `chunkKey`、Chunk 内容 Hash 与 Embedding 兼容性指纹匹配上一个成功版本。只有规范化 Embedding 输入、Provider、模型、维度、解析/预处理/切分版本均一致时才复用向量。

```text
compatibilityFingerprint = SHA-256(
  normalizedEmbeddingInputHash + provider + model + dimensions +
  parserVersion + normalizerVersion + chunkerVersion + embeddingTemplateVersion
)
```

Markdown 以标题路径为一级边界，记录完整 `headingPath`；代码块和表格原则上不从中截断。纯文本优先按空行段落拆分，并保存行号或字符区间。`chunkKey` 应基于结构区域、局部文本锚点和出现序号生成，不能仅使用全局数组下标，避免在文档头部插入内容后整篇 Chunk 失效。Hash 规范化仅统一换行、Unicode 表示和非语义尾随空白，不能压平可能影响代码或表格语义的空白。

### 5.6 重命名、删除与连续保存

| 场景 | 处理语义 |
|---|---|
| 连续保存 | 合并同一路径尚未执行的任务；Worker 以最终稳定快照为准。 |
| 重命名且内容不变 | 优先根据稳定文件标识识别，次选短时间窗口内的 Hash 匹配；保留 `assetId`，更新来源绑定和逻辑路径，不重复向量化。 |
| 无法确认重命名 | 保守按删除旧资料与新增资料处理；正确性优先于历史复用优化。 |
| 删除 | 创建删除任务，构建不含该资料的新候选索引；切换成功后从默认检索移除并标记相关版本 `deleted`。 |
| 构建失败或取消 | 标记任务/候选状态并记录错误；旧活动索引和历史版本保持可用。 |

### 5.7 候选索引与原子发布

候选索引完成后，激活事务至少检查：候选索引属于目标知识库且状态已就绪；任务未取消；输入资产版本和配置版本未被更新任务取代；候选成员范围、Chunk、Embedding 与配置快照完整兼容。

满足条件时，在同一事务内更新 `KnowledgeBase.activeIndexVersionId`、受影响 `KnowledgeAsset.activeVersionId`、候选索引状态及任务状态，并将先前活动索引标记为历史版本。任何检查失败、任务取消或构建异常都只能处理候选数据，旧活动指针保持不变。

## 6. 配置版本与受控重建

### 6.1 配置状态

| 状态 | 说明 |
|---|---|
| 编辑草稿 | 用户尚未保存的临时修改；离开页面时可提示未保存变更。 |
| 已保存配置版本 | 保存后生成递增版本；内容无变化时不创建无意义版本。 |
| 活动索引配置快照 | 当前默认检索实际使用的解析、切分和 Embedding 配置；可落后于最新已保存配置。 |
| 待重建状态 | 已保存索引兼容配置变更尚未完成候选重建时的显式状态。 |

### 6.2 配置影响分类

| 分类 | 示例 | 生效规则 |
|---|---|---|
| 查询行为变更 | 召回数、返回数、阈值、混合权重。 | 保存后用于后续查询，无需重建索引。 |
| 索引兼容变更 | 解析器、切分大小/重叠、标题策略、预处理、Embedding Provider/模型/维度/模板。 | 保存后显示待重建；仅在用户确认、候选重建和原子切换成功后影响默认索引。 |
| 接入行为变更 | 默认资料类型、路径规则、上传冲突策略。 | 保存后用于未来上传，不回写历史版本。 |

### 6.3 重建流程

1. 保存索引兼容变更后，配置服务计算并展示影响范围、当前活动索引与待重建状态。
2. 用户确认后创建 `rebuild` 任务，固化目标配置快照和资产范围。
3. Worker 以固定范围构建候选索引；已有活动索引持续提供默认检索。
4. 首次建库尚无活动索引时，页面应显示“正在建立首个索引，向量检索暂不可用”，不得显示为空结果或系统故障。
5. 校验成功时原子切换；失败或取消时保留旧活动索引和旧配置快照。
6. 新旧索引版本元数据按保留策略保存，支持追溯、问题定位和后续恢复决策。

## 7. 检索、证据与服务契约

### 7.1 检索与证据规则

检索服务先读取项目默认知识库的活动 `IndexVersion`，再在固定成员范围内执行关键词、向量或混合检索，并按资料类型、来源和逻辑路径过滤。

- 关键词检索用于术语、错误码、字段名、变量名和精确短语。
- 向量检索仅在 Embedding Provider 和活动向量索引可用时执行。
- 混合检索分别召回关键词和向量候选，以可解释权重融合，并按保存配置选择是否执行二阶段 Reranker；向量或 Reranker 不可用时保留关键词结果并标记降级。
- 原文读取必须以 `assetVersionId` 为主键，并按照行号或字符区间定位；不得仅以 `assetId` 读取当前最新文件。

每条检索结果至少返回：

```json
{
  "projectId": "...",
  "knowledgeBaseId": "...",
  "indexVersionId": "...",
  "assetId": "...",
  "assetVersionId": "...",
  "assetType": "technical_design",
  "sourceType": "upload",
  "logicalPath": "架构设计/服务拆分.md",
  "chunkId": "...",
  "chunkKey": "...",
  "headingPath": ["同步", "失败恢复"],
  "location": { "startLine": 120, "endLine": 150 },
  "snippet": "...",
  "retrievalMode": "hybrid",
  "scores": { "keyword": 0.7, "vector": 0.86, "final": 0.92 }
}
```

### 7.2 检索不可用与降级语义

| 业务码 | 场景 | 响应语义 |
|---|---|---|
| `KB_NO_READY_VERSION` | 知识库尚无就绪资产版本。 | 提示上传资料或等待同步。 |
| `KB_NO_ACTIVE_INDEX` | 尚无活动索引。 | 展示首个索引建立状态。 |
| `KB_INITIAL_INDEXING` | 首个索引构建中。 | 明确向量检索暂不可用。 |
| `KB_VECTOR_DEGRADED` | Embedding Provider 不可用。 | 向量/混合检索降级；关键词检索可继续。 |
| `KB_FILTER_EMPTY` | 筛选条件排空结果。 | 提示调整资料类型、来源或路径。 |
| `KB_NO_MATCH` | 正常检索无匹配。 | 保留查询条件并提示无匹配。 |

### 7.3 API 与异步约定

本方案定义资源和行为边界，具体路径与认证方式应服从平台 API 规范。

| 资源组 | 推荐操作 | 关键契约 |
|---|---|---|
| 项目默认知识库 | 获取知识库、概览、统计、活动索引状态。 | 返回稳定知识库 ID、当前活动索引和最近任务状态。 |
| 配置 | 读写草稿、保存、版本列表、影响分析、发起重建。 | 草稿不覆盖已保存版本；兼容变更返回待重建信息。 |
| 上传接入 | 上传资料及类型、显示名称、逻辑路径、冲突模式、幂等键。 | 使用服务端快照；内容变化返回 `202` 和任务标识。 |
| 未来连接器 | 接入 Wiki 等文档来源。 | 一期不提供接口；未来只投递固定快照和标准任务。 |
| 资产与版本 | 列表、筛选、详情、版本历史、指定版本原文与 Chunk。 | 固定原文读取必须携带 `assetVersionId`。 |
| 任务与索引 | 查看任务、进度、错误、重试、取消；查看候选/活动/历史索引。 | 返回配置快照、成员范围摘要与可取消性。 |
| 检索与证据 | 关键词、向量、混合检索和固定版本证据读取。 | 返回完整证据字段，不替换为最新版本。 |

上传同步、删除和重建都是长任务：受理后返回 `202 Accepted` 与 `jobId`；无变化复用结果可返回 `200 OK`。客户端可通过轮询、SSE 或平台统一事件机制获取进度，最终选型待平台能力确认。

错误响应应具有稳定业务码、用户可理解信息、`traceId`、可重试标识和失败阶段。`400` 用于参数、路径、编码或配置无效；`409` 用于拒绝式路径冲突、并发保存冲突和条件切换失败；`413` 用于超限；`422` 用于内容不能处理；`429` 用于限流。错误内容不得包含完整原文、Embedding 请求文本或密钥。

## 8. 安全、可靠性与可观测性

### 8.1 安全控制

| 风险 | 控制措施 |
|---|---|
| 恶意文件与伪造 MIME | 服务端校验扩展名、实际内容、UTF-8 解码与大小，不仅依赖 MIME。 |
| 路径穿越与符号链接逃逸 | 受控根目录白名单、真实路径解析和根目录包含性校验，拒绝目录/设备/不可读对象。 |
| 上传覆盖服务器文件 | 服务端生成对象键或受控路径；原始文件名仅用作展示。 |
| 资源耗尽 | 配置单文件、批量、项目总量、最大 Chunk 数、任务并发和请求速率上限。 |
| Markdown 外部副作用 | 不抓取或执行 Markdown 链接、图片与 HTML；一期仅解析允许的文本结构。 |
| 密钥与资料泄露 | Provider/数据库密钥通过 Secret 或环境变量注入；日志不记录完整原文、向量和密钥。 |
| 云端模型数据出域 | 评审数据驻留、供应商保留策略与合规要求；敏感项目支持选择合规本地或私有 Provider。 |

一期不新增产品级 ACL、审批或审计模型，但部署接入层应使用平台已有认证、网络隔离和最小权限运行账户。

### 8.2 可靠性机制

- 使用持久化任务表或同等可靠队列保存状态、尝试次数、可执行时间、租约、锁定 Worker、错误与配置快照。
- Worker 定期心跳；租约超时任务可安全重新领取。
- 按路径或资产范围使用互斥锁/条件更新，防止多个 Worker 竞态提交。
- Worker 与索引发布通过条件事务确认输入范围和配置版本，阻止陈旧任务覆盖新版本。
- 候选失败、取消或进程异常仅影响未提交数据；旧活动索引持续可读。
- 数据库或 Worker 恢复后，可依据原始快照、来源绑定、资产版本和配置快照重新构建候选结果。

### 8.3 可观测性

结构化日志应关联 `traceId`、`jobId`、`projectId`、`knowledgeBaseId`、`assetId`、`assetVersionId`、`indexVersionId` 与规范化路径摘要。建议持续观测：

- 上传量、格式/编码拒绝率和重复内容短路比例；
- 队列深度、同步延迟、成功/失败/取消/重试率；
- 无变化资料比例、Chunk 新增/修改/删除/复用量、Embedding 复用率；
- Provider 调用耗时、限流、超时、错误与可用 Token/费用；
- 候选构建时长、校验失败率、活动索引年龄、切换次数、重建成功率；
- 关键词/向量/混合检索延迟、零结果率、筛选排空率、固定证据定位成功率；
- 中断任务恢复数、陈旧候选拒绝数和未处理异常数。

上述指标用于建立试点基线，不构成未经评测的上线性能承诺。
